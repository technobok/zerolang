This document describes the I/O subsystem: stream protocols, file operations, error handling, and the design principles that shape them. The goal is to make the easy case terse and make the wrong case hard to write.

1. Status

The subsystem lands in phases. Currently shipped:

  • result t: type, e: type generic union, ioerror failure variant, bytes / byteview / path / pathview typedefs
  • Stream protocol declarations (reader, writer, closer, seeker) -- the type-level contract
  • File class scaffold (fields only; methods to come)
  • One-shot file helpers: io.read_text, io.write_text, io.append_text
  • Filesystem ops: io.exists, io.mkdir, io.remove, io.rename
  • Stderr helper: io.eprintln

Not yet shipped:

  • File class methods (read / write / close / seek) and explicit stream composition
  • Standard streams (io.stdin / io.stdout / io.stderr) beyond print and eprintln
  • Directory iteration (io.list_dir), io.stat, io.mkdirp
  • Buffered and text wrappers (bufreader, bufwriter, textreader, textwriter), UTF-8 validation
  • Process / argv / env (planned for a separate os unit)
  • Async, sockets, symlinks, advisory locks

2. Design Principles

  • Make bad code unrepresentable where the type system can express it -- forgotten close becomes a destructor invocation rather than a runtime leak; use-after-close becomes a type error; binary data in a text signature becomes a type mismatch.
  • One way to say it -- I/O errors are always result(T, ioerror). There is no second error convention.
  • Sync only, for now -- the compiler targets C and has no async runtime. Streams are synchronous. Async lands later as a separate layer.
  • Ambient stdio, cooperative sandboxing -- io.stdout, io.open(...) are callable from anywhere, matching the mental model of C / Go / Python. Test sandboxing is available by shadowing lib/system/io.z in the VFS bind order before the stdlib copy.
  • Cross-platform, POSIX first -- initial backend uses libc (open, read, write, close, ...). Windows is a follow-up with the same Z-level API.

3. The result Type

Every fallible I/O operation returns a two-armed generic union:

: result: union {
      ok: t
      err: e
  } as {
      t: any.generic
      e: any.generic
  }

Callers pattern-match:

: r: io.read_text "notes.md".string
  match (
      r
  ) case ok then {
      # use ok arm (when full narrowing is wired)
      print "ok"
  } case err then {
      # handle the ioerror
      io.eprintln "read failed"
  }

The generic is used directly (result.ok 42 e: i64) with explicit type arguments for whichever arm the value does not pin. In signature position, parenthesise the instantiation: out (result t: string e: ioerror).

result lives in lib/system/system.z and is re-exported through core.z. It is not I/O-specific -- any stdlib API that wants pattern-matchable success / failure can reuse it.

4. The ioerror Union

All I/O failures collapse into one of these arms:

ArmMeaning
notfoundENOENT -- the entity at the given path does not exist
permissiondeniedEACCES / EPERM -- the kernel rejected the access
interruptedEINTR -- a signal interrupted the call (low-level helpers retry)
invalidpathpayload: a string describing why the path was rejected
eofEnd of file reached unexpectedly (streaming-only; v1 is silent)
badencodingReserved -- future use when a text reader validates UTF-8
existsEEXIST -- a create-new operation hit an existing entity
isdirEISDIR -- file operation attempted on a directory
notdirENOTDIR -- directory operation attempted on a non-directory
nospaceENOSPC -- filesystem full
otherpayload: a string with an opaque message for anything uncategorised

ioerror is defined as a union (not a variant) because two arms carry a string payload; variants may only hold valtype arms.

The compiler's C runtime builds ioerror values from POSIX errno via z_io_errno_to_ioerror (see src/zemitterc_runtime.py). Any errno not in the explicit table falls into other.

5. Bytes, Strings, and Paths

Zerolang strings are heap-backed UTF-8 byte buffers (z_string_t at the C level). Stringviews (stringview) are borrowed views into that storage. For binary I/O, the same byte-buffer primitive is wrapped under a distinct type:

: bytes:     class { val: (list of: u8).typedef }
  byteview:  class { val: (listview of: u8).typedef }

Signatures that say bytes announce "binary data, no UTF-8 guarantees"; signatures that say string announce text. The wrappers delegate to the list / listview machinery, so operations like .length and .append flow through.

path / pathview are similar typedefs over string / stringview, introduced so filesystem operations read as "this is a path" rather than "this is an arbitrary string". Cross-platform normalisation (forward-slash / backslash, drive letters) is a future enhancement; v1 treats paths as opaque byte sequences.

6. Stream Protocols

The stream contracts are declared in lib/system/io.z. Each one returns its payload in a result wrapper so failures propagate uniformly.

: reader: protocol {
      read: function {:this into: bytes max: u64}
          out (result t: u64 e: ioerror)
  }

writer: protocol { write: function {:this from: byteview} out (result t: u64 e: ioerror) flush: function {:this} out (result t: null e: ioerror) }

closer: protocol { close: function {:this} out (result t: null e: ioerror) }

seeker: protocol { seek: function {:this to: i64 from: seekorigin} out (result t: u64 e: ioerror) }

A file implements all four via separate conformance labels. Buffered and text wrappers (future phase) hold a .lock borrow of their underlying stream so the parent cannot be touched directly while a wrapper is live -- the interleaved-write footgun is unrepresentable.

seekorigin is a variant for the offset basis:

: seekorigin: variant {
      start: null
      current: null
      end: null
  }

7. File Handles

The file class wraps a POSIX file descriptor:

: file: class {
      fd: i32
      closed: bool
  }

Lifetime model: RAII + with sugar, both supported.

  • A file falls out of scope with an open fd? The destructor calls close(fd). Forgetting is physically impossible -- the language does not have an "uninitialised local" path for a reftype.
  • Need to surface a close error? Use with f: io.open(...) do { ... } and call f.close explicitly inside the block. The destructor sees closed = true and skips the second close.

(File methods land with the next phase; this section documents the contract the scaffold already establishes.)

8. One-Shot File Helpers

For programs that do not need streaming, three entry points cover the typical script work:

CallReturnsNotes
io.read_text path: stringresult(string, ioerror)Reads the full file as UTF-8 bytes; no encoding validation in v1
io.write_text path: string content: stringresult(null, ioerror)Truncates (or creates) with mode 0644. Parent directory must exist.
io.append_text path: string content: stringresult(null, ioerror)Opens O_APPEND | O_CREAT.

All three take ownership of their string arguments (zerolang's string-arg convention -- the callee frees). Callers typically pass literals projected through .string:

: io.write_text path: "notes.md".string content: body.string

9. Filesystem Operations

CallReturnsNotes
io.exists path: stringboolThin access(F_OK) wrapper. Permission-denied on parent surfaces as false; use stat (future phase) when the distinction matters.
io.mkdir path: stringresult(null, ioerror)Creates a single directory with mode 0755. Parent must exist. For recursive creation see mkdirp (future).
io.remove path: stringresult(null, ioerror)Deletes a file or an empty directory. Non-empty directories error with the other arm (ENOTEMPTY).
io.rename from: to:result(null, ioerror)POSIX rename. Atomic within the same filesystem. If to exists, POSIX replaces it (file over file, empty dir over empty dir).

10. Stderr Helper

io.eprintln msg: stringview writes msg followed by a newline to stderr. Void return; errors are silently dropped by design.

For per-write error reporting use a writer handle over stderr (future phase when the stdio constants land).

11. Ownership Conventions for Native Calls

The stdlib's native I/O entries follow a few rules that are useful to know when extending them:

  • String arguments are consumed. Passing a string to a native (for example path) transfers ownership; the C callee is responsible for z_string_free. The compiler invalidates the caller's temp so there is no double-free. If you want to keep the caller's string, make a copy before the call.
  • Values returned by value. Native functions that return a result return the union struct by value. The caller's scope owns it; its destructor (z_result_<T>_<E>_destroy) is called on scope exit and frees any heap payload (the boxed ok value or the boxed ioerror).
  • errno to ioerror mapping. A shared helper maps POSIX errno values to ioerror variants. New natives can reuse it (see z_io_errno_to_ioerror and z_io_wrap_null_result in the runtime).

12. Sandboxing

Because io.* lives in lib/system/io.z and all accesses route through that unit, a caller who needs to lock down I/O can bind a replacement io.z before the stdlib copy in the VFS. The replacement can expose only a whitelisted subset of operations, or redirect everything to a test harness. This controls what cooperating code can see.

It does not prevent malicious code from re-declaring is native against the runtime C symbols and reaching libc directly. Treat unit-replacement sandboxing as protection against accidents and against well-behaved dependencies, not against a deliberate attacker.

13. Recipes

13.1. Read a file with error fallback

: main: function is {
      r: io.read_text "config.txt".string
      match (
          r
      ) case ok then {
          print "loaded config"
      } case err then {
          io.eprintln "config missing; using defaults"
      }
  }

13.2. Atomic write via temp + rename

Write to a temp file, then rename over the target. Rename is atomic within the same filesystem so a reader never sees a half-written file.

: main: function is {
      wr: io.write_text path: "out.tmp".string content: payload.string
      match (
          wr
      ) case ok then {
          io.rename from: "out.tmp".string to: "out.txt".string
      } case err then {
          io.eprintln "write failed"
      }
  }

13.3. Create a directory if missing

: main: function is {
      if io.exists "cache".string then {
          print "cache present"
      } else {
          io.mkdir "cache".string
      }
  }

14. Implementation Notes

The I/O stack is three layers:

  • Native C primitives (in src/zemitterc_runtime.py) -- thin libc wrappers: z_io_read_text, z_io_write_text, z_io_mkdir, z_io_remove, z_io_rename, z_io_exists, z_io_eprintln, plus the z_io_errno_to_ioerror helper. The emitter includes only the helpers the program actually calls (tracked in needs_io_natives).
  • Zerolang declarations (in lib/system/io.z) -- each native has an is native signature. The typechecker sees them as ordinary functions for the purpose of argument checking and return-type monomorphization.
  • User-facing API -- the names in core.z (io.read_text etc.). User code writes calls, the emitter dispatches to z_<qualified> via the generic native path, and the runtime provides the body.

When the user calls a native that returns a monomorphized result, the emitter walks the corresponding mono_types entry and emits the tag enum + struct + destructor. The io runtime block that references those names is appended after the struct definitions so the generated C compiles cleanly.

15. Future Work

Roughly in implementation order:

  1. File class methods -- file.open, file.read, file.write, file.close, file.seek, and the four protocol conformance labels. Enables streaming and buffered wrappers.
  2. Standard streams -- io.stdin, io.stdout, io.stderr as singleton borrowed handles.
  3. io.stat returning result(metadata, ioerror) for file size / mtime / type.
  4. Directory iteration -- io.list_dir returning a closable iterator of direntry.
  5. Buffered + text wrappers -- bufreader, bufwriter, textreader (.lines iterator), textwriter (write_line).
  6. UTF-8 validation on text reads; the badencoding ioerror arm becomes reachable.
  7. Recursive mkdirp, symlink handling, advisory file locks.
  8. Process / argv / env -- separate os unit.
  9. Windows backend behind the same zerolang-level API.
