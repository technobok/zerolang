1. Introduction

A document showing the compiler architecture overview.

2. Virtual File System

A simple readonly Virtual File System (VFS) is used to present a unified source tree to the compiler.

The VFS allows union or bind mounts to mount one filesystem over another. The VFS tree presented to the compiler is usually the system directory with the user source directory union mounted over this. In this way, user units are found first, shadowing system units. This offers some backwards compatibility - future system modules that may be released that collide with user modules will be hidden by them. The user modules would need to be refactored/renamed to make use of the new system modules.

The VFS allows abstraction over a source tree that may be arbitrarily complex while providing a simple interface to the compiler. The VFS tree can be built from multiple physical filesystems or can be built from alternate data stores (such as databases or remote systems).

3. Tokenizer

The Tokenizer splits a single top level unit aquired from the VFS (usually represented by a file) into a stream of tokens. Every part of the source becomes part of a token and so the original source could be recreated by concatenating all of the tokens returned by the tokenizer.

The tokenizer splits the file into tokens such as:

  • horizontal whitespace (spaces and tabs only)
  • end of line
  • comments
  • identifiers
  • keywords
  • delimiters such as (, ), { and }
  • dot operator: .
  • parts of string literals (raw and interpolated)
  • numeric literals
  • error tokens

The tokens have a type, the string contents and an identifier for the file that they are contained within and the location (line and column) within that source file.

The Tokenizer follows the ITokenizer protocol which provides a single 'token' method. This allows alternate tokenizers to be provided or allows filtering or logging to be inserted.

4. Lexer

The Lexer provides another layer over the Tokenizer to ease the job of the parser which will follow.

The Lexer filters comments and whitespace as these are not used by the parser. It also keeps a small stack to maintain context for handling end-of-line (EOL).

In Blocks (the top level of a unit or a brace delimitered block), multiple EOL's are coalesced into a single EOL token. In parenthesized expressions, all EOL's are ignored. In string literals, all EOL's are passed through as-is and form part of that string literal.

5. Parser

The parser creates an abstract syntax tree (AST) based on the source units supplied. It begins with the 'core' system module and the user 'main' module and compiles each, following any unresolved references as they are found.

The Parser maintains an Environment stack for recording definitions at each scope. At the beginning of each new scope, a new frame is pushed onto the Environment. The Parser records each definition made as compilation proceeds. References to these definitions are recorded as they are found.

Since forward references within a unit are allowed, a list of undefined identifiers are also maintained in the Environment. After compiling the current unit, an attempt is then made to resolve these reference.

Firstly, a check is made to see if there is a file with the same name as the reference in a subdirectory with the same name as the current source file (an implimentation subunit). If found, this unit file is then compiled following the same process as above, recursively.

If a subunit is not found, the undefined reference is promoted to the next higher unit to see if it is defined there. This continues until the reference is promoted to top level 'global' unit. If still not found, the system 'core' unit is searched. And if not found there, a syntax error is returned.

Only the first (left-most or only) reference in a dotted path is attempted to be found by the Parser. The goal is to ensure a complete AST that includes all referenced units is created.

The Parser is not concerned with any typing information and does not ensure dot operators against the left-most reference will succeed. That task is given to the type checker.

The Parser will return the first syntax error found immediately. It does not attempt to reset and continue compilation.

6. Generator Desugaring

A function whose declared return type is Iterator gives: T (takes: U) and whose body contains at least one yield is a generator. The generator desugaring pass (src/zgenerator.py) rewrites every such function into a pair: a synthesised class implementing the iterator protocol and a factory function that constructs an instance with all parameters captured. The pass runs as the first step of TypeChecker.__init__ -- it operates on the parsed AST before resolution -- so the rest of the typechecker sees only ordinary class + function code.

Pipeline placement:

  Lex -> Parse -> [Generator desugaring] -> Typecheck -> Emit

6.1. Synthesised class shape

For a generator F: function {p1: T1 ...} out (Iterator gives: G takes: K) is { ... body ... } the pass produces approximately

  F_iter: class {
      state: i64               # state-machine cursor
      p1: T1                   # captured params (with original ownership)
      <crossing-locals>: ...   # locals whose live range crosses a yield
      _resume_input: K         # only when takes is non-null
  } as {
      create: function {p1: T1 ...} out this is {
          return meta.create state: 0 :p1 ... <local-defaults>
      }
      call: function {:this} out (option-wrapper-for-G) is {
          # original body, references to promoted names rewritten to this.<n>;
          # carries synth_origin = "generator-call" so the emitter routes it
          # through the state-machine codegen below.
      }
  }

  # original F is replaced with a factory of the same signature:
  F: function {p1: T1 ...} out F_iter is { return F_iter :p1 ... }

The synthesised class name is <funcname>_iter for standalone generators and <typename>_<methodname>_iter for methods. Numeric suffix on collision.

6.2. Liveness (which locals get promoted)

A local must become a class field iff its live range crosses a yield -- otherwise its C-stack slot in .call is gone between invocations. _crossing_locals(body, param_names) in src/zgenerator.py runs a structural traversal applying two rules:

  • Yield-count rule -- track a running yield counter (pre-bump for the yield's own expr). For each local, record first_def_yc and max_use_yc. If max_use_yc > first_def_yc, the def-to-use path crosses a suspension.
  • Yielding-loop rule -- any reference to a local inside the body of a for loop whose body contains a yield is treated as crossing (the next iteration's read sits "after" the previous iteration's yield).

Parameters are always promoted (the factory captures them into the synth class). Non-crossing locals stay as ordinary C locals inside the generated .call body.

6.3. State-machine codegen

The emitter (src/zemitterc.py) routes any function whose synth_origin == "generator-call" through a dedicated state-machine path. Helpers _setup_generator_ctx, _emit_generator_prologue, _emit_yield_fragment, and _emit_generator_epilogue assign one state number per yield (entry = 0, terminal = -1) and emit:

  switch (this->state) {
      case 0:  goto L_entry;
      case 1:  goto L_resume_1;
      ...
      default: goto L_done;
  }
  L_entry:
      /* original code up to first yield */
      this->state = 1;
      return OPTION_SOME(<yield expr>);
  L_resume_1:
      /* if takes != null: bind local from this->_resume_input */
      ...
  L_done:
      this->state = -1;
      return OPTION_NONE;

C17 goto plus switch arbitrates cleanly; no GCC extensions. The yield-state numbers are pre-allocated in _setup_generator_ctx so the prologue, body fragments, and epilogue agree.

6.4. Termination + bidirectional value flow

A bare return anywhere in the body or fall-through past the last yield transitions to the terminal absorbing state. return <value> is rejected during desugaring (_validate_no_return_value). Code placed after the last yield runs on the call that crosses into terminal; it does not run on early drop, where class destruction releases locks and destroys owned fields via the standard class-lifetime path.

For takes: K generators, _build_call_method adds a value: K parameter; _setup_generator_ctx emits an entry-time store of value into this->_resume_input (destroying any prior owned value first). Expression-form yield (x: yield v) in the body reads this->_resume_input after the resume label so x binds to the caller's most recent input. The first reachable yield must be statement form -- the desugarer rejects expression-form first-yields (_validate_first_yield_not_expression_form), reflecting that the resume-input slot is undefined before the first .call value: arrives.

6.5. Known gaps

The for-loop relay pattern (for x: (subgen ...) loop { yield x }) compiles today but stops after the first inner value, because the synthesised for-loop iterator is a C-stack local in .call that doesn't survive yield-suspension. Closing the gap requires either a desugarer-side synthetic local for each yielding-for-loop iterable (so _crossing_locals sees it and promotes it) or an emitter-side carve-out that promotes the for-loop iterator into a synth-class field. Tracked in the generator plan as deferred item P9.

7. Semantic Analyser (Type/Ownership/Lock checking)

The Semantic Analyser assigns a type to each node in the AST. It compares types of definitions, assignments and arguments being passed to ensure the types are compatible.

A stack of nodes being typechecked is maintained as checking proceeds. This allows the type checker to catch circular dependencies.

7.1. Demand-Driven Resolution

The type checker is demand-driven: it starts from main and resolves names on demand as they are referenced. Types, functions, and definitions that are never reachable from main are never type-checked. This ensures the compiler only processes what is needed, and circular dependencies are detected when a name is encountered that is already on the resolution stack.

7.2. Monomorphization

Generic types are instantiated through monomorphization. When a generic type (e.g., List(String)) is used, the type checker creates a concrete copy (a monomorphized type) with the generic parameters replaced by concrete types. The monomorphized type name is mangled (e.g., list__string) to produce a unique C identifier.

Generic method bodies are deep-copied during monomorphization and type-checked with the concrete type arguments in scope. This ensures that each instantiation is independently validated.

Compiler-injected methods on collections (.append, .set, .get, .byteview, .listview, .stringview, .iterate, .each, .length, and friends) are not synthesised by the monomorphizer. They are declared with the is native form in lib/system/system.z and lib/system/collections.z; the monomorphizer clones those declarations alongside everything else. Resolution flows through the standard child-lookup path -- there is no syntactic special case in the type checker for these names, and a user-defined member of the same name on a non-collection type resolves normally. Methods that return a borrowed reference into a parameter or receiver declare that source as a .lock-annotated parameter (e.g. function {t: this.lock} out T.borrow); the standard .lock-param transfer at call resolution drives the call-site locking, with no name-based pattern matching.

A no-user-arg method dotted path (v: s.stringview, p1.distance inside string interpolation) is auto-called when read in value position. The coercion lives in _check_dotted_path and is gated by a coerce_method_to_return flag that _check_call flips off when resolving its callable -- so explicit method calls (obj.method named: arg) see the function type and dispatch normally instead of falling into construction-of-return-type. Path resolution itself (_resolve_dotted_path) is context-free: it always returns the FUNCTION type for a method path. The behaviour is identical for is native and user-defined methods.

7.3. Ownership and Lifetime

The ownership status of each expression is determined using a 2-state model: OWNED (the variable owns the instance and is responsible for its lifetime) or BORROWED (a temporary reference; the variable does not own the instance).

Function parameters carry ownership annotations: take (caller transfers ownership to callee), borrow (callee gets a borrowed reference), or lock (callee locks the argument for the duration of the call). See doc/ownership.pdoc for the full specification.

7.4. Scope Environment

All variable state -- type, ownership, lock state, type narrowing, and taken (consumed) status -- is tracked in a unified scope-based environment. Each scope holds a List[Entry] rather than a dictionary. Scopes are small (measured maximum of approximately 6 entries in practice) so linear scan outperforms hash lookup. This also simplifies the eventual self-hosted implementation.

An Entry carries: the variable name, its resolved type, whether it is a definition or a shadow overlay, optional ownership info (VarInfo), optional lock state (LockInfo), optional type narrowing (narrowed_subtype and excluded_subtypes), and optional taken state.

Three scope kinds are used:

  • BLOCK -- language constructs (function, do, for, if, with, match, match arm). Can contain variable definitions.
  • CALL -- pushed before processing call arguments, popped after the call returns. Lock entries placed here are released automatically when the scope is popped.
  • OVERLAY -- per-statement state changes. Contains shadow entries that record a new state (lock, narrowing, taken) for a variable defined in an outer scope.

State changes follow an immutability principle: new shadow entries are created in overlay scopes rather than modifying existing entries. The scope chain is the history. Scope exit (via pop_to) naturally releases all state from that scope -- locks vanish, narrowing resets, taken markers are merged into the parent.

Each scope carries a monotonically increasing scope_id for tracing and SQL output.

7.5. Lock Checking

Lock keys are paths -- tuples (root, f1, f2, ..., fn). Scope chain entries (Entry.name) are keyed by the root so the linear scan stays cheap; the full path lives on Entry.lock.Path and is consulted to apply the prefix-overlap conflict rule. Two lock scopes exist:

  • Call-scoped locks -- taken on function arguments during a call. A call scope is pushed before argument processing. For each reftype argument, SHARED locks are taken on every intermediate prefix of the source path and EXCLUSIVE on the leaf. The receiver path locks identically. After the call returns, the scope is popped and all call-scoped locks are released. For .lock parameters, the leaf lock transfers to the return value (installed as a borrow-scoped lock in the parent scope).
  • Borrow-scoped locks -- taken when .borrow, .lock, .stringview, .listview, a with-binding, or a .lock-param transfer creates a borrowed binding. SHARED entries are placed on each intermediate prefix and EXCLUSIVE on the leaf in the scope where the borrow variable is defined. The locks persist for the lifetime of that scope (released on scope exit or explicit .release).

Conflict check (multi-granularity): same-path locks dedupe when both SHARED and conflict when either is EXCLUSIVE; a strict ancestor SHARED is intent-shared and permits any lock on descendants; a strict ancestor EXCLUSIVE conflicts with any descendant. Sibling paths (no prefix relation) never conflict. Enforcement for access / mutation sites uses the same prefix-overlap test: any EXCLUSIVE whose path prefix-overlaps the target path is a conflict.

Key code: try_lock / find_exclusive_lock / _lock_acquire_conflict in src/zenv.py; _lock_arg / _lock_receiver / _install_borrow_locks / _get_dotted_path_tuple / _check_not_locked in src/ztypecheck.py.

All parameters default to .borrow regardless of type family. For valtypes, .borrow means a copy (no lock, source not invalidated). For reftypes, .borrow means a reference (source not invalidated). .lock on a known valtype is a compile error -- valtypes are pass-by-value so identity-based locking is meaningless. Generic type parameters are exempt because they may monomorphize to a reftype.

7.6. Numeric Literal Coercion

Bare numeric literals are typed as one of two compiler-internal singletons during typecheck: LITERAL_INT or LITERAL_FLOAT (both flagged is_literal=True on ZType). Constant expressions built from them stay literal-typed too — the binop folder operates at unbounded Python-int precision when both operands are literal, with no intermediate range check. A LITERAL_INT only becomes a concrete numeric when one of the following fires:

  • Typed-location coercion via _coerce_literal / _coerce_literal_by_id in src/ztypecheck.py. The helper reads node_const_value, applies the rule appropriate to the source/target combination (int→int range check via NUMERIC_RANGES; int→float exact mantissa via int_fits_float; float→float round-trip via float_fits_float; float→int rejected outright), and on success rewrites node_type to the target. Wired sites: call arguments (_check_call_arguments), function implicit return (_check_function_body_inner), reassignment RHS (_check_reassignment_inner), match-arm numeric patterns (_check_case_inner), if-/match-expression branch unification via _unify_literal_branches.
  • Bind-site materialisation via _materialise_literal. Called from _check_assignment_inner and _resolve_unit_name so a literal-typed RHS never escapes into long-lived storage (ZVariable.ztype, TypeChecker._resolved). Also wired into generic-arg inference (_infer_generic_function_call, _infer_generic_record_construction, _infer_generic_union_construction, _check_box_construction) so the monomorphization key is keyed on the concrete default (e.g. id_i64, not id_literal_int).
  • Default-resolution late pass _resolve_literal_defaults, invoked from the module-level typecheck() entry point after check() returns. Walks Typing.node_type and blindly replaces any remaining literal-typed entry with its concrete default (i64 for LITERAL_INT, f64 for LITERAL_FLOAT). No range check fires here — that is the coercion sites' job. Guarantees the emitter never sees a literal pseudo-type.

Method dispatch on literal-typed receivers (e.g. 5.iterate) follows the existing typedef_base walk. _link_literal_typedefs runs at the start of check() and points LITERAL_INT.typedef_base = <i64> and LITERAL_FLOAT.typedef_base = <f64>, so the operator-method lookup in _check_binop_inner and child-name lookups elsewhere resolve through the concrete type without per-site is_literal guards.

_types_compatible treats literal-vs-concrete-numeric as structurally compatible so multi-arm constructs unify cleanly; the range check happens at the typed-location coercion sites, not in the compat check.

Three pre-existing typechecker gaps cover the remaining sites the original design called out — record/class field-init type validation, data-block element type unification, and separable parameter defaults — and are tracked separately from this work.

7.7. Type Narrowing

Type narrowing for union and variant types is tracked through scope-based overlay entries. When a match arm narrows a variable to a specific subtype, a narrowing entry is placed in the arm's scope. When the arm scope is popped, the narrowing resets automatically.

Post-match exclusion works by recording which subtypes were handled by diverging arms (arms that return or break). After the match scope is popped, exclusion entries are placed in the parent scope. Subsequent access to an excluded subtype produces a compile error.

Assignment-based narrowing (e.g., x: Result.ok 42) also places a narrowing entry in the current scope. Reassignment resets the narrowing.

If all branches of an if or match diverge, the parent scope is marked unreachable and subsequent statements are reported as dead code.

7.8. Synthesis Toolkit

The type checker may need to introduce nodes that did not appear in the parsed source -- temporaries introduced by per-call argument hoisting, equality functions auto-generated from a type definition, and any future ANF / SSA rewrites. Those nodes are created through a small toolkit in src/zsynth.py:

  • FreshNamer -- monotonic per-typechecker temp namer. Returns names of the form _t0, _t1, ... in the order they are requested. The monotonicity matters: tests rely on the numeric suffix to assert source-order evaluation of call arguments.
  • make_atom_id, make_assignment -- helpers that build AST nodes with the right node-id allocation, source location defaulting, and synth_origin stamping. Direct construction of nodes outside the toolkit is discouraged because it bypasses these defaults.
  • register_synth_var -- registers a synth temp's ZVariable in the current symbol-table scope so downstream resolution finds it. Used in tandem with make_assignment to install a hoisted temp.

Every synthesised node carries a synth_origin: str field on the Node base class (and on ZVariable in the symbol table). The field names the rewrite that introduced the node (for example, "anf" for per-call argument hoisting). The field is persisted in the SQL dump (src/zsqldump.py) so rewrite provenance survives into post-mortem inspection.

Current consumers:

  • _hoist_arg in src/ztypecheck.py -- per-call atomic processing rewrites every non-trivial call argument to a synth temp inside a per-call sub-scope. See doc/ownership.pdoc:Per-Call Atomic Processing for the model.

8. Code Generator

The code generator (C emitter) walks the fully typed AST and emits C source code. The output is compiled by a C compiler (gcc/clang) to produce a native executable.

The emitter targets the C17 language standard (-std=c17 -Wall -Wextra). Emitted code relies only on designated initializers, fixed-width <stdint.h> types, and <stdbool.h>; no C23 features (typeof, constexpr, #embed) are used.

The emitter produces several output sections in order: struct/type definitions, forward declarations, function typedefs, function bodies, and a main() entry point. Each section is collected separately and assembled at the end.

8.1. Type Emission

Each zerolang type maps to a C representation:

  • Records → C structs (stack-allocated value types)
  • Classes → C structs behind pointers (heap-allocated with constructors and destructors)
  • Unions → tagged struct with void* data (heap-allocated)
  • Variants → tagged struct with inline data union (stack-allocated)
  • Protocols → vtable struct + instance struct (dynamic dispatch via function pointers)
  • Facets → vtable + inline data union (value-type dynamic dispatch)
  • Enums → integer constants with a tag enum type

8.2. Memory Management

The emitter implements ownership-based memory management without a garbage collector.

String values (ZStr*) are heap-allocated and freed via zstr_free(). Class instances are heap-allocated via malloc() and freed via generated z_<type>_destroy() functions that recursively free owned fields.

The emitter tracks variables that need cleanup in a per-function scope. At scope exit (function return or fall-through), all owned variables are destroyed in reverse declaration order. Each type carries needs_destructor, destructor_name, and is_heap_allocated metadata set by the type checker, so the emitter delegates cleanup decisions to the type system.

Temporary values (intermediate results within a statement) are tracked separately and freed at the end of each statement. When a temporary is assigned to a named variable, ownership transfers and the temporary is removed from the free list.

The .take operation transfers ownership from one variable to another, nullifying the source to prevent double-free.

8.3. Unrecoverable Errors

All runtime-terminal exits route through a single helper emitted in the stdlib preamble of every program:

  static _Noreturn void z_panic(const char* msg) {
      fprintf(stderr, "zpanic: %s\\n", msg);
      exit(1);
  }

User-level panic msg: ... lowers to a direct call to this helper. Internal unrecoverable exit sites -- allocation failure in z_xmalloc / z_xcalloc / z_xrealloc, bounds checks on arrays, lists, listviews and stringviews, numeric cast overflow, empty-list pop -- all funnel through z_panic too, so every termination produces a zpanic: <msg> line on stderr and an exit code of 1.

Formatted messages (bounds with indices, numeric cast overflow) use a small stack buffer and snprintf at the call site to build the full message before invoking z_panic. The helper itself takes a plain const char* -- no variadic <stdarg.h> is pulled in. _Noreturn is C17 and lets GCC's control-flow analysis eliminate any "control may reach end of non-void function" diagnostics in callers that panic on every path.

Source location (.z file + line) is intentionally not included in the zpanic: output; the SQL ast_nodes ↔ emitted_lines source map is the authoritative post-mortem debugging link.

8.4. Code Deduplication

Monomorphized methods are deduplicated using AST content hashing. During monomorphization, each cloned method body is hashed (SHA-256 over the typed AST structure). If two methods produce the same hash, the second becomes an alias of the first. The emitter reads these aliases and emits C preprocessor #define directives instead of duplicate function bodies. This is particularly useful for numeric generics where method bodies may be identical across different numeric parameter values.
