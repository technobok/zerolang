2. Ownership rules

Rules for 'objects':

  • all objects have a single owner or parent which cannot be itself
  • ownership is to direct children only, it does not extend to grandchildren and there is no direct access to grandchildren
  • valtypes are copy by value. Each can only have a single owner, but easily copied so can make a new copy with a new owner.
  • reftypes are copy by reference. Can only have a single owner, but can be loaned via function calls or given away to a function call via a 'take' param
  • objects can only be disposed by their owner and no self-ownership so objects cannot dispose of themselves
  • all reftypes in object fields must be initialised and cannot be removed, only swapped
  • and, methods on reftype fields cannot be passed anything that is not owned (no borrowed references, eg self to prevent killing ourselves) Why not other borrowed references besides self?
  • ...so, reftypes in calls can only be borrowed (from parent call) or owned directly (local var). this.reftype???
  • returned reftypes always give ownership from function to caller. 'take' arguments give ownership from caller to function (and are no longer valid at the caller from that point forward)
  • initialisation? The one true constructor must be called supplying all values, although default values can be used for valtypes.
  • destructor? There is a default destructor to recursively free all reftypes. Can be overridden to free/cleanup non-memory resources. Can return a single reftype from the class. How would that look?

5. Attributes of expression result types:

These are all (mostly) orthogonal?

5.1. Ownership:

  • immutable - top level (unit level) immutable type. The names of top level items are constant, cannot be redefined. Functions, data, numeric types (although, these are passed by value so not a limitation that they are immutable)
  • free - (anonymous). Ownership is free to convey to the LHS/receiver. eg. return values from functions (not from iterators though); or all valtypes. An owned reftype variable can be converted to a free variable by using the take predefined type the the variable as the parameter. If ownership of a reftype is not taken by the LHS, the variable will be destroyed after the current statementline.
  • owned - var created locally or a "taken" parameter in a function body
  • borrowed - reftypes passed in via standard (non- take) function parameter
  • linked - dotted member object of a composite type (eg record/class)

Valtypes are always 'free' because they are always copied to new locations.

'linked' are dotted members of another parent variable (as with all vars, the parent also is owned, borrowed or linked). This is a special class of ownership (eg dotted members are not automatically owned just because you own the parent) and have some additional restrictions (ie. they must be replaced if changed).

Items returned from function calls are always free.

immutable - yes/no (if yes, cannot be assigned to) - top level item usually

writeable - yes/no (if yes, can be assigned to) Could be no, but not immutable eg an expression with no name

free - (anonymous) - yes/no (if yes, can be assigned from)

5.2. Lock type:

All vars have a lock type. Unused local vars and parameters start with 'none'.

(For arguments at call sites; also 'cast', 'match', 'for' can lock target object)

  • blocked - target of match (if not take -en), named (private members too?); local var definitions (including 'casts', eg. newvar: object.protocol) that take a non-'free' var on the RHS will take a lock while in scope.
  • exclusive - a reftype directly passed into a call by name, or receiver in a method call
  • shared - a parent of a linked reftype passed into a call
  • none - for the 'free' ownership types: valtypes, result of function calls, unused local vars/arguments (?? what is that?)

'blocked' makes the target totally unreferencable by that name.

'exclusive' can have methods called on it but no reuse elsewhere and no child access. All methods must take an exclusive lock on the receiver immediately before the call.

'shared' will be for all parent items in a dotted chain except the last (which will have an exclusive lock if it is a reftype field being passed); EXCEPT if the final item is a valtype (in which case, there are no shared locks on the parent items or the valtype)

'none' is essentially a noop for completeness; these situations have lock type of 'none':

  • 'free' ownership types
  • as yet unreferenced vars
  • the results of function calls (they are 'free')
  • unit level items (because they are immutable (?)) - incl. functions, and unit level constant numbers and 'data'.
  • valtypes (they are free)... this includes the entire "path" up to that valtype - ie. no shared locks on parent components of a valtype reference

The typesystem will check the locking of the parts of all calls. Multiple shared locks on an object are ok, but only one exclusive lock on an object is permitted. This includes the call (method) itself, which will need an exclusive lock on the parent (if any).

After a call, all related locks are popped and cleared and the resulting object is 'none'/'free' (except for iterators).

All locks take a reference to the location where the lock was taken for error reporting purposes.

5.4. Ownership and Ownership movement

What is owned/not owned:

  • creating a protocol instance (object.protocolname), by default will be a borrowed object existing to the end of the current scope. The source name can be accessed again at the end of that scope.
  • creating a protocol instance from a taken object (take object.protocolname) creates an owned object that owns the source object. The source name can never be accessed again. XXX can't do this. Do protocol.create to make a 'taken' protocol instance

How to assign:

  • to assign to a field: source expression must be owned. Then use assignment (existing value before assignment will be destroyed) or swap
  • to use in a return expression: source expression must be owned
  • to call a function with take parameters: take must be used where the argument is specified and the source must be owned.
  • take may be used in match target variable and in if optional variable initialisation that want to take ownership of the item in the then clauses. (if not specified the item is borrowed)

Note:

  • if an owned expression is not taken (by LHS (variable or field assignment) or call argument -> function parameter), it is automatically destroyed after use
  • using take on a variable that is not owned is an error

6. Ownership v3

6.1. Attributes: type, ownership, mutability, lock

6.1.1. Type

Type is an attribute of the object (not of the name).

- definition type (deftype) - top level object definition (record, class, enum, functions/methods, numeric constants, etc.) Compiler metadata only and are immutable. - reference type (reftype) - has a runtime 'pointer'/'reference' to a type of possibly indeterminate size, has identity etc. Can only be created in run-time code. - value type (valtype) - an inline block of data with a fixed size for this type. Has value semantics. Can be sources from some top level definitions (compile time numeric constants, function 'references'). Can only be created in run-time code.

6.1.2. Ownership

Ownership is an attribute related to each expression in a function body.

- free: no name (anonymous); result of a standard call (except from a 'borrowed' return as from an iterator); result of 'take' call; all valtypes are essentially 'free' to copy. - owned: local owned var; taken reftype parameter - borrowed: local borrowed var; non- taken reftype parameter; 'borrowed' return type of a function; result from an iterator; 'this' is always borrowed - linked: child reftype field of an object

linked....for returning borrowed, must be linked or borrowed (cannot be owned because it wont exist afterwards)

6.1.3. Locality

Attribute related to the location of the object:

  • local - local variable in the current function scope. Variable will be destroyed when that scope ends. Includes:
    • free (anonymous) objects
    • Local variables (owned variables; and shared variables that are bound to owned local vars)
    • 'taken' function parameters
    • (others?)
  • non-local - variable sourced from outside the current function scope that will live on after the function returns. Includes:
    • non- 'taken' parameters (shared parameters)
    • linked variables; and shared vars bound to linked variables
    • (others?)

Locality is passed in from the source. 'taken' variables become local but otherwise a shared var will have the same locality as its source.

6.1.4. Mutability

Mutability is an attribute related to the name of a 'variable'. The variable value may still be mutable.

  • immutable - top level definitions; method names of object; 'this' inside an object
  • mutable - name can be copied to (with a compatible RHS, previous reftype value will be destroyed, valtypes will be copied over); or swapped

6.1.5. Lock

Lock type is an attribute related to each expression part in a function body. Expressions 'hold' locks over other variable names/values. Only for reftypes; valtypes and deftype always have a lock type of 'none'. When an expression is assigned to a name, the expression locks are then carried with that name and released when the name falls out of scope.

It is an error to attempt to take a lock that breaks the below rules:

  • exclusive - holder can do anything (call method, 'take' etc). No other expression can do anything with this expression while the lock is held. (ie. method calls, use as argument, reference children). No other lock of any type can be taken. Expression type must have a locktype of 'none' before an exclusive lock can be taken.
  • shared - can reference children, cannot call methods, cannot take an exclusive lock. Expression type must have a locktype of 'none' or other 'shared' locks only before taking a shared lock.
  • none - no restrictions, can take a shared or exclusive lock. Top level definitions always have lock type of 'none' and take no locks on their RHS. Top level definitions are immutable and shareable - they are always deftypes or valtypes.

6.2. Operations

6.2.1. createval

Create a new valtype variable. LHS must be a new label. RHS must be a valtype, LHS becomes a copy of that value type.

6.2.2. createref

Create a new reftype variable. LHS must be a new label. New variable will hold all the locks that the RHS has taken.

RHS:

  • must be a reftype
  • RHS parents and 'leaf' must have lock type of 'none', else error
  • if free (use 'take' if required), LHS owns that expression. No locks.
  • if owned or borrowed, LHS borrows that expression and is given an exclusive lock to the RHS leaf and shared locks to all RHS parents until end of LHS's scope (the locks are transferred to the LHS name)

6.2.3. copyval

Only for valtypes. LHS must be owned (valtypes can never be borrowed or linked). LHS name must be 'mutable'.

RHS must be assignable to the type of the LHS type (same type or losslessly convertable to a larger numeric type in all cases).

6.2.4. moveref

Only for reftypes. LHS must be owned or linked.

  • RHS must be assignable to the LHS (the same type?)
  • RHS must be free or owned. Owned will be automatically promoted to free
  • if RHS is owned, its name will be destroyed after move
  • LHS prior value will be destroyed after move

6.2.5. swapref

Only for reftypes. Both LHS and RHS must be:

  • same type (exactly)
  • mutable variable names
  • owned (not borrowed)
  • must be able to take an exclusive lock on both immediately before the swap

6.2.6. call (change this to 'locking'? Happens in all expressions)

A series of locking steps are done around a call.

  • new lock frame added (?)
  • arguments processed left to right recursively (may involve a sub call)
  • shared lock taken on each parent
  • exclusive lock taken on each 'leaf' (final part of path)
  • 'loaned' returns ???

6.2.7. take

  • convert a 'mutable' name to a 'free' (unnamed) variable
  • variable must be 'owned'
  • can only 'take' local undotted variable names (no field member names)
  • the previous name is no longer available after the 'take'

6.2.8. usage?

  • iterators - return loaned items
  • 'object.myprotocol' - returns a loaned item? Exclusively locks parent object. This item can be created on the stack since it will not outlive the current scope; no allocation is required.
  • 'myprotocol from: take object' - returns a new owned item. Can never get back to enclosed item

7. More thoughts on ownership/locking

For function returns:

  • take - (default) returns an owned object. Arguments are locked during the call but released when the object is returned. Caller is responsible for destroying returned object (this will happen when it falls out of scope)
  • borrow - returns a borrowed object. Arguments are locked during the call and remain in place until the returned object falls out of scope. Caller is NOT responsible for destroying returned object (it lives inside the associated object) A borrow return REQUIRES a lock on the parameter where the returned value lives
  • lock - N/A

For parameters:

  • take - will take an exclusively owned object. It will no longer be valid in the caller
  • borrow - (default) will borrow the object. Cannot store or return it. Can only call methods on it
  • lock - will borrow the object, LOCKING it in the caller even after the call returns (lock is bound to the life of the returned object). Can store (in a new object only...) or return it. Can only call methods on it

Inline (code):

  • take - will take an exclusively owned object. Target (if a named address) will no longer be valid going forward
  • borrow - (default) will borrow the object. Will lock the target
  • lock - same as borrow.

In arguments:

same as 'Inline'. The specifier is optional and can be included to be explicit/readable. But if the specifier is missing, the ownership type specified in the function definition will still apply which may affect availability of a named var after the call (for lock and take)

It is an error to explicitly provide an incompatible ownership/locking type. eg. borrow arg, when the param is take. Can this happen? if you take arg at the call site for a locked arg, then the call site must keep a lock on the anonymous var until the end of scope.

  return t
}
# NOT allowed - cannot borrow local var and there is no lock over 
#   the returned type
myfunc: function {t: reftype.lock} out reftype.borrow is {
  t: makereftype
  return t
}
# NOT allowed. cannot return a borrow without a lock
myfunc: function {t: this} out reftype.borrow is {
  return t.myreftype
}
# OK - locking parent locks all chidren
myfunc: function {t: this.lock} out reftype.borrow is {
  return t.myreftype
}
mycall: function {t: reftype.lock} out refchild.borrow is {
  return t.myrefchild
}

# OK - locking is maintained in caller
myfunc: function {t: reftype.lock} out refchild.borrow is {
  x: mycall t # x is borrowed over a lock on t
  return x  # ok, x is escaping into an env that also locks t
}
