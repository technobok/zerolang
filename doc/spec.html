1. Introduction

This is a description of the Zero programming language.

2. Lexical structure

Program source code is parsed from file-like objects containing the program text.

Program text is comprised of characters that are combined to form tokens.

Each token is one of: whitespace, newline, eof, comment, delimiter, separator, identifier, label, keyword, reserved word, or string literal.

2.1. Character set

Source code is UTF8 encoded unicode text. Arbitrary non-null unicode characters are allowed in comments and string literals. The remaining elements of the language (identifiers, keywords, delimiters etc.) are limited to a subset of ASCII.

Some ASCII characters are reserved and are a syntax error outside of comments or string literals.

All non-printable characters (ie. that have a character code less than Space (32)) are reserved EXCEPT:

  • Horizontal tab (9)
  • Line feed (10)
  • Carriage return (13)

All characters with a code of Delete (127) or greater are reserved.

Other reserved characters:

  • comma , (44)
  • square brackets [ (91) ] (93)
  • backtick ` (96)

2.2. Whitespace and Newlines

Outside of string literals, the only valid whitespace characters are space, tab, linefeed and carriage return. Linefeed only denotes a new line.

When directly within a primary expression block, linefeeds are statement line separators and are explicitly part of the grammar. In all other locations, linefeeds are treated as whitespace only.

Whitespace can serve as a delimiter between identifiers that would otherwise be considered a single token.

2.3. Comments

Comments begin with a hash # (35) and extend up to (but not including) the following linefeed or else to the end of the file. They may contain any UTF8 text other than NUL (0).

2.4. End of file

A token marking the end of the file. All top level units are contained within a file-like object that ends when the parser returns an end of file token.

2.5. Delimiters

The following are the delimiter characters:

TokensDescription

{ }Block delimiters

Parentheses are used for enforcing or clarifying precedence. They surround a single expression or type reference. They are not required for function calls. Linefeeds are ignored (considered whitespace only) when immediately within parentheses.

2.6. Separators

The semicolon (;) is a statement separator along with newline -- when in primary expression blocks.

2.7. Identifiers

Identifiers are used to name constants, variables, functions/methods (including operators) types and numeric literals.

Identifiers consist of one or more letters, digits, special-characters and optionally a single dot in the case of numeric identifiers.

The special-characters are:

! $ % & ' * + - / < = > ? @ \ ^ _ | ~

Numeric identifiers optionally start with + or - then a decimal digit. The + or - is part of the literal and must immediately precede the remainder of the literal without intervening whitespace.

Numeric identifiers may contain a single decimal point (.) which must be followed by a decimal digit in all cases. They then continue up to the word boundary (whitespace, newline, delimiter, separator). Numeric identifiers may not have leading or trailing decimal points. Always use leading or trailing 0s if required to prevent this.

Numeric identifiers may include underscore characters for readability purposes in all but the first character and immediately after the decimal point. The underscores are ignored.

Identifiers are case sensitive.

Identifiers must be separated from surrounding tokens with whitespace or the delimiters or separators. Specifically note that mathematical operators are standard identifiers and so must also be surrounded by whitespace.

Exceptions and special cases:

  • some identifiers are predefined
  • some otherwise valid identifiers have been designated keywords
  • some otherwise valid identifiers have been designated reserved words

2.7.1. Predefined Identifiers

All numeric identifiers (integer or floating point) are predefined. All numeric identifiers start with a decimal digit after an optional sign.

2.7.2. Integer Identifiers

Integer literals follow the rules above for numeric identifiers.

Decimal integer literals default to 64 bits wide and signed. Non-decimal default to 64 bits wide, unsigned.

If an optional sign is provided (+ or -) or there is no base prefix specified, the numeric identifier is decimal. If a sign is not provided, a lower case numeric base prefix can be specified.

Leading zeros are not significant. Specifically, they do NOT denote octal notation.

Character literals are also an integer type. Their underlying storage is equivalent to u8 (for c8; ASCII) or u32 (for c32; Unicode).

2.8. Labels

Labels are used when assigning names to definitions. They are identifiers followed by a colon (:). The name assigned is the identifier only, without the colon.

Labels are case sensitive.

A shortened syntax is available where the definition refers to an item in scope that has the same name -- a single colon precedes the identifier. For example: :id is (somewhat) equivalent to id: id. The lookup of the value specifically excludes mapping to itself but otherwise searches the current environment hierachically as usual.

The colon must immediately precede the identifier without intervening whitespace.

2.11. String Literals

A string literal can be raw or interpreted. Both may have embedded newlines and have the same rules for handling the first and last newline and for stripping leading whitespace as detailed below.

2.11.1. Interpreted String Literals

An interpreted string literal is enclosed by double quotes ("). Within the quotes are zero or more literal UTF8 characters or escape sequences. A string may not include NUL (0) characters.

Valid escape sequences are:

\t\x09horizontal tab
\b\x08backspace
\\\x5cbackslash

\xhhunicode value; 8 bit hexadecimal
\uhhhhhhunicode value; 24 bit hexadecimal
\{ blocklines }interpolated block

Where each h is a single hexadecimal digit which must be specified. The expr expression will be evaluated and the result embedded in the string literal.

Backslashes may only appear as part of a valid escape sequence.

    dosomething "hello"        # even with comments
}\nmore string\n"

"an interpolated string showing the value of variable myvar.member: \{myvar.member}\n"

"an invalid string \a"      # syntax error, invalid escaped character

2.11.2. Raw String Literals

A raw string literal is free form text included without interpretation, except for special handling of the first and last line and leading whitespace.

A raw string starts with three or more double quotes and ends with the same number of double quotes. Within the quotes are one or more literal UTF8 characters. A string may not include NUL (0) characters.

An empty string cannot be specified with a raw string literal because the number of opening double quotes cannot be determined. Use "" to represent an empty string. Likewise, a single line raw string cannot start with a double quote.

2.11.3. String Newline and Whitespace Handling

Both string types (interpreted and raw) have similar handling of newlines and leading whitespace.

Literal newlines can be included, they are always encoded as LF (\x0a) regardless of the line ending used in the program source.

If the first line is blank after the opening delimiter (whitespace only), then that text including the first newline is excluded from the string literal.

If the last line is blank up to the closing delimiter (whitespace only), then that final newline and the text up to the closing delimiter is excluded from the string literal.

Also, any whitespace on a newline to the left of the closing delimiter, will be stripped from every line in this string literal where possible. The whitespace must match exactly (number and combination of tabs and spaces) to be stripped.

3. Definitions

A program is comprised of one or more units. Units are made up of zero of more definitions.

The entry point for a standalone program is the function called main in the initial unit. The name of initial unit is provided to the compiler as the compilation target and all referenced units are determined by tracing out references used in that initial unit.

Definitions are one of:

  • function
  • objects: unit, record, class, variant, union, facet, protocol
  • constant if definition
  • data
  • typeref (aliases?)

The descriptions of the syntax for each of these are described in their respective sections, but at a high level, they are built from keywords and blocks.

3.1. Blocks

The larger components of the language syntax are built from blocks.

Blocks are used wherever a series of optionally named items is required, eg. declarations, function arguments, function body, function calls, record/class/list/union members etc.

Blocks are delimitered by the beginning and end of file for a unit file and by matched braces {} in all other cases.

The general format of a block body is a list of items each being: a 'key' (an identifier with a colon or a keyword) then a 'value'. In some contexts, the key or the value may be omitted.

A shortened form of key and value is permitted when a variable of the same name as the required key holds the value. In this case, the identifier with a leading colon can be used to represent both the key and the value. 

# Given:
arg: 0

# then:
myfunc :arg

# is equivalent to:
myfunc arg: arg

Each item in the block body ends with the beginning of the next item (the next key) or by a delimiter or a newline or EOF.

Blocks of code also define a unit of scope.

The 'value' depends on context but is one of:

  • type declaration or type reference (Declarations and References)
  • an expression
  • for statement lines in function bodies, can be expression, assignment, swap, 'continue', 'break', 'return'

Summary of uses:

Special cases:

For calls and call-like syntax (ie. type references with generic parameters) the key for the first argument is optional (see Call Arguments and Declarations and References). All other arguments must be supplied by name.

For data literals, the optional key can be an integer. The expression specified will be the value at that index.

3.2. if definition

An if may be used as a definition at the unit level. The syntax is similar to an if expression, but there are different contraints. Each clause of an if definition must evaluate to a constant. Each then arm of an if definition is permitted to produce a constant item of a completely different type and the result of the definition will be the exact type of the first arm whose predicate evaluates to true.

4.4. Calls

A call executes a function by passing zero or more arguments and usually returning a value or owned reference.

They are one of the following forms:

  • the function name followed by its arguments (function call syntax)
  • the receiver object followed by a dot then the function name followed by its arguments (method call syntax)
  • the receiver object followed by the method name followed by it's single unnamed argument (operator syntax).

Calls may only be used in function bodies. Type specifiers may only be used where types are required - argument definitions, components of classes, records, units etc.

Note that parentheses are not used for function calls but as the call itself is an expression, the entire call may be surrounded by parentheses for grouping or precedence purposes.

A function call without arguments is still a function call (not a function pointer). Use .take on the function name to retrieve a function reference.

Named arguments can be specified in any order. Arguments are evaluated left to right as specified in the call.

Function parameters that have default values may be omitted in the call and will use those defaults in the body of the call. If the caller does specify that argument, it must be of the same type as the default value.

Generic arguments must be specified in a call or must be able to be derived from another argument declared to be of that generic type. They must be a member of the union or implement the protocol specified as the 'type' in the parameter definition.

Parameters declared with .take on their type take ownership of the supplied reftype argument in the function body. The variable supplied to a .take parameter is not valid after the call returns. Ownership transfer is performed implicitly; the caller does not need to call .take explicitly (though it may do so for clarity).

In a call, the 'primary' parameter is the first parameter listed in the in parameter list of the function definition. In a type specifier, the 'primary' is the first generic argument. If that is also the first argument supplied in a call or type specifier, then the parameter name may be omitted in some cases. When either:

  • it is a normal argument (non-take and non-generic)
  • it is a .take parameter and the supplied argument is anonymous (i.e. the result of a function call)
  • it is a generic parameter but a concrete arg is being provided (i.e. not a generic arg)

In all other cases, the parameter name must be supplied in the call or type specifier argument. e.g:

  • all arguments after the first argument, of any type
  • all .take arguments taking their value from a named variable

The argument name rules above are designed to:

  • unambiguously document the arguments to a call or type specifier
  • allow arguments in any order at the call site
  • clearly show when local variable ownership is being given away to a call (by requiring the parameter name to be specified for .take parameters)
  • not overburden calls with names for the 'primary' (or only) argument. The 'primary' argument is often redundant following the name of the function/type

Argument evaluations are also subject to ownership locking. See Ownership.

The expression for an argument value is first evaluated in the current scope. If the value is a single identifier (undotted) and it does not resolve up to the current module level, it is assumed to be within the scope of the argument type. This allows enum arguments to be specified without qualifier (as long as there is no local definition of that same identifier).

4.6.3. if

ifexpressions allow conditional execution. Arguments:

ArgumentDefaultDescription
whenYesCondition expression that results in a boolean
[var]:NoInitialisation expression bound to 'var'
thenNoBlock of code
elseNoBlock of code

An if expression has one or more clauses. Each clause is made up of one or more conditions (when) followed by a then block that will be executed if all prior conditions succeed.

Each when condition must be an operation that results in a boolean. when is the default argument and the when keyword may be omitted for the first condition specified immediately after the if.

The optional initialisation expression results in a variable binding that remains in scope for all conditions and clauses that follow it.

The conditions are evaluated in turn. If all conditions before a given then block return true, the then block will execute. If any fail, the tests for the remainder of the block stop (short-circuit) and the process continues after that related then block.

If no blocks of conditions completely succeed, the else block will execute, if present.

A successful clause will exit the if expression -- at most, one then or else clause will execute.

When if is used as an expression, the if must be exhaustive and each arm must evaluate to the same type or must not complete (return, break or continue). The result of the block is the result of the last expression of that block.

ifmay only have one else argument and it must be the last argument.

if x == 1 then {
    # will match if 'true'
}

# with initialisation expression
if x: func_that_returns_result when x != -1 then {
    # ...
} else {
    write "Got error \{x}"
}
# x is now out of scope

# an if expression
result = if x < 0 then {
    0
} else {
    1
}

# can be a single line
result = if x < 0 then 0 else 1

4.6.4. match

A match is used to dissect a union, variant, enum or generic parameter type into its child types. See also type narrowing.

Parameters:

ArgumentDefaultValue/Description
onYesTarget variable to be dissected: union/variant/enum/generic
caseNoExistence test. Value must be subcomponent id of the target variable
thenNoStatement or block of code (executed after successful match)
elseNoStatement or block of code (executed if no previous matches)

The case argument specifies the specific child type being tested. case is the default argument and the argument name can be omitted if it is the first argument.

An assignment expression may be specified. This will then become the target variable and all case clauses will specify subitems (ids) relative to this variable. The expression must evaluate to a union/variant/enum instance or must refer to a generic parameter. If the expression (RHS) is a named variable, that name will be unavailable in the match body to prevent aliasing or locking issues. The prepended colon : syntax may be used to nominate an existing variable name as a target within the match. The label in the assignment expression is within scope only within the match expression. If a labelled expression is provided, it must be the first argument to match. There can only be at most one labelled expression in a match.

The assignment expression binding may take ownership of the RHS by specifying the take keyword. The matching then or else clause will have full control of the target variable. After the match the RHS of the assignment expression will be inaccessable, ie. if it was a named variable, it can no longer be referenced.

If an assignment expression is specified, the match must be exhaustive.

If an assignment expression is not specified, all case clauses must specify the target variable and the subitem id as a single dottedid. All case clauses must specify the same target variable.

Multiple case clauses may be specified before a related then. (???)

A then clause must be specified after each group of case clauses. This clause will execute if the preceding case clauses match the subtype of the target variable. Within the then clause, the type of the target variable will be narrowed to the type(s) matched in the related of. If the matched type is a union/variant that has no type associated with it (null) then the narrowed type is 'null' and it is forbidded to use the target variable.

A then clause that terminates (with a return, break, continue) will narrow the target variable type (if a named variable) for the remainder of the scope after the match completes ie. that terminated subtype is excluded from the type definition of that variable going forward.

An else clause can be specified and will be executed if no previous case clause matches. It must be the final clause if present.

Only one successful then or else clause will execute.

An exaustive match can be used as an expression. The match is exhaustive if an assignment expression is specified or if the compiler can guarantee all subtypes have been specified in case clauses. When used as an expression, all then and else results must then be of the same type.

If the target variable is a generic type, then the match becomes a constant expression that is evaluated at compile time for each specialisation during code generation.

    # if this is a reftype, it is borrowed
} case none then {

}

    # ..
} case none then {
    # ..
}


match take optionalthing2 case some then {

    # optionalthing2 is owned within this block and can be used in any way
    # ...
} case none then {
    # optionalthing2 is narrowed to null
    # ...
}
# .. optionalthing2 has been destroyed...

    thing1

    # x is narrowed to the 'some' side

forexpressions allow iteration or repeated execution. It can be used to handle all looping variations:

  • for/while/loop - boolean pre-condition
  • for/loop/while - boolean post-condition
  • for loop - loop forever
  • 'for'/'in' style loop against iterators and function calls

Arguments:

Pre-conditions can be boolean expressions specified by the while keyword. This is the default argument and the argument name can be omitted if it is the first argument.

A single binding ([var]: call) can be specified before the loop. The 'call' is to an iterator (specified with in) or a standard function.

A binding to an iterator (in operation) is set up once on first loop, then each yielded value will be assigned (a borrow) to 'var'. The binding will fail (loop terminates) when the iterator calls return or comes to the end of the iterator function.

A binding to a standard function will call that function before each loop and assign it to 'var'. If the result is an optional and is null, the for will terminate. 'var' will be owned in the loop body as with a normal function call (TODO: unless function returns a borrow).

A single loop clause may then follow. The body of this clause will be executed if all conditions succeed. The loop may contain a continue (early skip to next iteration) or break (terminate for clause). Or a return.

Postconditions can be boolean expressions specified by the while keyword after the loop. They must succeed for the for to continue on a new loop.

Any combination of pre and post conditions and bindings may be specified on a single for.

Order matters, conditions/bindings can short-circuit on failure and will stop processing subsequent tests.

forrequires at least one argument (condition or loop). for <condition> will loop until condition is false; for x: operation will loop until operation is or returns null (discarding x each loop); for loop { ... } will loop unconditionally and indefinitely.

forcan be used as an expression. The result will be a list of the final values of the loop statement.

# normal while 
# (precondition)
for x == y loop {
    # ...
}

# normal for..while 
# (post condition; at least one iteration guaranteed)
for loop {
    # ...
} while x == y

# 'foreach' style, myiterable is an iterator function
for x: in myiterable loop {    
    # x is available as a borrowed reference here
    # the myiterable object is unavailable (locked until 'for' completes)
}

# 'foreach' style combined with boolean condition
# only loops for x = 0, 1, 2, 3, 4
# TODO: skip 'range'? Make integers iterable?
for x: each 10 while x < 5 loop {
    # ...
}

cont: true
for x: in iterator while cont loop {
    # can specify multiple conditions
    # they all must be true
    # will short-circuit at first false condition
    cont = false
}

4.7. .take on Function Names

To obtain a function reference value (rather than calling the function), use the .take dotted operator on the function name. This is consistent with the existing .take semantics for value types.

myfunc: function {a: i64} to i64 is { return a }

# obtain a function reference without calling the function

.takeon a spec name is a compiler error because specs have no value — they are types only. Use .take on a function name that matches the spec.

5. Type System

5.1. Overview

A type (or data type) is a group of values and the operations that can be performed on those values.

All function parameter definitions and object fields must have a type specified. Variables and top level definitions have types inferred from their specified values.

All object types may have methods defined on them.

There are two broad categories of types: value types and reference types.

5.1.1. Value Types (valtype)

  • pass-by-value
  • includes all fixed numeric types; and arrays
  • 'record' is the user defined value product type
  • 'variant' is the user defined value sum type
  • 'facet' is the user defined value interface type
  • records and variants can only contain other value types
  • records have constructors
  • no destructor (why?) (default destructor only)
  • no other resources that need freeing (eg. can't have a file, db handle or any reftype as a member)
  • single, contiguous block of memory of a fixed size (for all instances of this type)
  • variant is size of tag + size of largest possible member
  • mutable (but private members may be protected with accessor methods)

5.1.2. Reference Types (reftype)

  • pass-by-reference
  • eg String, File, List, Dict, protocol, union, (others?)
  • 'class' is the user defined reference product type
  • 'union' is the user defined reference sum type
  • 'protocol' is the user defined reference interface type
  • can contain value types -- the value types will be embedded
  • can contain other reftypes (reftypes will not be embedded - store reference to them only). But must have ownership.
  • classes have constructors
  • can have destructor (but it doesn't destroy memory, compiler handles that)
  • single, contiguous block of memory of a fixed size (for all instances ?? what about List?) but may have references to other reftypes
  • mutable (but private members may be protected with accessor methods)
  • a reftype variable must always contain a valid reference. The swap operator can be used to take a new reference and return the old one. The destructor can be called but then the variable is no longer valid (unless reassigned).

5.1.3. Product types

Product types are a compound type that contain one or more fields of specified types. record is the value product type. class is the reference product type.

5.1.4. Sum types

Sum types hold a single value that can be one of a fixed list of types. variant is the value sum type. union is the reference sum type.

Sum types are available at run time but can also be used to specify generic parameter constraints.

Type Narrowing

The compiler keeps track of the type of every expression at every point in the program source.

Sum typescan be 'narrowed' back to their specific child types by using match expressions.

Within a successful then clause of a match, the target variable will be narrowed to the type(s) listed in the related of clause(s).

Also, a match clause that exits the current scope (using a return, break or continue) will also 'narrow' a named target for the remainder of that scope -- ie. the type of that named target will exclude the subtypes where a matching of clause has resulted in exiting the current scope.

5.1.5. Interface types

Interface types specify a list of methods names and signatures. Items may implement an interface by supplying implementations of each those methods listed in an interface and explicitly listing the interface in an as member.

facetis the value interface type. protocol is the reference interface type.

Interface types are used for polymorphism at runtime but can also be used to specify generic parameter constraints.

5.1.6. Generic types

Classes, records, unions, variants, protocols, facets, units and functions/specs can be generic types.

All defined unions, variants, protocols, facets and numeric types have a compiler generated 'generic' method, that allows that defined object to be used as a generic constraint when used as a type in a field or parameter. Also, 'generic' is a predeclared type that takes a 'from' parameter of type 'any' that can be used to specify generic fields or parameters.

Any object (unit, function/spec, union, variant, class, object, protocol, facet) that has one or more such generic fields or parameters is a generic type.

Generic types do not exist at runtime. Top level code must define concrete types for all generic parameters (explicitly or implicitly). At that point, a specialised implementation is created, using the generic type as a template. This specialisation will be shared by other client locations that use this definition with the same type parameters.

Numeric generic parameters are parameterised by the numeric value. The top level code must provide a constant submember to make the generic type concrete. Unlike the other generic types which are parameterised by a type. Note that when passing a generic argument, the type itself is passed to the generic parameter - an instance is not created; create is not automatically called is would be done for non-generic arguments. To make this optionally obvious at the call site, the generic method can be 'called' on the argument.

A generic parameter may have a default type. The call site may override this default type. In all other cases, the call site must specify the types of all generic parameters being used in that type by either:

  • explicitly providing the generic argument with a concrete type that is a member of the declared union, interface etc.
  • explicitly providing the generic argument with a generic type reference at the call (when used in a caller which is itself generic)
  • implicitly, by providing a concrete value to another argument/field that has been defined to be an instance of this generic parameter/field. The compiler can then infer the type of the generic parameter from that value. This is the preferred syntax where possible

Note that attempting to store a reference type into a value type via a generic type will fail to compile.

The any type is a compiler defined union of all possible types declared in the current program. any can ONLY be used for generic definitions, not for creating fields or parameters.

Examples:

# an option type

    some: t
    none: null
} as {

# because 'none' does not take a concrete argument with a specific type,
# so must be specified
# none takes only a single generic argument so the argument name can be 
# omitted

Embedding a generic type in another type:

5.1.7. Visibility

Definitions within blocks defining: facets, protocols, variants, unions, records, classes and units; may each belong to 0 or more interfaces (protocols/facets).

There are predeclared units called 'public' and 'private' for each but new definitions of interfaces by any other name can be created.

The default definitions for both 'public' and 'private' include all members . But the 'public' name can be redefined within the block body to create a unit that include a different subset of methods and fields. It is not possible to redefine 'private'.

The methods/fields accessible from outside the 'item' or 'unit' (static) namespace are those included in 'public'. By default, 'public' will also list the methods/fields that are available on item instances. But instances can be modified to present a different public interface by specifying 'this.myinterface' as the return type in all constructors of the instance. 'myinterface' will then be the public protocol/facet available on instances. Note that 'units' do not have instances so their public interface is always 'public'; it would not be useful to define an interface with a different name on a 'unit' (other than for inclusion in 'public').

'private' is the interface presented on 'this' when inside the definition of an 'item' or 'unit'.

The 'private' interface can also be explicitly referred to in 'friend' functions/'items'. A function/method accepting 'myitem.private' will be able to access the private interface of a 'myitem' instance passed into it. But note that only a method defined in 'myitem' itself or another friend of 'myitem' can pass the private interface of 'myitem' to this friend function/method.

Types must be public to the caller to be directly instantiated. But if an object is acquired (eg. via another public function in that unit), the public fields and methods of that instance are accessible.

5.1.8. Ownership

Ownership is determined by how the variable is created/declared or received. Each instance has exactly one owner at most. When the owning variable falls out of scope, the object is destroyed. Top level objects (ie. data) never fall out of scope.

See doc/ownership.pdoc for the full ownership and locking model.

Ownership states:

  • Owned - the instance is owned at this location. All instances have exactly one owner. Reftype constructors return owned instances. For valtypes, every copy is a new owned instance.
  • Borrowed - only for reftypes. The instance at this location has been borrowed from an owned reftype. A function parameter borrows by default (borrow is the default for in parameters).

thiswithin object methods is always borrowed (you cannot own yourself).

Instance fields are always borrowed even if the parent is owned. Valtype fields can be reassigned freely with =. Reftype fields must be set in the constructor and can only be changed later with swap -- this ensures that both locations always contain a valid instance throughout the lifetime of the parent object.

Aliasing is never permitted for reftypes (except for data). A reftype can only appear once as an argument or as a receiver in any single function call (enforced via locking).

Ownership changes only through function calls (.take parameters and return values) and the .take method on runtime instances. Normal arguments are borrowed within the called function. In match statements, the target is borrowed in the body by default; match with .take takes ownership with type narrowing. .take arguments pass ownership into the called function: the caller's name is invalid after the call.

Unnamed instances (the result of a function call not assigned to a label) are transient. They are destroyed at the end of the current statement unless assigned or passed to an owned parameter.

Ownership Type Attributes on Functions

Ownership is specified on function parameters and return types using dotted type paths:

  • type.take - for in parameters, ownership is taken from the caller. The caller must provide an owned reftype. The function must store or destroy the value before it returns. For out return types, .take is the default (function returns a new owned instance to caller) and does not need to be specified.
  • type.borrow - for in parameters, ownership is not taken. The function can call methods on the parameter but cannot store or destroy it. .borrow is the default for in parameters and does not need to be specified. For out return types, .borrow means a borrowed instance is returned. The parameter housing that instance must be marked .lock.
  • type.lock - for in parameters, this parameter will be locked by the return value for its lifetime. Multiple parameters may be marked .lock; all locks transfer to the return value. It is illegal to specify .lock on a return value. It is illegal to .lock a parameter when there is no return value.
Ownership Methods on Instances

.takeand .borrow can also be used as compiler-generated methods on runtime instances:

  • x.take - takes ownership from the variable x. The name x is invalid after the call. Calling .take on an expression that is not used (not assigned or passed) destroys the instance.
  • x.borrow - returns a borrowed reference to x. The variable x is exclusively locked for the lifetime of the borrowed reference.

5.1.9. Locking

Locking is an orthogonal attribute tracked alongside ownership. It prevents aliasing and data races at compile time. Locking is purely a compile-time mechanism with zero runtime overhead.

Lock states:

  • unlocked - the instance is not locked (default)
  • exclusive - an exclusive lock is held over this instance
  • shared - a shared lock is held over this instance

An exclusive lock is required to pass an instance as an argument to a method. A shared lock is taken on the parent when an exclusive lock is taken on a child/grandchild member.

Accessing a reftype field (eg. x: obj.field) produces a borrowed reference to that field. The borrowed reference holds a shared lock on the parent and an exclusive lock on the field.

For a function that returns a borrowed instance: that instance retains an exclusive lock over the .lock parameter where the returned object lives, until the borrowed object falls out of scope.

Locking Algorithm

During a call, locks are taken/checked/assigned on all arguments and return values:

  1. Each argument is processed left to right as specified in the call.
  2. If the argument is a top level (data) item, no lock is taken (immutable, freely shared).
  3. Otherwise, an exclusive lock is taken on the argument.
  4. If the argument is a dotted chain, a shared lock is taken on each parent and grandparent in that chain.
  5. Additional shared locks on already shared-locked parents are permitted.
  6. If there is already an exclusive or shared lock on the argument being passed, it is an error (prevents aliasing).
  7. If the argument is a sub-call, locking is applied recursively; after that call, sub-call argument locks are cleared or assigned to the sub-call result.
  8. After all arguments are processed, the receiver is locked (exclusive lock on each component of the receiver chain).
  9. The call is made.
  10. After the call, for any .lock parameters, their exclusive locks (plus shared locks on parent items) are assigned to the returned value. All other argument locks are freed.

Locks are also taken on targets in match, if and for to prevent mutation in the body. match can also .take its target, in which case the target is owned with the narrowed type in the match bodies.

When an instance falls out of scope, all of its locks are freed. If the variable is borrowed, the borrow is dropped. If the variable is owned, it is destroyed.

5.1.10. Declarations and References

New user defined types are declared using the class, record, protocol, union, variant, function (for specs) keywords. Each user definition must be bound to a name (no anonymous or inline use). All must be defined directly within a unit.

Every local-variable, constant and each subcomponent of the above user defined types (ie. fields, methods, parameters etc) are declared using a name binding to a type reference (of the form name: typereference).

The declaration name must be a valid identifier.

The declaration typereference follows the name. It is:

  • the name of the type for standard fields and parameters (non-generic, non-default, non-taken types)
  • a constant numeric expression (for fields and parameters with default values)
  • the name of the type followed by each generic argument using syntax similar to function calls (to specify a generic type). As with function calls, the name of the first generic argument may be omitted
  • a type followed by .generic (eg. any.generic) to create a generic type parameter. For functions and protocols, the generic parameter is declared in the body. For records, unions, classes and variants, the generic parameter is declared in the as section.
  • a type followed by .take, .borrow, or .lock in function parameter definitions to specify ownership and locking semantics. See ownership.

5.2. Unit Types

Programs are constructed from one or more units.

A unit is used to hold logically related constant definitions. A unit must have a name and creates a new namespace.

Each definition within a unit may or may not be public. See Visibility.

A source file is a single unit. The name of that unit is the name of the file excluding the file extension. Unit names have additional restrictions over standard identifiers so that they can be represented as filenames across filesystems of differing capabilities. They must:

  • consist only of lowercase ASCII letters and digits

A file unit can have 'implementation subunits' by placing file units within a directory of the same name of the current unit. The name of that subunit is its filename excluding the file extension. To be loaded, the subunits must be explicitly referenced by name in the 'parent' unit. They are always private to the parent unit.

The never type is a type that is used to represent an expression that never returns. The return value of the expressions: break, continue and return are of type never within the block where they are called.

A never type is permitted in if or match that are used as expression values. The never type is ignored when determining the type of the entire expression.

Any statements following a never typed expression in the same scope is dead code and is raised as a compiler error during the type checking phase.

5.3.10. typedef

A typedef wraps an underlying type to provide a new type. The new type has no run-time overhead beyond the underlying type, it is simply a compile time construct.

To define a new typedef, define a class or record with a single concrete field of type typedef with a subtype (argument) of the underlying type. The typedef may also have generic parameters and functions/methods that operate on the new type. It may satisfy protocols (with as) using a combination of methods defined on the new type and on the underlying type.

typedefis a predefined type that takes a single generic argument of type any.

A typedef must not have any constructor methods and all typedef methods can not call the one true constructor. Wrapping is handled by the compiler and type system and there is no way to intervene.

All generic parameters in the typedef MUST be used in the this definition line. That is, only generic parts of the underlying type may be referenced. Any generic parameters must be able to be automatically derived from the source item.

Any methods defined will override methods of the same name on the underlying type. All unshadowed methods of the underlying type will be exposed on the typedef.

The typedef may only use public methods and fields of the underlying type.

To create a reftype typedef use a class or a union. The underlying type must be a reftype.

Create a valtype typedef using a record or a variant. The underlying type must be a valtype.

The typedef has a compiler generated constructor called create that will return a borrowed instance of the typedef.

The typedef retains the underlying type and so can continue to be passed to functions that expect the underlying type as is. But the opposite is not true - a function expecting the new typedef must explicitly be given the wrapped type.

5.3.11. tag

A predefined generic type for specifying the discriminator in unions and variants. tag is a predefined identifier, not a keyword.

- the data labels must exactly match the union/variant subtype names (one-for-one) - all data values must be unique - the data values are used as the discriminator constants

- the compiler auto-generates sequential values starting from 0 - the specified numeric type is used for discriminator storage

- in a union (not a variant) - there are only two subitems - one of the subitems must not have data associated with it (null member), the other must have data associated with it

5.4. Predefined Methods on Types

5.4.1. take (method)

takeis a compiler-generated method predefined on all variable names that refer to a reftype. It takes ownership from the variable, converting it to an anonymous owned value. The variable name is invalid from this point forward.

The take method is zero overhead -- it is a manipulation of the compile time type system only. Use of the .take method is optional when passing to .take function parameters; the compiler will implicitly take if required.

.taketakes no arguments.

.takecan be used for defining local variables -- the new variable takes the value and the old name becomes undefined. Calling .take as an expression that is not used (not assigned or passed) destroys the instance.

See also the ownership.

5.4.2. borrow (method)

borrowis a compiler-generated method predefined on all variable names that refer to a reftype. It returns a borrowed reference to the receiver. The receiver is exclusively locked for the lifetime of the borrowed reference.

The borrow method is zero overhead -- it is a manipulation of the compile time type system only. Use of the .borrow method is optional when passing to function parameters (parameters are borrowed by default).

.borrowtakes no arguments.

.borrowcan be used for defining local variables -- the new variable borrows the value and the source name is exclusively locked while the new name is in scope.

See also the ownership.

5.4.3. lock (method)

lockis not a runtime method on instances. It is a type attribute used only in function parameter declarations (type.lock) to indicate that the parameter will be locked by the return value. See locking for details.

5.4.4. generic (method)

genericis a 'class method' predefined on all union, variant, protocol, facet and numeric types. It explicitly converts the receiver type to a generic argument. This makes it obvious at the call site that a generic argument is being passed and that a create method is not being called on the type. The generic method is zero overhead -- it is a manipulation of the compile time type system only. Use of the generic method is optional, the compiler will always pass a type to a generic receiver.

generictakes no arguments.

The generic method cannot be used for defining local variables, only on arguments.

See also the built-in generic type or a description of generic types in general.

5.7. function

The function keyword defines a:

- standard function: a signature with a body defined at unit level or within an object - a method: defined within an object and has a this receiver as the first argument - iterator function: using yield - 'spec': a signature only; no body (only at unit level)

It may also be generic by having generic parameters.

Functions are structurally equivalent. Two function types are considered equivalent when, in both definitions:

- the names and types of all non-receiver parameters are the same - the return types are the same - if there is a receiver (this), it must refer to the same object. The name of the receiver parameter is ignored for the comparison

Function definitions must be equivalent when (re)assigning Function References.

See also Specs.

5.7.1. Function Definition Arguments

Arguments to function:

ArgumentDefaultDescription
inYesBlock of standard and generic parameters
outNoThe return type of the function
over/each?NoThe return type of each item being yielded
asNoFacet of parent object after method call
isNoFunction body expression/block

- in is optional; omitted when no parameters are required - out is optional; omitted for procedures and iterators - over/each?? is optional; required for iterators, omitted for standard functions - as is optional; only permitted on methods and only those providing typestate - is is optional; omitted when defining a spec.

A function defined with no arguments is a spec that takes no arguments and returns no value. It only has side-effects.

The in argument takes a brace delimited block of parameter definitions of the form name: typereference. See Declarations and References for details. A parameter must not be private (_) but may have ownership attributes (type.take, type.borrow, type.lock) or be generic (type.generic) or it may have a default constant numeric value or a default function reference value.

The order of the parameters is significant. The first non-generic parameter may be the 'primary' argument that can be used without specifying the parameter name in a call. See Calls for details. The first generic parameter may also be specified without name in type specifiers.

A function usually will return a value. The type of the returned item is specified by the argument named out. Ownership of a returned reftype value after the call implicitly lies with the caller (.take is the default for return types and does not need to be specified). A function may return a borrowed reference by specifying type.borrow as the return type; in this case, at least one parameter must be marked type.lock.

The is parameter to function specifies the body. It may be a single expression or a block of code. Each line of code is a local variable definition or a statement. All arguments supplied can be referenced by name but are read-only and cannot be reassigned. The function result can be the value of the last expression in the body or the return statement can also be called in the body for early return of a value. Iterator results are specified by using the yield statement.

# function takes an int and has no return value
myfunc0: function {a: u32} is {
    print a  # return value of call is ignored
}

# function takes an int and returns an int
myfunc1: function {a: u32} out u32 is {
    a
}

# function with no parameters that returns an int
zerofunc: function out u32 is {
    0.u32
}

# function with no parameters as no return value
sayhello: function is {
    print "hello"
}

A function may also take ownership of arguments. To do so, the parameter type uses the .take attribute (type.take) and the type must be a reftype or generic type that may include a reftype.

The caller must own the variable before the call. The function body is free to use the taken argument as if it created it itself (i.e. can be stored, returned, destroyed). The caller's variable name is invalid after the call.

Passing a value type via a .take parameter of a generic type passes a copy as usual. This allows the same notation to be used for generic container types regardless of the type of subcomponents being stored (eg. for Lists). In all other cases, it is an error to pass valtypes to .take parameters.

.takearguments from a named variable cannot be specified without the parameter name at the call site even if they are the first argument. This is to make it obvious that ownership is given up at the call site. .take arguments from an anonymous value (eg. from the result of a direct function call) can be specified without the parameter name since the value would be inaccessible/destroyed after the call in any case.

taker: function {arg: reftype.take} is {
    # arg is owned in this scope
}

# call site:
taker arg: myreftype
# myreftype is unavailable in the current scope from this point

# passing a lock parameter and returning a borrow:
getter: function {t: myclass.lock} out refchild.borrow is {
    return t.myrefchild
}

In the in block, the function argument types specified (after the function argument label), can be:

- a builtin type (eg. u8, i32, f64, c8) - a numeric compile time constant (this is also a type). Function argument types can NOT be:

- a bare function name (use .take on the function name to obtain a function reference for a spec argument)

5.7.3. Specs

A 'spec' is the type of a function. When defining a complete function (with an is argument), both a spec and a function value are being defined.

If the function definition has no body (is argument) then only the spec (function type) is being defined.

A spec (without body) must be defined at the unit level, within a protocol, or within a record/class is section (as a function pointer field). They cannot be defined in code blocks (function bodies).

Specs and functions use structural equivalence: if two function definitions have the same parameter names and types, and the same return type, they are considered type-compatible. This is one of the few exceptions to Zerolang's general preference for nominal typing.

Function references are valtypes — small, copyable pointers to immutable program text. They can be freely copied and do not participate in ownership tracking beyond simple value semantics.

It is not possible to specify a method spec without a function value.

5.7.4. Iterators

An 'iterator' defines a function that can yield a series of values. This is a function with a yield argument specifying the type of each value. It must not have an to argument.

Creates an object definition that can iterate over a set of values. Instances can not be created directly, rather they are created by using the generator in a for loop. The yielded object is borrowed only for the body of the for loop.

The is block should contain one or more yield statements to return intermediate values. Any return statement or the end of the definition will end the iteration and the calling do loop. A return statement in an interator must not have a return value.

The generator is implemented by creating a state machine to track variable state including the progress through the generator between subsequent calls. The created object has a single next method that returns an optional instance of the type specified in the generator's yield argument.

5.7.5. Methods

Methods are a specialised form of function. They are defined similar to a normal, named function call but they must be defined in the namespace of the receiver definition and the first argument must be of type this.

This function will then be considered a method of that object. The caller may call that method using method syntax (myinstance.mymethod args...). Calling with normal function call syntax is also still permitted (myobject.mymethod myinstance args...).

The argument designated as this is unusual in that it is the receiver instance itself (i.e. directly addresses the instance, even value types - not pass by value as is usually the case).

Methods can change subcomponents of the receiver. Methods can directly change entire value objects (by assigning to this, but has to be same type).

Methods can be attached to user defined types, usually in the same module as the type definition. But they can also be attached to system types, type aliases and user defined types defined in external modules. In these cases, the method is only available to the local module as an extension method over the type. They do not change the type itself. They must also not collide with existing methods on the type. This may cause a maintenance burden (changes to the external type may cause code that previously compiled correctly to now fail) so should be used with caution.

5.7.6. Constructor

There is only one true low-level constructor for an object and it is defined automatically by the compiler at meta.create. It will provide the memory for the object and set any members directly from parameters supplied or from any default field values. meta.create is only available directly within the object definition but if no user defined constructor is defined with the name create, then the meta.create will be exposed as object.create.

User defined constructors can be defined on an object. These constructors return type this and create instances by calling meta.create directly, or by calling other user defined constructors which eventually call meta.create.

Constructors must expose (as parameters) any generic fields in the object. It must be able to be determined at the call site the actual type of the item being created (although this can be explicitly passed to a higher level by passing a generic variable at the call site into the constructor.

myobj: record {
    a: u32
    b: string

    create: function {a: 0.u32 b: string} out this is {
        return meta.create :a :b      # no stuttering... can use default args
    }
}

5.7.8. Function References

A function reference is a variable that stores a pointer to a function. It may be called with the same syntax as used when calling a standard function. It may be reassigned to a reference to an equivalent function (one with the same parameter names, types, and return type). It must always have a valid function as a value.

A function reference variable cannot be defined at the unit level (unit level declarations are immutable).

To obtain a function reference value at run time (within a function body), use .take on the function name.

Function references can be used as:

  • local variables
  • fields of an object (in the is section)
  • function parameters
  • function return values

For local variable function references, declare and assign using .take:

myspec: function {a: i64 b: i64} to i64
add: function {a: i64 b: i64} to i64 is { return a + b }

main: function is {
  ref: add.take                # local function reference
  result: ref a: 1 b: 2       # call through reference
  ref = subtract.take          # reassign to another compatible function
}

For field function references in record/class is sections:

  • define a function in the is section — it becomes a per-instance function pointer field with that function as the default value
  • OR define a spec (no body) in the is section — it becomes a per-instance function pointer field with no default; a function reference must be passed to meta.create using .take
  • the function reference can be reassigned to an equivalent function at any time using .take

Functions in the as section are shared methods on the type, not per-instance fields.

To create a function reference as a function return value:

  • define a function spec at unit level
  • in the function definition, use the name of the spec as the return type
  • in the body of the function, return an equivalent function reference

The above applies to standard function references (non-method pointers; no this parameter). Method pointers have additional restrictions:

  • all methods must be defined within the object (this is not available outside the object definition)
  • method pointers must be declared by referring to an existing method
  • so, method pointers always have a default value (there is no way to define a method pointer spec)
  • method pointers can only be replaced by another equivalent method within the object
  • method pointers cannot be function return values (they can never escape the object)

5.10. protocol

A protocol is defined as a list of function specs. The receiver (this) is always omitted from each spec definition (NO!!!, this must be specified. Static methods can be part of a protocol). Note that a protocol cannot include definitions for constructors or destructors because they are not on the object, only in it's namespace.

Protocol methods cannot have function bodies. The protocol defines the interface only without any implementation. If additional methods over a protocol object are required, a type definition can be defined over the protocol object and addition methods (that can only interact with the object via protocol methods) can be defined.

Types (records, classes, unions, variants and enums, including protocols themselves) can claim support for one or more protocols by naming each protocol in an as clause and implementing (or relisting for protocols) all specs listed in that protocol as methods. All protocols supported must be given a label that exists in the same namespace on the object as its fields and methods.

Protocols can be used:

- to create protocol instances: see protocol instance creation. - as generic constraints: see Generic types

    read: function {:this b: buffer} out boolean

5.10.1. Protocol Instance Creation

Fields, variables and parameters can be declared to be instances of a named protocol. Protocol instances can then be assigned to those fields, variables or call arguments.

Protocol instances can be created by accessing the named protocol member in the 'as' clause of an object that implements a protocol.

Protocol instances are reference types. They contain a pointer to the original data, a pointer to a vtable with function pointers for each spec, and a destructor pointer (NULL for borrowed instances).

In v1, protocol instances are always borrowed: they hold a pointer to the original data without taking ownership. For value types, the address of the original is stored; for reference types, the existing pointer is stored. The protocol instance struct is heap-allocated and freed when it goes out of scope, but the underlying data is not affected.

Accessing a named object's protocol in a named context (ie. assigning to a new variable; in a 'give' argument; in a swap statement), will cause the original name to be inaccessible from that point. The new named context will have the same ownership as the original name.

If the protocol instance is created in a lower scope (eg. a 'do' clause; passing as a standard loaned argument) then after that scope, the created protocol object is destroyed and the underlying variable is again available for use. 

reader: protocol {
    read: function {:this b: i64} out i64
}

myfile: record {
    fd: i64
} as {
    myreader: reader
    read: function {f: this b: i64} out i64 is {
        return f.fd + b
    }
}

use_reader: function {r: reader} out i64 is {
    result: r.read b: 5
    return result
}

main: function is {
    f: myfile fd: 10
    r: f.myreader        # r is a borrowed protocol instance
    print "\{use_reader r}"  # prints 15
}

5.12. union

For controlling the discriminator, see also the tag type.

Unions are sum-types that can hold exactly one of their child types at any given time. A discriminator (tag) is auto-generated for each subtype. A union is a reference type — always heap-allocated with a uniform two-word layout: a tag and a void* data pointer. All member types may be reference types or value types. Valtype payloads are boxed (heap-allocated copy). Reftype payloads are stored as pointers directly.

Methods can be defined on the union body. These methods operate on the union type itself. To access subtype-specific data, the union must first be narrowed via match.

Arguments for union:

ArgumentDefaultDescription
isYesBlock of independent subtypes, generic parameters and associated methods
asOptional block for tag specification and shared attributes

A sum-type definition is a block listing one definition for each subtype, providing a name and a type.

If there is no associated type for a subtype, the type is specified with a null.

myunion: union {
    u1: null       # no user stored value. Discriminator only
    :u32           # shortened syntax for 'u32: u32'
    u4: string
    u5: myclass    # a class defined elsewhere
}

5.12.1. Constructing union instances

After a union type has been defined, instances are constructed using a per-subtype constructor accessed via dot notation: myunion.subtype expr. Each subtype automatically gets a constructor. Null subtypes take no argument. Valtype arguments are copied (boxed) into the union. Reftype arguments require ownership transfer via .take or by passing an anonymous/literal value.

result: union {
    ok: i64
    err: string
    none: null
}

a: result.ok 42                     # valtype: value is boxed
b: result.err "something went wrong" # reftype: string literal, ownership taken
c: result.none                      # null subtype: no argument

# named argument form
d: result.err from: mystr.take      # explicit .take transfers ownership

5.12.2. Ownership

Union instances are owned reference types. The same ownership rules that apply to classes also apply to unions:

  • .take transfers ownership of the union itself
  • .borrow borrows the union (locks it)
  • swap exchanges two union variables
  • Aliasing prevention applies (same as classes)
  • The union owns its payload — destruction frees both the payload and the union instance

The compiler generates a per-union destructor that switches on the tag to correctly free each subtype's payload: free() for boxed valtypes, the appropriate destructor for reftype payloads, and nothing for null subtypes.

5.12.3. Match on unions

The match statement dispatches on the union's tag. All subtypes must be covered by case clauses, or an else clause must be present. Inside a case clause, the payload is accessed via dotted path on the union variable.

describe: function {r: result} out string is {
    match r
        case ok then {
            # r.ok accesses the i64 payload (unboxed)
            print "value = \{r.ok}"
            return "ok"
        }
        case err then {
            # r.err accesses the string payload (cast)
            print "error = \{r.err}"
            return "error"
        }
        case none then {
            # no payload to access
            return "none"
        }
}

Accessing subtype data outside of a match clause is a compile error — the union must be narrowed first.

5.12.4. Creating sum-type instances

Creating/setting a union value:

myclass: class {a: u32 b: u32}

myunion2: union {
    t1: null      # discriminator only
    t2: u32       # a valtype in addition to the discriminator
    t3: string
    t5: myclass
}

v1: myunion2.t1                       # null subtype, no argument
v2: myunion2.t2 0                     # valtype, boxed
v3: myunion2.t3 "hello"              # string literal, ownership taken
myclassi: myclass a: 1 b: 2
v4: myunion2.t5 myclassi.take        # .take transfers ownership
v5: myunion2.t5 (myclass a: 1 b: 2)  # anonymous instance, no .take needed

myenum_vals: data { a: 0 b: 1 }

myvariant: variant {
    a: u32
    b: f32
} as {
    tag: myenum_vals.tag
}

Unions are also used to define generic types along with protocols. any is a system defined union that includes all possible types used in the current program.

smallint: union { i8: null u8: null }

5.13. variant

Variants are sum-types that can contain only one of it's child types at any given time. A descriminator is usually generated or set for each case. A variant is a value type and all child types must be value types. A variant is always as large as its largest containing type plus the descriminator.

Arguments for variant:

ArgumentDefaultDescription
isYesBlock of independent subtypes, generic parameters and associated methods
asOptional block for tag specification and shared attributes

See also the union section. Other than variants being a value type, all other instructions for a union also apply to a variant.

5.13.1. Enumerated Types

A variant or union without additional data on each variation creates an enumerated type based on an underlying numeric type. To specify that there is no additional data on each arm, null can be specified as the type, or better, leave the type specifier out completely. A constant numeric value can be specified for each arm and provided it is distinct for each option and can be stored within the tag type, then that value will be used as the discriminator value.

The constant values are accessed by dot notation (they are accessed, not constructed). Each value must be unique and if defined without a value, will be one higher than the previous value or start from 0.u8.

The default underlying type is u8 if no values are specified. A numeric constant of a different type can be specified if required. All numeric values specified must be of the same type. Any unspecified values will be one higher than the previous value using the same type as the previous value.

Any member value that collides with another or overflows the underlying type will cause a compilation error. 

# underlying type defaults to u8
# first unspecified value starts at 0.u8
myenum: variant {
    FIRST: null
    SECOND: null
}

# can specify unique values if required. Subsequent unspecified members will be
# 1 higher than the largest previous member. A different numeric type for the
# tag can be specified in the 'as' block.

myenum2: variant {
    FIRST: null
    SECOND: null
    THIRD: null
} as { tag: u16.tag }

# can be out of order but is usually poor style
myenum3: variant {
    SECOND: null
    FIRST: null
    ANOTHER: null
}

# usage

v: myenum.FIRST     # myenum.FIRST is a value (of type myenum)

if v.FIRST then {  # v.FIRST is a predicate
    # ...
}

# effectively equivalent to...

if v == myenum.FIRST then {
    # ...
}

Enum-style types (where all subtypes are null) should use `variant`, not `union`. A union is a reference type requiring heap allocation, which is pointless for a type that only carries a tag discriminator. Using `variant` makes the enum a stack-allocated value type with no allocation overhead.

The built-in 'boolean' type is a system defined enumeration. The boolean values may be created directly by name or indirectly via the comparison or logical operators.

# boolean types are a system defined enumeration equivalent to:
boolean: variant { FALSE TRUE }
true: boolean.TRUE
false: boolean.FALSE

5.13.2. Using sum-type instances

A sum type instance can be tested to see if it is a specific subtype. A predicate method is generated for each member named as the member name followed by a question mark. The method returns a boolean. 

myunion: union {
    t1: u32
    t2: string
    t3: null
}

x: myunion t2: "hello"

if x.t3 then {
    print "x is an instance of subtype 't3'"
}

To use submember data from a union instance, it must be narrowed (or destroyed?). Assignment to a union's subtype may only be attempted in an if expression. If successful, the subtype can be used in that body specified in do.

Note that subtypes with no associated data cannot be assigned in this way.

myunion: union {
    t1: u32
    t2: string
    t3: _
}

x: myunion t2: "hello"

if y: x.t2 then {
    print y     # y is a string ("hello") here
}

if y: x.t3 then {    # ERROR: syntax error, cannot assign myunion.t3 to y
    # ...
}

5.14.2. Binary operations

All operations are binary. There are no unary operations. Operation syntax is an alternate function call syntax for methods with a receiver and a single parameter.

Operations are evaluated left to right. Only parentheses can be used to change precedence.

Operator names are simply method names. Identifiers used for method names can include the usual mathematical operator names eg. + - / *
