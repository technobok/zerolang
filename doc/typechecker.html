1. Introduction

This document is the architectural map of the zerolang typechecker. It is intended to be read before opening src/ztypecheck.py (currently ~10,000 lines) -- after reading it, a contributor should know which structures the typechecker maintains, how its code is partitioned, and where a particular feature lives.

The companion document compiler.pdoc covers semantics -- what ownership means, how the multi-granularity lock-conflict rule works, what demand-driven resolution guarantees. This document covers structure: data layout, dispatch, code organisation. The two documents are deliberately split so that semantic rules have a single source of truth (compiler.pdoc / ownership.pdoc) and this one stays a navigation aid that does not rot when rule wording changes.

Audience. A new contributor who has already read spec.pdoc, ownership.pdoc, and the Semantic-Analyser section of compiler.pdoc.

2. Pipeline Position

The typechecker sits between the parser and the C emitter:

  zast.Program  (frozen post-parse)
      ↓   TypeChecker(program).check()
  ztyping.Typing  (mutable, holds all typecheck output)
      ↓
  CEmitter / zsqldump / zasthash

The parser hands the typechecker a zast.Program -- an immutable tree of Node subclasses, each with a monotonic nodeid. The typechecker does not mutate AST nodes; instead it writes its results into a single mutable container, ztyping.Typing, keyed by nodeid. After check() returns, the emitter and SQL dumper read from both the Program (shape) and the Typing (types, ownership, locks, monomorphisations) without further analysis.

Demand-driven, single pass. Resolution starts at main and descends only into names that are actually referenced. Names that are never reached are never typechecked. The same property gives cycle detection for free -- a name that is encountered while it is already on the resolution stack is reported as a circular dependency. See compiler.pdoc:Demand-Driven Resolution for the rationale.

3. Core Data Structures

The typechecker is built around five core record types and one mutable container. Reading this section first makes ztypecheck.py considerably easier to follow, because nearly every function in it produces, mutates, or consults one of these.

3.1. ZType -- the unified type descriptor

ZType (in src/ztypes.py) is the only type representation in the compiler. There is no separate "kind", "shape", or "schema" object: every type -- record, class, variant, union, protocol, facet, function, unit, numeric, native -- is a ZType with a typetype discriminator (the ZTypeType enum) and a set of optional fields.

Key fields:

  • name: the simple name (e.g. Point)
  • typetype: the ZTypeType enum (RECORD, CLASS, VARIANT, UNION, PROTOCOL, FACET, FUNCTION, UNIT, numerics, ...)
  • parent: the enclosing unit type, or None for top-level units
  • children_id_map: per-name monotonic child id, minted lazily by child_id_for(name). The id is the authoritative key for children in Typing.type_child rows and for narrowing entries that need a numeric subject
  • return_type, return_ownership, this_param_name: populated for function types
  • generic-state fields: generic_params, generic_defaults, generic_origin, numeric_generic_params, is_tag_generic_origin
  • const_value: a literal value carried on a ZType used as either an as-block constant or a numeric generic-arg carrier (the 8 in array<i64, 8>)
  • layout / memory metadata: destructor_name, is_heap_allocated, needs_field_cleanup, is_nullable_ptr, is_box (see representation_of_types.pdoc)
  • create_disabled, meta_create: unified-call-dispatch metadata: bare-name calls route through children[create] unless disabled, with meta_create holding the raw allocator
  • cname: the C identifier the emitter uses
  • nodeid: monotonic id for SQL dumps

A ZType carries identity-data only. Per-edge metadata -- parameter ownership annotations, field defaults, lock/private modifiers -- lives on Typing.type_child rows (the flat children table), not on the parent ZType.

3.2. ZVariable -- ownership and naming carrier

ZVariable (src/ztypes.py) wraps a ZType with the state that is per-variable rather than per-type:

  • variableid: monotonic id
  • ztype: the variable's resolved type
  • ownership: OWNED or BORROWED (the 2-state model described in compiler.pdoc:Ownership and Lifetime)
  • named: naming class (named binding, anonymous, ...)
  • is_private_access: carried through .private typedef projections
  • borrow_origin: escape-analysis sentinel naming the function-local source this variable borrows from; None for parameters
  • synth_origin: pass-name string for variables synthesised by a compiler pass (e.g. "anf" for hoisted call temps); None for user-source variables

Keeping ownership off ZType lets the same type be bound in two scopes with different ownership, which is the common case.

3.3. Entry -- a scope binding

Entry (src/ztypes.py) is the unit of scope state. Every piece of information the scope chain needs to track about a name lives on an Entry:

  • entry_id: monotonic id
  • name, ztype, is_definition
  • var: optional ZVariable (absent for type/function definitions and lock-only overlays)
  • lock: optional LockInfo carrying lock type, holder, and the addressable path tuple (root, f1, f2, ...)
  • narrowed_subtype, narrowed_subtype_id, excluded_subtypes, excluded_subtype_ids: type narrowing for union / variant subjects in match arms, with id parallels for the future hot path
  • original_ztype: the un-narrowed type, so the emitter can generate the C-level unwrap
  • is_taken, taken_at: consumed-binding state

An entry is either a definition (it introduces a name) or a shadow / overlay (it adjusts state for a name defined in an outer scope). Definitions live in BLOCK scopes; shadows live in OVERLAY or CALL scopes and disappear automatically when their scope is popped. See compiler.pdoc:Scope Environment for the three scope kinds.

3.4. ExprResult -- the result of checking an expression

Most expression-level helpers return ExprResult (src/ztypes.py), not a bare ZType:

  • ztype: the inferred type
  • borrow_target: the addressable path tuple (root, f1, ...) that this expression borrows from; the enclosing _check_assignment / _check_call_arguments consumes it to install a borrow-scoped lock at the binding site
  • private_access: propagated through .private projections

The borrow_target mechanism keeps borrow-lock installation out of deep expression code: the leaf method that knows the source path stamps it onto the ExprResult, and the eventual binding site installs the lock. The flow is described under ownership.pdoc:Borrow propagation.

3.5. Typing -- the typecheck-output container

Typing (src/ztyping.py) is the single mutable container holding everything the typechecker computes. It is intentionally ECS-shaped: instead of decorating parsed AST nodes with init=False columns, the typechecker writes into a set of Dict[nodeid, value] component tables.

Aggregate state:

  • parsed: back-reference to the immutable zast.Program
  • errors, is_error: error accumulator
  • mono_types, mono_functions: monomorphised generic instantiations, each paired with its origin AST definition
  • func_aliases, cloned_methods, resolved: monomorphisation deduplication caches
  • unit_types_by_id: Unit-AST nodeid → resolved unit ZType
  • symbol_table: the live SymbolTable (typed via SymbolTableProto to keep ztyping decoupled from zenv)

Component tables (each keyed by parsed-node nodeid):

  • node_type, node_const_value: per-Node type and compile-time constant
  • call_kind, call_callable_type_name, expr_call_kind: call classification and control-kind routing
  • do_has_break, case_subject_taken, for_iter_bindings: per-construct flags
  • if_taken_vars, case_taken_vars: post-branch ownership-cleanup lists
  • atom_narrowed_subtype, atom_original_ztype, atom_child_id: per-AtomId narrowing stamps
  • dp_parent_tagged_type, dp_child_id: per- DottedPath parent-type and child-id stamps
  • with_ownership, with_alias_of, assign_alias_of: per-binding alias/ownership stamps
  • projected_args: per-NamedOperation protocol-projection stamps
  • type_child, type_generic_arg: flat children / generic-args tables, replacing the prior per-ZType dict columns

The SQL dumper writes one row per dict, so every component table has a direct SQL representation.

3.6. SymbolTable and Scope

SymbolTable (src/zenv.py) is the scope stack the typechecker uses for all variable state -- type, ownership, lock state, narrowing, taken state. There is no separate flow- typing structure: state changes are recorded as new entries in overlay scopes; scope pop is the only thing that undoes them.

  • Scope: holds entries: List[Entry] (linear scan rather than a dict; typical scope has fewer than ten entries), a kind (BLOCK, CALL, or OVERLAY), a name, an unreachable flag, and a monotonic scope_id
  • SymbolTable: stack of Scope frames plus _history (popped scopes, retained for SQL dump) and scope_log (append-only log of push/pop events with parent ids and sequence numbers)

Three scope kinds (full description in compiler.pdoc:Scope Environment):

  • BLOCK -- language constructs (function, do, for, if, with, match arm). Can contain definitions.
  • CALL -- pushed before processing call arguments, popped after the call. Locks installed here vanish when the call returns.
  • OVERLAY -- shadow records for a variable defined in an outer scope (new lock, narrowing, taken state).

Lock state is keyed on a path tuple (root, f1, ..., fn) stored on LockInfo. Entry.name always equals path[0] so the linear scan stays cheap; the full tuple is consulted only by the prefix-overlap rule. Key APIs (all on SymbolTable):

  • lookup, lookup_var, lookup_var_by_id, lookup_entry -- name → type / variable / entry resolution
  • try_lock -- attempt to install a lock; returns LockInfo or a conflict description. Multi-granularity rule is in _lock_acquire_conflict
  • find_exclusive_lock, is_path_locked -- access- site enforcement (used by lock-escape checks)
  • narrow, exclude -- install a narrowing or exclusion overlay entry on the current scope

The lock conflict logic lives in zenv.py, not in ztypecheck.py. The typechecker only chooses when to install a lock; the rule for whether it conflicts is owned by the scope module.

5. Dispatch Mechanism

Per the project convention (CLAUDE.md), the typechecker uses field-based dispatch on the NodeType enum rather than isinstance. The pattern appears in two forms:

  • Dict-of-bound-methods built at init time -- e.g. _definition_resolvers (NodeType → resolver method). Used where the set of cases is closed and known up front (type-definition resolvers, statement-line kinds). Lookup is O(1) and avoids getattr-by-name.
  • Inline switch on node.nodetype -- used inside _check_expression, _check_path, and a few similar high-fan-out helpers where a flat if / elif chain reads more clearly than a table.

cast() is used after the discriminator check to narrow the node type for the static type checker (ty); see src/zast.py.

6. Resolution Stack and Cycle Detection

TypeChecker._resolving is the list-shaped resolution stack: each entry is a (qualified_name, ZType) tuple. The flow is:

  1. _resolve_unit_name(unit, name) checks self._resolved -- a cache of fully-resolved unit-level names. Hit: return immediately.
  2. Cache miss: push (qualified, placeholder_ztype) onto _resolving, dispatch to the appropriate _resolve_*_type via _definition_resolvers.
  3. If a nested resolution re-enters the same qualified name, the resolver finds it on the stack and reports a cycle error.
  4. On success, the resolver pops the stack frame and writes the final ZType into _resolved.

The type keyword (in receiver position) resolves to the type at the bottom of the resolving stack, so a method body can refer to its own enclosing type without naming it. this works similarly via _resolve_this_keyword.

7. Cross-Cutting Concerns

Each subsection below points at where a concern lives in the typechecker. For why it works that way, follow the links to the semantic documents.

7.1. Ownership enforcement

The 2-state ownership model (see compiler.pdoc:Ownership and Lifetime) is enforced at the assignment and call sites:

  • _check_assignment -- definition site. Records the new variable's ownership in a fresh ZVariable, consumes any ExprResult.borrow_target by installing a borrow lock.
  • _check_reassignment -- enforces that the LHS variable is currently OWNED and that the RHS provides the appropriate ownership; resets narrowing and re-runs lock escape checks.
  • _check_swap -- symmetric ownership preservation.
  • _apply_take_to_arg -- argument-site .take; transfers ownership from the caller's binding into the callee's parameter.

Parameters default to .borrow; .take and .lock are the opt-in transfers. The annotations live on Typing.type_child.param_ownership, not on the parameter ZType.

7.2. Lock checking

The typechecker chooses when to install a lock; zenv.py owns the conflict rule (_lock_acquire_conflict) and the prefix-overlap path test (_paths_overlap). Two scopes of locks, both described in compiler.pdoc:Lock Checking:

  • Call-scoped -- installed by _lock_receiver / _lock_arg inside a CALL scope that wraps argument processing. The leaf is EXCLUSIVE; every intermediate prefix is SHARED.
  • Borrow-scoped -- installed by _install_borrow_locks when a binding takes a borrow into a BLOCK scope (.borrow, .lock, .stringview, with, or a .lock-param transfer). Persists for the lifetime of the binding's scope; released by scope pop or explicit .release.

_check_aggregate_lock_escape runs at storage sites (field writes) and at return sites; it consults is_path_locked on the scope chain to reject a write/return that would let a locked value escape its scope.

Self-overlap during a single call (e.g. f.method bv: f.byteview) is permitted via the call-identity stack (_call_id_stack + LockHolder tagging): try_lock skips the conflict check when the existing holder matches the current call's identity.

7.3. Borrow correctness for parameters

Deep methods that take a borrow into a return value record the source path on _pending_borrow_lock, which _check_expression captures into ExprResult.borrow_target and clears so it cannot leak across statements. The binding site (_check_assignment or _check_call_arguments) then installs the borrow lock against the actual destination scope. The dedicated deep helpers live under _check_dp_* (_check_dp_borrow, _check_dp_lock, _check_dp_release, _check_dp_private, _check_dp_take).

7.4. Monomorphisation

Generic types and functions are monomorphised by deep-cloning the original AST subtree, binding the generic parameters to concrete types, and re-running the standard resolvers on the clone.

  • _monomorphize -- entry for generic type instantiation. Builds a deduplication key via _mono_arg_key (a tuple over arg (name, const_value) pairs so numeric generics also dedupe), reuses an existing instantiation if present, otherwise clones and registers into Typing.mono_types.
  • _monomorphize_function -- per-function variant. Clones the function AST, binds generic params, typechecks the clone with the mono context in scope.
  • _check_mono_constraints -- enforces generic constraints on a candidate argument type.

Cloned nodes carry synth_origin (the pass-name string) so the SQL dump and asthash can attribute them to monomorphisation rather than to the user. The synthesis toolkit (src/zsynth.py) is documented in compiler.pdoc:Synthesis Toolkit; the same toolkit is used by per-call argument hoisting.

is native methods (the compiler-provided collection methods on array, List, Map, str, etc.) are not special-cased -- they are cloned alongside user methods. The only requirement is that their declaration in lib/system/system.z or lib/system/collections.z carries the is native form. Resolution flows through the standard child-lookup path.

7.5. Type narrowing

Narrowing for union and variant subjects in match arms is implemented via overlay entries. When a match arm narrows subject s to subtype ok, an overlay Entry with narrowed_subtype: "ok" is pushed onto the arm's scope. Inside the arm, name lookup for s finds this entry first and returns the narrowed type. The arm scope is popped at end of arm; narrowing resets automatically.

Post-match exclusion is the dual: arms that diverge (return, break, panic) contribute their handled subtypes to an excluded_subtypes entry placed on the parent scope after the match. Subsequent code that re-accesses s as an excluded subtype is rejected.

Assignment-based narrowing (e.g. x: Result.ok 42) places a narrowing entry in the current scope at definition time; reassignment resets it.

Both narrowing and exclusion carry id parallels (narrowed_subtype_id, excluded_subtype_ids) minted via ZType.child_id_for. String fields remain authoritative; the ids are the hot-path key prepared for future migration to the SQL-backed representation.

7.6. Auto-call coercion

A no-user-arg method dotted path read in value position (v: s.stringview, p1.distance inside a string interpolation) is auto-called to its return type. The coercion lives entirely in _check_dotted_path and is gated by a coerce_method_to_return parameter. _check_call flips the flag off when resolving its own callable, so explicit calls (obj.method named: arg) see the function type and dispatch normally instead of constructing the return type. Path resolution itself (_resolve_dotted_path) is context-free: it always returns the FUNCTION type for a method path. Behaviour is identical for is native and user-defined methods.

7.7. Unified call dispatch

A type used in callable position is rewritten through the unified-dispatch route. The bare-name construction call Point x: 1 y: 2 resolves the callee to the Point ZType, then looks up children[create] on that type to get the actual constructor. The mechanics live in _check_return_call and friends; meta.create on the parent ZType carries the compiler-generated raw allocator when one is needed. create: null in the type's as block disables bare-name construction (the case for union and variant where subtype selection is required).

8. Error Reporting

Type errors are zast.Error value objects, appended to self.errors via the _error() helper. Errors are never raised as exceptions; the typechecker tries to continue collecting errors past the first failure where the AST permits. Error codes (the ERR enum in src/zast.py) are grouped:

  • E0001-E0099 -- parser / syntax (raised earlier in the pipeline)
  • E0100-E0199 -- type resolution
  • E0200-E0299 -- ownership
  • E0300-E0399 -- call / argument
  • E0400-E0499 -- generic / monomorphisation

The diagnostic location is taken from the offending AST node's loc / Token. After check() returns, the caller inspects typing.errors (or the return value of check()) to decide whether to proceed to emission.

9. What Lives Elsewhere

Module boundaries the typechecker relies on:

  • src/ztypes.py -- ZType, ZVariable, Entry, ExprResult, LockInfo, LockHolder, plus the ZTypeType / ZOwnership / ZParamOwnership / ZLockState / ScopeKind / ZSubType / ControlKind / BuiltinFunc / ZNaming / LockHolderKind enums
  • src/ztyping.py -- the Typing container, TypeChild / TypeGenericArg flat-table rows, TypedProgramView compat shim
  • src/zenv.py -- SymbolTable, Scope, ScopeLogRow, the lock-conflict logic (_lock_acquire_conflict, _paths_overlap), narrowing / exclusion installers
  • src/ztypeutil.py -- collection-type predicates (is_array_type, is_list_type, list_element_type, ...) shared between typechecker and emitter
  • src/zsynth.py -- node synthesis toolkit (FreshNamer, make_atom_id, make_assignment, register_synth_var) used by per-call hoisting and auto-generated equality
  • src/zsymtab_proto.py -- the duck-typed SymbolTableProto that breaks the zast ↔ zenv import cycle
  • src/zasthash.py -- structural hashing used by the monomorphisation dedup key

10. Reading Order

When opening the typechecker for the first time, read in this order:

  1. src/ztypes.py -- the dataclass definitions in this file are the vocabulary for everything else.
  2. src/ztyping.py -- the Typing container makes the component-table architecture concrete.
  3. src/zenv.py -- the scope stack and the lock-conflict rule. The semantics of "what makes two locks conflict" lives here, not in the typechecker.
  4. src/ztypecheck.py -- enter via the Code Map above. A good first dive is check() → _resolve_unit_name → _resolve_function_type → _check_function_body → _check_statement → _check_expression → _check_call. That path touches almost every cross- cutting concern in turn.
  5. src/zsynth.py -- read once you encounter a hoisted temp in a SQL dump or test snapshot.
