1. Overview

This document defines the implementation roadmap for the Zero language compiler. The language is ambitious; development is staged to allow a working compiler as early as possible. Each version builds on the previous.

2. v1 Minimal Core Subset

v1 targets a minimal subset sufficient to compile simple programs. The Python compiler emits C code which is compiled with gcc or clang.

2.1. v1 Includes

  • Numeric types: u8 u16 u32 u64 i8 i16 i32 i64 f32 f64
  • record (value type product types, zero-initialized, no constructors)
  • Functions with in/out/is (no generators, no yield)
  • Control flow: if/when/then/else, for/loop/while, with/do
  • Assignments (: declaration, = reassignment), swap
  • Simple strings (immutable, string literals, interpolation)
  • data (constant arrays)
  • Binary operations (left-to-right, no precedence)
  • Named parameters, unnamed first parameter
  • Units/modules (file = module, basic imports via forward reference resolution)
  • break, continue, return
  • Boolean logic via if short-circuit when chains
  • Basic match on simple types (not full union pattern matching)

2.2. v1 Defers

  • Classes (reftype product types) -- deferred to v2
  • Unions, variants -- deferred to v2
  • Protocols, facets -- deferred to v2
  • Generics -- deferred to v2
  • Ownership checking (owned/borrowed/free enforcement) -- deferred to v2
  • Lock checking -- deferred to v3
  • Generators/yield -- deferred to v3
  • Compile-time evaluation (beyond simple constants) -- deferred to v3
  • Type coercion (beyond explicit conversions) -- deferred to v2
  • Typedefs -- deferred to v2

2.3. v1 Note on Ownership

v1 will NOT enforce ownership rules at compile time. All heap allocations will be freed at scope exit (simple stack-based lifetime). This is incorrect for the full language but sufficient to get programs running. Ownership enforcement comes in v2.

2.4. v1 Note on Strings

v1 uses a simple immutable string type with a home-grown implementation (ZStr). Strings are heap-allocated as a single contiguous block using a C flexible array member:

typedef struct {
    int32_t len;    // byte length (excl NUL)
    char data[];    // NUL-terminated UTF-8
} ZStr;

This layout was chosen over external libraries (bstrlib, SDS) because:

  • Immutability is fundamental to zerolang strings. Both bstrlib and SDS are mutable-by-design; adopting either means paying API complexity for features that are actively unwanted.
  • Single allocation (header + data together) is cache-friendly and matches the spec's zstring design (see TODO).
  • v1 needs very few operations: create, concatenate (returns new string), compare, length, print, free. ~100 lines of implementation.
  • The struct layout evolves directly into v2 string types: stringdef is ZStr as-is (immutable); stringref adds a capacity field for mutation; stringval is stack-allocated with a compile-time-known max size.

String literals and interpolation are supported. Advanced string designs (stringref, stringval, stringdef) are deferred to v2.

3. v1 Example Programs

These programs define the v1 compilation targets. Each must parse, type-check and emit correct C.

  1. examples/hello.z -- hello world, string literals, print
  2. examples/fibonacci.z -- loops, integer arithmetic, functions
  3. examples/factorial.z -- recursion, return values
  4. examples/records.z -- record definition, field access, methods
  5. examples/strings.z -- string interpolation, string methods
  6. examples/control.z -- if/when/then/else, for/while/loop, with/do
  7. examples/multimod.z -- multi-module program (imports)
  8. examples/data.z -- constant data arrays
  9. examples/case.z -- basic match expressions on enums
  10. examples/swap.z -- swap keyword, reassignment

4. Documentation Cleanup Tasks

These tasks should be resolved before or during implementation.

4.1. Contradictions to Resolve

  1. Assignment as expression: RESOLVED. Assignment is a statement, not an expression. FAQ updated.
  2. Macros: RESOLVED. No macros. FAQ updated to state final position only.
  3. Typedefs: RESOLVED. Yes, typedefs exist. FAQ updated. Spec has typedef section.
  4. Embedding: RESOLVED. Records can be embedded. FAQ updated.
  5. Records with constructors: RESOLVED. No constructors, zero-initialized, use static factory methods. FAQ updated.
  6. Protocols with functions: Protocols have specs (function signatures) only, no implementations. FAQ updated.
  7. Case sensitivity: RESOLVED. Case sensitive, underscores are part of identifier. FAQ updated, spec already states this.
  8. case vs switch naming: RESOLVED. Grammar and spec now use match as the top-level keyword and case for branches. Grammar production name is match_expression.
  9. Ownership versions: Multiple versions (v3, v4) exist in TODO. These should be consolidated into a single canonical version in the spec when ownership is implemented (v2).

4.2. Gaps to Fill

  1. Error handling: Not needed for v1 (no unions). v2 should define a result union type.
  2. String type: v1 uses simple immutable strings. Advanced designs deferred.
  3. Destructors: Deferred to v2 with classes.
  4. Generics implementation: Deferred to v2. Strategy (monomorphization vs erasure) to be decided then.
  5. Lock semantics: Deferred to v3.
  6. Standard library: v1 needs only: print (via system.io_print).

4.3. Structural Cleanup

  1. Triage TODO.pdoc: categorise each section as DECIDED (move to spec), OPEN (keep with marker), or OBSOLETE (delete).
  2. Add status markers to spec sections: STABLE, DRAFT, PROPOSED.
  3. Reconcile grammar.pdoc with spec.pdoc for v1 constructs.
  4. Archive or remove zerolang-doc/ if it is superseded by zerolang/doc/.
