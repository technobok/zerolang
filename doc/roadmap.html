1. Overview

This document defines the implementation roadmap for the Zero language compiler. The language is ambitious; development is staged to allow a working compiler as early as possible. Each version builds on the previous.

2. v1 Minimal Core Subset

v1 targets a minimal subset sufficient to compile simple programs. The Python compiler emits C code which is compiled with gcc or clang.

2.1. v1 Includes

  • Numeric types: u8 u16 u32 u64 i8 i16 i32 i64 f32 f64
  • record (value type product types, zero-initialized, no constructors)
  • Functions with in/out/is (no generators, no yield)
  • Control flow: if/when/then/else, for/loop/while, with/do
  • Assignments (: declaration, = reassignment), swap
  • Simple strings (immutable, string literals, interpolation)
  • data (constant arrays)
  • Binary operations (left-to-right, no precedence)
  • Named parameters, unnamed first parameter
  • Units/modules (file = module, basic imports via forward reference resolution)
  • break, continue, return
  • Boolean logic via if short-circuit when chains
  • Basic match on simple types (not full union pattern matching)

2.2. Implemented (originally deferred)

The following features were originally deferred but are now fully implemented:

  • Classes (reftype product types with heap allocation, destructors, methods)
  • Unions (tagged heap-allocated sum types with pattern matching)
  • Variants (tagged stack-allocated sum types)
  • Protocols (vtable-based dynamic dispatch with boxing)
  • Facets (value-type interfaces with inline data)
  • Generics (monomorphization with AST-hash deduplication)
  • Typedefs (type aliases with transparent resolution)
  • Ownership checking (2-state OWNED/BORROWED model with take/borrow/lock)

2.3. Still Deferred

  • Lock checking (enforcement) -- infrastructure exists, enforcement deferred
  • Generators/yield -- deferred
  • Compile-time evaluation (beyond simple constants) -- deferred
  • Type coercion (beyond explicit conversions) -- deferred

2.4. v1 Note on Ownership

v1 will NOT enforce ownership rules at compile time. All heap allocations will be freed at scope exit (simple stack-based lifetime). This is incorrect for the full language but sufficient to get programs running. Ownership enforcement comes in v2.

2.5. v1 Note on Strings

v1 uses a simple immutable string type with a home-grown implementation (ZStr). Strings are heap-allocated as a single contiguous block using a C flexible array member:

typedef struct {
    uint64_t size;  /* bits 62-0: byte count; bit 63: static flag */
    char data[];    /* NUL-terminated, starts at 8-byte boundary */
} ZStr;

#define ZSTR_STATIC_FLAG  0x8000000000000000ull
#define ZSTR_SIZE(z)      ((z)->size & ~ZSTR_STATIC_FLAG)
#define ZSTR_IS_STATIC(z) ((z)->size & ZSTR_STATIC_FLAG)

The size field uses its top bit as a static flag. When set, the string is a compile-time literal embedded in the binary and must not be freed. The remaining 63 bits hold the byte count (excluding the NUL terminator). The ZSTR_SIZE and ZSTR_IS_STATIC macros extract these components.

This layout was chosen over external libraries (bstrlib, SDS) because:

  • Immutability is fundamental to zerolang strings. Both bstrlib and SDS are mutable-by-design; adopting either means paying API complexity for features that are actively unwanted.
  • Single allocation (header + data together) is cache-friendly and matches the spec's zstring design (see TODO).
  • v1 needs very few operations: create, concatenate (returns new string), compare, length, print, free. ~100 lines of implementation.
  • The struct layout evolves directly into v2 string types: stringdef is ZStr as-is (immutable); stringref adds a capacity field for mutation; stringval is stack-allocated with a compile-time-known max size.

String literals and interpolation are supported. Advanced string designs (stringref, stringval, stringdef) are deferred to v2.

  4. Embedding: RESOLVED. Records can be embedded. FAQ updated.

  7. Case sensitivity: RESOLVED. Case sensitive, underscores are part of identifier. FAQ updated, spec already states this.

5.7. Phase 7: Ownership Checking (Compile-Time Enforcement) [COMPLETE]

Add a new pass (or extend the existing type checker) to enforce ownership rules. This is the core safety analysis.

  1. Track ownership state per variable through function bodies: each local variable starts as owned (for declarations) or borrowed (for parameters without take).
  2. Enforce: take parameters consume ownership -- caller's name is invalid after the call.
  3. Enforce: borrowed variables cannot be stored, returned (without lock), or destroyed.
  4. Enforce: reftype fields can only be changed with swap, not direct assignment.
  5. Enforce: valtype fields can be freely reassigned with =.
  6. Enforce: this is always borrowed within methods.
  7. Enforce: unnamed instances (function return values) are destroyed at end of statement if not assigned or passed to an owned parameter.
  8. Enforce: swap arguments must be owned (if single variable) or parent must be owned (if dotted).
  9. Enforce: a function returning borrow must have at least one lock parameter; the returned value must originate from a locked parameter.
  10. Enforce: cannot return a local variable as borrowed.
  11. Enforce: no reftype aliasing -- a reftype cannot appear twice as an argument in a single call.
  12. Track ownership flow through if/match/for branches: if a variable is taken in one branch, it must be taken in all branches (or not used after the branch). [DEFERRED -- requires reftype usage in branches; valtypes are always owned by copy so this rule is trivially satisfied for v1 programs.]
  13. Implement take as a compiler method: invalidates the source name, returns an unnamed owned instance.
  14. Implement borrow as a compiler method: returns a borrowed reference, exclusively locks the receiver.
  15. Error messages: clear diagnostics for ownership violations with source location and explanation.
  16. Tests: ownership checking tests for each rule above using small programs.
  17. Tests: verify all v1 example programs still pass (valtypes only, should trivially satisfy ownership rules).
  18. Tests from ownership.pdoc examples -- each must produce the expected error or pass:
    • OK: y: x.borrow -- borrowed over lock on x.
    • OK: z: y.borrow -- chained borrow maintains lock.
    • OK: a: z.childref.borrow -- borrow over lock on parent. [DEFERRED -- requires reftype fields.]
    • ERROR: function returning borrow without lock parameter.
    • ERROR: function returning local var as borrow.
    • OK: function with lock parameter returning borrow.
    • ERROR: function with lock param but returning a different local var.
    • ERROR: method returning borrow of field without this.lock. [DEFERRED -- requires class methods with reftype fields.]
    • OK: method with this.lock returning borrow of field. [DEFERRED -- requires class methods with reftype fields.]
    • OK: chained lock -- function calls sub-function that locks, caller also locks, borrow escapes correctly. [DEFERRED -- requires reftype function calls.]

5.8. Phase 8: Lock Checking (Compile-Time Enforcement) [COMPLETE]

Add lock checking as part of the ownership analysis. Locking prevents aliasing and data races at compile time.

Implementation: locks are stored as a list of LockEntry (lock_type + holder) on each ZVariable. Each variable also tracks held_locks (names of variables it holds locks on) for cleanup. During function calls, a local call_locks list tracks locks taken for that specific call so that only those locks are released/transferred after the call completes; pre-existing locks are untouched.

  1. Implement lock state tracking per variable: unlocked, exclusive, shared. Replaced single lock_state field with locks: List[LockEntry] on ZVariable and held_locks: List[str] for bidirectional tracking.
  2. Implement the locking algorithm for function calls: process arguments left to right; exclusive lock on each argument; shared lock on parent chain for dotted arguments; error on double-lock.
  3. Implement: top-level data items are exempt from locking (immutable).
  4. Implement: recursive locking for sub-calls in arguments -- locks from sub-call are cleared or assigned to sub-call result.
  5. Implement: after call, lock-typed parameter locks transfer to return value; all other argument locks are freed.
  6. Implement: lock on receiver (exclusive lock on each component of dotted receiver chain) after arguments are processed.
  7. Implement: locks on match/for targets to prevent mutation in body.
  8. Implement: match with .take takes ownership of target with narrowed type in branches. [DEFERRED -- requires union types for meaningful type narrowing.]
  9. Implement: field access locking -- shared lock on parent, exclusive on field. [DEFERRED -- requires reftype fields (classes) for meaningful testing.]
  10. Implement: scope exit frees all locks held by the exiting variable.
  11. Error messages: clear diagnostics for lock conflicts.
  12. Tests: locking tests for each rule, including:
    • ERROR: same reftype passed as two arguments to a single call (exclusive lock conflict / aliasing). [DEFERRED -- requires reftypes; aliasing check from Phase 7 covers this.]
    • ERROR: exclusive lock on item that already has shared lock from parent chain. [DEFERRED -- requires reftype dotted access.]
    • OK: two dotted arguments sharing a parent (shared locks on parent are compatible). [DEFERRED -- requires reftype dotted access.]
    • OK: data items passed freely without locking.
    • ERROR: mutating a match/for target inside the body. [DEFERRED -- requires reftypes for meaningful mutation check.]
    • OK: match with .take -- target owned with narrowed type in branches. [DEFERRED -- requires union types.]

Implementation notes: added LockEntry dataclass to ztypechecker.py. Added try_lock, release_lock, release_held_locks methods to SymbolTable in zenv.py. Lock checking integrated into _check_call (argument locking with call-scoped tracking), _check_dotted_path (.borrow creates exclusive lock), _check_assignment (borrow assignments create borrowed variables), _check_for and _check_case (target locking). 26 new tests (2 LockEntry, 10 SymbolTable locking, 3 borrow locking, 1 data exempt, 1 scope exit, 11 example programs). 5.29. Phase 29: Typedefs (Wrapper Types) [COMPLETE]

The grammar has been changed: an item_definition can no longer take a bare operation, it must always take a series of child definitions in braces.

Add typedef support. Typedefs wrap an underlying type with zero overhead. A typedef is created by using a standard object definition (class, record, union, variant). In the 'is' section is a single named item that refers to the 'typedef' of the underlying type. eg. "mybase: basetype.typedef". 'mybase' can be any name provided by the user.

Kind matching is strict for like types: class wraps class, record wraps record, union wraps union, variant wraps variant. Protocols and facets cannot have their own typedef syntax (no 'items' in their grammar), so class is also used to create typedefs over protocols (both are reftypes) and record is used to create typedefs over facets (both are valtypes). This avoids grammar changes for an edge case.

'typedef' is a synthetic member that the compiler adds to all types to signify that this is a typedef. It can only be used in the definition of a typdef item.

Similar to other features (such as 'tag'), 'typedef' is also a top level predefined type that takes a generic 'any' parameter called 'from'. So the definition could also have been 'mybase: typedef basetype'.

On the resulting typedef created by the compiler, 2 compiler generated constructor methods are created.

1. 'take' and 'create' are aliases for the same constructor that will take ownership of the item of the basetype. signature is take: function {from: T.take} out typedeftype

2. 'borrow' is a constructor that will return a borrowed typedeftype while taking a lock over the underlying basetype. Signature: borrow: function {from: T.lock} out typedeftype.borrow

There is older documentation that refers to a different way of creating typedefs but it is obsolete, remove that. Ensure that all documentation is consistent and complete as the feature is described here.

Other details:

  • A typedef is a typesystem modification only, it does not affect the original object definition and is zero overhead at runtime.
  • Compiler returns an error if the typedef object type is not the same as the object type specified in 'is' (eg typedef type is class but 'is' basetype is record). Strict kind matching: class-class, record-record, union-union, variant-variant.
  • Class typedef also accepts protocol base types (both are reftypes). The class kind check allows class or protocol as the base type.
  • Record typedef also accepts facet base types (both are valtypes). The record kind check allows record or facet as the base type.
  • The typedef is a new type that is backwards compatible but not forwards compatible ie. the new type can be passed to existing functions that expect the old type but functions that require the new typedef'd type cannot be passed the underlying type.
  • There can only be 1 .typedef included in the 'is' section (in fact, only 1 definition in 'is' of any type for a typedef) else error.
  • New methods can be added in the typedefs 'as' section. The new methods may shadow/replace existing method on the underlying object.
  • If access is required to the old methods that are being shadowed, this can be done by accessing the methods on the named base type from the 'is' section ('mybase' in the example above). Or they could be accessed on the type of the mybase type itself, with the 'this' argument passed in as the first argument.
  • All new methods plus all unshadowed methods of the base object are exposed to the user of the new typedef'd item.
  • The name given to the typedef in the 'is' section allows access to all methods on the underlying type as usual (and none of the new methods).
  • Typedef generic parameters: all generic params of the underlying type must appear in the typedef field definition. They must also be redeclared in the 'as' section of the typedef type. There can be no additional generic types beyond those used in the underlying basetype, as there would be nowhere to specify them, since the compiler controls generation of the constructors and handles conversion from the underlying type.
  • Null method hiding: setting a method to null in the 'as' section hides it from the typedef, producing an error if called.
  • Typedef chaining: a typedef of a typedef is supported with compatibility chain walking.
  • Expand test suite: resolution, type identity, method shadowing, null hiding, constructors, backward/forward compatibility, typedef chaining (10 type checker tests).
  • Emit C for typedef: no separate struct emitted. Typedef emits as the underlying C type (zero overhead). Methods emit with the typedef-mangled name. 5.33. Phase 33: arrays (valtype, static length) [COMPLETE]

Implement arrays. Arrays are a valtype that are generic over the element types AND their length. This means that arrays of different lengths are different types, even if their element types are the same.

Arrays are part of the zerolang standard library but are not a builtin type. We need a array unit in the lib. Constructor is 'array.create of: type to: x' where 'type' is the type of each element which must be a valtype with an empty ('0') constructor. And 'x' is the size of the array (element indexes are 0 to x-1).

Arrays are:

  • fixed length
  • valtype, the whole array can be replaced by another array of the same type and length
  • array is in the collections unit as part of the standard library
  • array is also aliased in core... array: collections.array
  • 'data' can be copied into a new array of the same type and length (data has a .array method)
  • the element indexes of the array are integers
  • 'array.0' gets the zeroth element. This is compile time checked (since we know the length of the array)
  • 'array.0: 1' sets the zeroth element, again, compile time checked for bounds

5.43.2. 43.2 Iterator protocol [COMPLETE]

The iterator "protocol" is simply methods and callable object.

A callable object is an object that has a 'call' method. The compiler determines if the syntax represents a call (with 0, 1, or many arguments) and if the target/receiver of that call is an object (rather than a function/method) then the compiler checks if the object has a 'call' method and implicitly uses that method as the method being called.

An iterable object is one that has a method that returns an iterator callable object. The method returning this is called 'iterate' by convention, but it can have any name. The iterate method will lock the object being iterated and will return a borrowed iterator object.

The iterator object itself will usually be constructed (from the parent object) so that it locks the parent object that is passed to it. The parent object may also pass its private (internal) API so that the iterator has full access to the parent internals (public / private to be implemented in a later phase).

The iterator object is callable and returns an optional value. Once the iterator is exhausted, it returns the 'none' option. The iterator may also have arguments - for implementing a coroutine style iterator. But no arguments is the common case.

The 'for' loop can only be bound to an iterator like object/function (that returns an option). The iterator must be extracted from the iterable object explicitly (at this stage, may change in the future if a less verbose syntax is required). Retrieving the iterator would usually be done with a 'with' statement. So full syntax would look similar to:

"with i: myiterable.iterate for x: i loop { ... }"

'i' becomes the iterator and within the loop body, 'x' is bound to the current 'some' value of the option. If the iterator returns 'none', 'x' is not bound and the loop terminates.

  1. Callable objects: type checker (_check_call) detects when a variable (not a type name) is called and the variable's type has a call child method. Redirects dispatch through CallKind.CALLABLE, skips the this parameter for argument matching. Emitter generates z_type_call(receiver, args).
  2. For-loop iterator bindings: _check_for detects when a named binding's operation returns option (directly, via a callable object's call method, or via a function's return type). Marks the binding in fornode.iterator_bindings. Unwraps some type for the bound variable. Emitter generates a while(1) loop that calls the operation each iteration, checks TAG_NONE to break, and dereferences ->data for the unwrapped value.
  3. Generic return types in parens: out (option t: i64) now works for functions returning monomorphized generic types. The parser accepts parenthesized generic type instantiation in the out clause.
  4. Tests: callable objects (with and without extra args), iterator bindings with callable classes, empty iterators (9 new tests total across all 43.x sub-phases).

5.45. Phase 45: Box type [COMPLETE]

Objects may be defined to be generic over 'any' which could be either a valtype or a reftype when converted to a concrete type. To handle both situations in a similar way, introduce a new generic type called 'box'.

  1. 'box' has a generic parameter 't' of type 'any.generic' and a 'from' argument of type 't' which takes ownership
  2. when used, if the 'from' argument of type 't' is a reftype, the 'box' returns that argument unchanged. This is a typesystem 'change' only. There is no runtime cost. In subsequent code, the 'new' variable can be used exactly as the original type could be used. The old variable name in invalid as is usually done when an argument is taken
  3. when used, if the 'from' argument of type 't' is a valtype, an allocation is done to store that valtype and the new 'boxed' variable becomes a reftype. Destruction of the boxed type needs to free the underlying memory that was allocated
  4. 'box' can also be used in normal concrete code as a simple way to convert a valtype into a reftype. Note that this could also be done be creating a 'class' with the reftype as a single field, which would give more control (can create your own methods and could control access to the underlying field)

Implementation:

  1. System definition: box: class {} as { t: any.generic } in system.z, exported via core.z
  2. Type checker: _check_box_construction handles box from: val. For reftype T: call resolves to T directly (BOX_PASSTHROUGH). For valtype T: monomorphized box type created as reftype with inner type's children copied for transparent access (BOX_CREATE)
  3. Emitter: _ctype returns pointer to inner valtype ctype. _emit_mono_box emits destructor (free). _emit_box_create emits malloc+copy. _emit_box_passthrough emits ownership transfer. Auto-deref in _emit_binop_value for boxed operands
  4. Tests: type checker tests for valtype/reftype box, emitter tests for arithmetic, comparison, passthrough. box.z example

5.46. Phase 46: public / private accessibility [COMPLETE]

'public' and 'private' are predefined identifiers for controlling access to the members (fields and methods) of objects.

  1. public and private use a 'unit' construct to define their contents.
  2. private is automatically defined by the compiler to always include all members of the current object. It cannot be redefined by the user
  3. public is automatically defined by the compiler to always include all members of the current object but it can be redefined by the user
  4. the user can redefine public by using a 'unit' keyword inline within the 'as' section of the object, eg. public: unit { method1: mymethod1 field1: myfield1 }
  5. resolution of definitions within the public definition is done in the usual way and would be expected to resolve to other fields/methods in the 'is' and 'as' section of the current object
  6. it would be expected that the shortened label declaration syntax would often be used, but it would not be required if the user wanted to expose a method/field publicly with a different name to the interal declaration. Example of public with shortened syntax: public: unit { :method1 :field1 }
  7. inside the object, the 'this' definition always accesses the 'private' namespace and can access all methods/fields
  8. 'this.private' can also be passed to an external function or object definition. This allows creation of 'friend' objects similar to C++ that can share access to the internals of an object. Note that both the receiver must declare that it is receiving mytype.private as a parameter definition and only the mytype object itself can pass this.private. There is no way to gain access to the internals of an object without its cooperation
  9. from outside an object, only members listed in the 'public' definition can be accessed. Remember that by default, this is all fields/members. But if declared explicitly can be a subset of available methods and may have different names than the internal members
  10. 'public' could also be a useful tool for managing backwards compatibility. Members can be freely renamed internally while keeping the can external 'API'. Both the new name and the old name could be exposed in the public interface'
  11. 'public' also applies to 'unit's for controlling access to definitions within it.
  12. otherwise, 'public' and 'private' can be applied to all instantiable object types: record, class, union, variant, protocol, facet

Implementation:

  1. Parser: _getobjectbody extended to handle inline unit definitions in as blocks (for public: unit { ... }). Extern references from the public unit are not promoted (resolved by type checker instead)
  2. Type system: public_members (optional set of strings) field on ZType. None = all-public (default). Set during type resolution when public: unit { ... } is declared in the as block
  3. Type checker: _extract_public_members helper extracts member names from as_items. Called from _resolve_class_type, _resolve_record_type, _resolve_union_type, _resolve_variant_type. _is_non_public_access checks external member access against public_members, allowing this and same-type access
  4. Emitter: no changes needed (zero-overhead compile-time enforcement)
  5. Tests: type checker tests for default-public, restriction, error on non-public, this-access, class restriction, private redefinition, public renaming. Emitter tests for compilation. visibility.z example
  6. Private redefinition prevention: error on private: unit { ... } in as blocks
  7. Public renaming: public: unit { api_name: internal_name } redirects external access through the renamed public name to the internal member
  8. Friend access via this.private: implemented. A field declared as mytype.private grants the holder full access to all members of that type. Only code with internal access (methods of the type itself, or variables already holding a .private reference) can pass this.private. The .private reference is always a borrow — the parent retains ownership. If the parent method returns a friend object (eg. an iterator), it should take a lock on this (eg. {self: this.lock}) so the parent cannot be mutated or destroyed while the friend holds its reference. Zero runtime overhead — same C type, purely compile-time access control. Tracked via ZVariable.is_private_access and ZType.private_fields.
