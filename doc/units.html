1. Units

  • modules are files and packages are directories
    • one file, one module. Cannot spread a module over multiple files, no include
      • but can make a module that exports submodules, so you can control structure and expose it with a single exported var
      • can have a file (with extension) and a directory with the same name but:
        • the file is the one that is loaded from 'outside', ie. the public part
        • the dir is only accessable from the file. It contains private modules so the module can be split up into files but the top level file controls what is exposed to the outside world
        • submodules are searched in the dir first before looking in the global file tree
  • packages (dirs) can only contain modules (files)
  • modules can only refer to submodules further down the directory tree (can't refer to parents)
  • modules (and packages) 'own' the namespace where they are located
  • modules can shadow vars in other files and dirs (but only within themselves)
    • to find a var definition, look in current module, if not found, go look in the directory tree (first relative, then from the top level module path)
  • module vars are public if they are capitalised.
  • modules and packages are never capitalised (to cope with case insensitive filesystems). So they are always public (but see private submodules above)
    • there are other restrictions on module and package names because they must be valid identifiers (up to the extension) eg. can't start with a number
  • modules have no initialisation
    • so they don't need to be imported
    • can't have any special data structures (just static, readonly simple vars and arrays (and dicts, but special simplified dict that are arrays inside and are not O(1) on access ?....)
    • any special state must be provided by an api. Eg. an init function or similar
