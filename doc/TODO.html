4. Ownership rules

Rules for 'objects':

  • all objects have a single owner or parent which cannot be itself
  • ownership is to direct children only, it does not extend to grandchildren and there is no direct access to grandchildren
  • valtypes are copy by value. Each can only have a single owner, but easily copied so can make a new copy with a new owner.
  • reftypes are copy by reference. Can only have a single owner, but can be loaned via function calls or given away to a function call via a 'take' param
  • objects can only be disposed by their owner and no self-ownership so objects cannot dispose of themselves
  • all reftypes in object fields must be initialised and cannot be removed, only swapped
  • and, methods on reftype fields cannot be passed anything that is not owned (no borrowed references, eg self to prevent killing ourselves) Why not other borrowed references besides self?
  • ...so, reftypes in calls can only be borrowed (from parent call) or owned directly (local var). this.reftype???
  • returned reftypes always give ownership from function to caller. 'take' arguments give ownership from caller to function (and are no longer valid at the caller from that point forward)
  • initialisation? The one true constructor must be called supplying all values, although default values can be used for valtypes.
  • destructor? There is a default destructor to recursively free all reftypes. Can be overridden to free/cleanup non-memory resources. Can return a single reftype from the class. How would that look?

5. IDEA: Pointers, ownership, give/take on valtypes

  • 'this' is always passed as a pointer. object cannot be destroyed within a method, so should be safe even for reftypes
  • more efficient for large reftypes - records. 'this' implies this particular instance so signals that pass by reference is not used
  • can never assign to 'this'
  • can valtypes be passed by 'take'? Yes. But note that this would destroy the var in the caller. Would need to return this value if you want to pass it back. Compiler would need to be smart enough to pass a pointer back on return to avoid copy both ways. Would be more efficient for large records.

For:

The labelled target of a for (if present) is examined:

- if it is a function or an object method, call it every iteration - NOTE: the function/method is NOT immediately evaluated - if it is an object instance with an iterate method, call that to get the iterator, then call that iterator on every iteration - if it is an object instance with a call method (with or without arguments), call that every iteration (with the supplied arguments) - ownership: the for exposes the correct ownership since the compiler will check the types in the 'call' method of the iterator. If a taken type is returned, it will be owned in the body of the for. If borrowed, it will be borrowed in the body. - scoping in the body ends at the end of the loop or immediately after a break or continue. So a lock on the iterator held by the returned element will be released immediately before the start of the next loop so that the iterator is able to be called to return the next element (which itself will then take a lock for the loop body)

All vars have a lock type. Unused local vars and parameters start with 'none'.

  • as yet unreferenced vars

36. Ownership v3

36.1. Attributes: type, ownership, mutability, lock

36.1.1. Type

Type is an attribute of the object (not of the name).

- definition type (deftype) - top level object definition (record, class, enum, functions/methods, numeric constants, etc.) Compiler metadata only and are immutable. - reference type (reftype) - has a runtime 'pointer'/'reference' to a type of possibly indeterminate size, has identity etc. Can only be created in run-time code. - value type (valtype) - an inline block of data with a fixed size for this type. Has value semantics. Can be sources from some top level definitions (compile time numeric constants, function 'references'). Can only be created in run-time code.

36.1.2. Ownership

Ownership is an attribute related to each expression in a function body.

- free: no name (anonymous); result of a standard call (except from a 'borrowed' return as from an iterator); result of 'take' call; all valtypes are essentially 'free' to copy. - owned: local owned var; taken reftype parameter - borrowed: local borrowed var; non- taken reftype parameter; 'borrowed' return type of a function; result from an iterator; 'this' is always borrowed - linked: child reftype field of an object

linked....for returning borrowed, must be linked or borrowed (cannot be owned because it wont exist afterwards)

36.1.3. Locality

Attribute related to the location of the object:

  • local - local variable in the current function scope. Variable will be destroyed when that scope ends. Includes:
    • free (anonymous) objects
    • Local variables (owned variables; and shared variables that are bound to owned local vars)
    • 'taken' function parameters
    • (others?)
  • non-local - variable sourced from outside the current function scope that will live on after the function returns. Includes:
    • non- 'taken' parameters (shared parameters)
    • linked variables; and shared vars bound to linked variables
    • (others?)

Locality is passed in from the source. 'taken' variables become local but otherwise a shared var will have the same locality as its source.

36.1.4. Mutability

Mutability is an attribute related to the name of a 'variable'. The variable value may still be mutable.

  • immutable - top level definitions; method names of object; 'this' inside an object
  • mutable - name can be copied to (with a compatible RHS, previous reftype value will be destroyed, valtypes will be copied over); or swapped

36.1.5. Lock

Lock type is an attribute related to each expression part in a function body. Expressions 'hold' locks over other variable names/values. Only for reftypes; valtypes and deftype always have a lock type of 'none'. When an expression is assigned to a name, the expression locks are then carried with that name and released when the name falls out of scope.

It is an error to attempt to take a lock that breaks the below rules:

  • exclusive - holder can do anything (call method, 'take' etc). No other expression can do anything with this expression while the lock is held. (ie. method calls, use as argument, reference children). No other lock of any type can be taken. Expression type must have a locktype of 'none' before an exclusive lock can be taken.
  • shared - can reference children, cannot call methods, cannot take an exclusive lock. Expression type must have a locktype of 'none' or other 'shared' locks only before taking a shared lock.
  • none - no restrictions, can take a shared or exclusive lock. Top level definitions always have lock type of 'none' and take no locks on their RHS. Top level definitions are immutable and shareable - they are always deftypes or valtypes.

36.2.1. createval

36.2.2. createref

Create a new reftype variable. LHS must be a new label. New variable will hold all the locks that the RHS has taken.

RHS:

  • must be a reftype
  • RHS parents and 'leaf' must have lock type of 'none', else error
  • if free (use 'take' if required), LHS owns that expression. No locks.

Only for reftypes. LHS must be owned or linked.

37. More thoughts on ownership/locking

For function returns:

  • take - (default) returns an owned object. Arguments are locked during the call but released when the object is returned. Caller is responsible for destroying returned object (this will happen when it falls out of scope)
  • borrow - returns a borrowed object. Arguments are locked during the call and remain in place until the returned object falls out of scope. Caller is NOT responsible for destroying returned object (it lives inside the associated object) A borrow return REQUIRES a lock on the parameter where the returned value lives
  • lock - N/A

For parameters:

  • take - will take an exclusively owned object. It will no longer be valid in the caller
  • borrow - (default) will borrow the object. Cannot store or return it. Can only call methods on it
  • lock - will borrow the object, LOCKING it in the caller even after the call returns (lock is bound to the life of the returned object). Can store (in a new object only...) or return it. Can only call methods on it

Inline (code):

  • take - will take an exclusively owned object. Target (if a named address) will no longer be valid going forward
  • borrow - (default) will borrow the object. Will lock the target
  • lock - same as borrow.

In arguments:

same as 'Inline'. The specifier is optional and can be included to be explicit/readable. But if the specifier is missing, the ownership type specified in the function definition will still apply which may affect availability of a named var after the call (for lock and take)

It is an error to explicitly provide an incompatible ownership/locking type. eg. borrow arg, when the param is take. Can this happen? if you take arg at the call site for a locked arg, then the call site must keep a lock on the anonymous var until the end of scope.

union is always 2 words, stored inline:

  • the reference (pointer) to the underlying reftype value

39.4. variant

Can only store valtypes within it. Its layout is:

  • a descriminator, marking the specific subtype being stored
  • block of memory equal to the largest size of any of the subtypes

39.5. list

'list' is the only object instance that can have a variable size set at runtime. This will need compiler support. list has a capacity and a length. To destroy a list, each object in it will need to be destroyed. A list of classes is a list of 'pointers'. A list of unions or protocols store the top level of those structures inline. A list of valtypes is a single contiguous block storing the valtypes inline. For a list of reftypes, each element needs to be finalized and destroyed. Finally, the list itself is destroyed in both cases (reftype or valtype).

For homogenous types, the compiler knows the correct destructor to call. For object such as protocols, the protocol object knows how to destroy the underlying object and the protocol object itself is a fixed size.

39.6. protocol / facet

For each protocol/facet... the compiler must generate a constant method table for each object type that adopts a protocol/facet and uses it at runtime (ie. not only used for generic purposes or to restrict a public interface). This can be shared by all objects of that type that are converted to this protocol/facet.

Protocol construction is then allocating a small object that has:

  • one pointer to the method table
  • one pointer to the underlying reftype object. If the underlying object is a valtype, it will need to be 'boxed' into a reftype (requires an allocation then a copy of the value)
  • one pointer to the destructor of the underlying object - to destroy and free its memory

For borrowed protocol instances, the allocation is done on the stack and falls out of scope without destroying the underlying object so that it is available after the scope ends. And there is no destructor for these stored in the protocol instance...

Facet construction is allocating an object that has:

  • one pointer to the method table
  • block of memory equal to the largest size of any of the types that claim to implement this facet. The value is then copied into this block during assignment.

54. Why no short-circuit operations?

Operations are very simple: always left to right; alternate 'call' syntax with one argument.

So boolean 'and' and 'or' are methods on bool that always evaluate their argument.

a: 1 == 1 or (x == 1)
b: 1 == 1 and (x == 1)

But note that if will short-circuit. The first test that fails will halt execution. So, short-circuit 'and' (ie. fail on first false) is accomplished with consecutive 'when' clauses:

# safe is a bool that prevents dangerousfunc from running (when false)
a: if safe when dangerousfunc then true else false

Short-circuit 'or' (pass on first true) must be done with separate when/then clauses.

# picks first non-zero item from first, second, otherthing (default)
a: if first != 0 then first when second != 0 then second else otherthing

  • single identifiers only (no dotted identifiers) that must be refer to declarations within the object body
  • if a subview is included in a view, that subview identifier plus all members of that subview are added to the view being defined
  • an identifier may be included more than once without error (ie. perhaps via common children in included subviews), subsequent inclusions are ignored (it is a set)
  • views may be defined directly as a view argument to an object: this limits the namespace of that object to that view (ie. the children available directly via the defined object name). Usually used to control static methods, constructors etc
  • views defined directly on object namespaces will also be used for limiting available children on instances by default. Although, this can be overridden by specifying this.viewname as the return value for each of the constructors
  • typedef's may ONLY limit with a view specified directly on the typedef definition as an argument. This will then be applied to instances of that typedef (it cannot be overridden because it is forbidden to have constructors for typedefs)
  • If a this.viewname is specified for one constructor, it must be specified for all constructors.
  • views can be defined in the body of an object
  • views defined in the body can be referenced as subviews in the namespace view or in any other view defined in the body or on methods
  • views can be specified with a 'view' argument to function definitions of methods (only). This will limit the parent ('this') object to that view after the method returns. This may require runtime support to record this state.
  • views can be specified in external function definitions (as typename.viewname) provided the view name is exposed. This will limit that function body to only those methods/fields listed in the view specified

63. IDEA: is vs as

An item can have an is clause and optionally an as clause. (TODO: can have as clause by itself? - would be the same as a unit? no)

isspecifies the contents of the object instance.

asspecifies shared functionality and data for an instance (but all shared data must be static and can only be valtypes). Also, interfaces listed here signify that the object supports that interface (whereas if they were in the is clause, they would signify the type of that field in the instance).

If as lists any generic fields, they must be specified first to make the definition 'concrete' before instances can be constructed (eg. (mything int).create) -- the declaration is then like a template. If is lists generic fields, they must be specified as fields in the constructor. Units can only list generics in as because they have no is.

The items in is and as both share the same namespace.

For record/class:

  • is has the fields of each instance. It can specify generic fields (which do not take any memory in the instance). Functions can be specified here but there will be a copy of the function reference in each instance, so this should only be used when the function is to be changed at runtime. All other declarations here are field member names, their types and optionally, their default values.
  • as has shared function definitions optionally including the custom constructor(s). Can have 'template' generic fields. Named interface references to interface definitions or inline interface definitions, indicate that this object supports that interface - and become converter functions on instances to convert that instance to that interface type at runtime (result will be a borrow and will lock 'this'). Any other fields here are static valtype fields that are shared across all instances and are readonly.

For variant/union:

  • is has the list of childtypes for this sum-type (only one of which is 'active' in each instance at runtime). Can have generic fields which then must be declared/specified in the implementor.
  • as has shared function definitions. CANNOT include any constructors (compiler will generate constructors for each child type). Can have 'template' generic fields. Can list interfaces supported by this object (for all child types). Any other fields here are static valtype fields that are shared across all instances and are readonly.

For protocol/variant:

  • is has the list of function specs of this interface, equivalent to the shared runtime table created for each instance that supports this interface. At runtime, interface instances are created that hold a reference to this shared method table and a reference to the underlying instance (or for valtypes, a copy of the instance data itself).
  • as has shared function definitions that may ONLY refer to fields in both the is and as clauses. A protocol/variant can also implement another interface by listing it here that becomes a converter function (same as record/class). Any other fields here are static valtype fields that are shared across all instances and are readonly. CANNOT include any contructors - the compiler will generate a constructor that will take its argument and returns an owned interface instance.

64. NOTE: public / private

publicis an automatically defined unit (if not specified) that includes all members in the current item or unit.

If specified explicitly, it is done so in the as section of the item. For items that can be instantiated, the public specifier applies to both the item namespace ('classmethods/fields') and to the instance namespace.

If you would like to make the publicly accessible namespace of the 'class' different from the public instance methods/fields, then hide that item definition in the containing unit (by not including it in the unit's public interface), then make a subunit that specifies and links to each static element in the item definition that you want to make public. The caller can then use this subunit to access the constructor etc.

myobject: unit {

  } as {

    mymethod2: function in { :this } out xxx is { } # ...
    mymethod3: function in { :this } out xxx is { } # ...
  }

  public: unit {
    create: myobject_internal.create
    create2: myobject_internal.create2
  }
}
