1. IDEA: Pointers, ownership, give/take on valtypes

  • 'this' is always passed as a pointer. object cannot be destroyed within a method, so should be safe even for reftypes
  • more efficient for large reftypes - records. 'this' implies this particular instance so signals that pass by reference is not used
  • can never assign to 'this'
  • can valtypes be passed by 'take'? Yes. But note that this would destroy the var in the caller. Would need to return this value if you want to pass it back. Compiler would need to be smart enough to pass a pointer back on return to avoid copy both ways. Would be more efficient for large records.

  target: myiterabletype.private # do we need to specify private here?

For:

The labelled target of a for (if present) is examined:

- if it is a function or an object method, call it every iteration - NOTE: the function/method is NOT immediately evaluated - if it is an object instance with an iterate method, call that to get the iterator, then call that iterator on every iteration - if it is an object instance with a call method (with or without arguments), call that every iteration (with the supplied arguments) - ownership: the for exposes the correct ownership since the compiler will check the types in the 'call' method of the iterator. If a taken type is returned, it will be owned in the body of the for. If borrowed, it will be borrowed in the body. - scoping in the body ends at the end of the loop or immediately after a break or continue. So a lock on the iterator held by the returned element will be released immediately before the start of the next loop so that the iterator is able to be called to return the next element (which itself will then take a lock for the loop body)

for x: myfunc loop {
  # x is set to myfunc result before every iteration
}

for x: myfunc2 myarg loop {
  # x is set to myfunc result before every iteration
  # function can have arguments
}

for x: (myfunc3 myarg name: mynamedarg) loop {
  # x is set to myfunc result before every iteration
  # function can have arguments, named arguments must be in parens
}

myobject: make_an_object_with_a_call_method
for x: myobject loop {
  # x is set to myobject.call result before every iteration
}

myobject2: make_an_object_with_a_call_method_with_args
for x: myobject2 myarg loop {
  # x is set to myobject2.call result before every iteration
  # 'call' can have arguments, 
  # named arguments must be in parens (not shown)
}

5. TODO: sugar

  • calling a 'type' (object definition) in a runtime expression is the same as mytype.create (with arguments, if any). There can be other constructors with different names but they must be called explicitly (eg. mytype.create2). This also applies to meta.create = meta and also _ (underscore) can be used as _.create. Only for normal expressions, this does NOT happen for generic parameters.
  • 'callable' instances. If *all* apply such that:
    • a term as accessed as the callable within call syntax (ie. with a named argument (call_labelled); or a single unnamed argument (term binop in an operation) or with no argument (term *only* in a binop)
    • that is *not* in an binop with an operator and argument (term id term)
    • that is in a non-constant context (eg. not in a data clause)
    • that term refers to an instance with a call method

      Then, the call method will be implicitly called on that instance. Note that this makes referring to the instance itself more complicated. To refer to the instance itself, an ownership specifier must be used (eg. myinstance.take or .borrow or .lock). For this reason, creating a call method on an object should be carefully considered. Note that simple for sugar (see below) over an interable instance *requires* the default iterator to return a callable instance (with no arguments) as described here.

  • binding to an instance in a for where that instance has an iterate method that returns an iterator that takes no argument, is equivalent to calling that iterate method before any iteration has begun and then using that returned iterator before each iteration. eg. for i: myiterable is equivalent to with _x: myiterable.iterate for i: _x (_x is an example only and will actually be a compiler generated reference that is not otherwise accessible)
  • (this is not really sugar but compiler added attributes. see also take, borrow, lock, limit, generic). Item definitions implicitly have 'public' and 'private' as members that expose the public interface (as defined with the as keyword) and the private interface (all methods/fields of an item). 'private' is only accessable from within the item definition, although external 'friend' methods and items can declare that they support 'item.private' and methods of the orginal item can call/instantiate those functions/items. If those friend methods/items return a borrowed item, then the original item is locked... iterators can be implimented this way
  • Generators with yield is the largest 'sugar'. It creates an entire compiler generated class/record to turn the yielding function 'inside out'

16. What can be generic?

  • function: yes, generic param required
  • class: yes, generic field required
  • record: yes, generic field required
  • union: yes (required for optionals), generic field required
  • variant: yes (same as union), generic field required
  • protocol: yes, declared the normal way (with a shared generic field), then used in the destination (eg class) by declaring the fully qualified protocol name with generic argument, then all related functions must resolve to that consistent type (or all refer to a common generic parameter)
  • enum: no, tag must be concrete numeric type

19. IDEA: how to make a single function that takes a generic numeric item?

Without specialising it. eg a function that can operate on an array of any size.

To fix, define the length as a variable of the type of the generic length variable. Then the compiler enforces that var must be a constant (exactly the same as the generic length 'type') and, since the generic type is a constant, this variable will have a default value equal to the length so need not be separately specified in the caller. Then refer to that variable in the body of the function and the generated function will be deduplicated.

# function is generic over array length 'type' (lt)... will be specialised 
# for all lengths...
# lt = generic length 'type' (constant)
# arr = the array itself declared in relation to the length lt
# l = an actual (runtime) parameter of type lt, so a constant

f: function {lt: int.generic arr: (array u32 to: l) l: lt} is {
    for x: arr {
        # ...
    }
        # l can be used as a constant value anywhere
        print l
        x: l + 1
}

# referencing 'lt' is the same a referencing a literal int BUT will be 
# different for each specialisation (same as being replaced with each 
# different constant). This will not be deduplicated

# BUT, defining an actual parameter as for 'l' above and then referencing
# that WILL be deduplicated because all function AST's will be the same
# (a reference to a parameter name)

23. cast to compile time numeric constant

At unit level:

x: 4

is equivalent to:

x: cast 4 as 4

So x is specifically of type '4' but also an 'i64'

All the const types live at the synthetic namespace at eg. system.const.i64_4 etc?

---

Compile time numeric contants are a separate series of types based on the smallest number of bits (rounded up to nearest power of 2 that is >= 8) that can hold them. Each type has different methods for 'call' which determines their default runtime type, and other conversion methods provided this constant can losslessly be converted to those types. Some rules:

  • default numeric type is i64 for all integer constants that can be stored in an i64. Larger numbers have no 'call' method and must be explicitly converted to a runtime type.
  • default numeric type is f64 for all float constants that can be stored in an f64. Larger numbers have no 'call' method and must be explicitly converted to a runtime type.
  • negative constants have no 'u' methods (eg 'u64') since negative numbers cannot be converted to an unsigned type

The constant types:

  • cip8 - constant positive int that can fit in 8 bits (supported methods: u8 u16 i16 u32 i32 u64 i64 u128 i128 c8 c32) - note, not i8 because it may not fit. Default (result of 'call') is i64.
  • cin8 - constant negative int that can fit in 8 bits (supported methods: i8 i16 i32 i64 i128). Default (retult of 'call') is i64.
  • cip16 - supports methods as for cip8 but NOT u8, i8, c8
  • cin16 - supports methods as for cin8 but NOT i8
  • cip32
  • cin32
  • cip64
  • cin64
  • cip128 - no default (result of call method)
  • cin128 - no default (result of call method)
  • cf16 - constant float that can fit in a f16
  • cf32 - constant float that can fit in a f32
  • cf64 - constant float that can fit in a f64
  • cf128 - constant float that can fit in a f128
  • cc8 - constant character that can fit in 8 bits, always positive (supported methods: u8 u16 i16 u32 i32 u64 i64 u128 i128 c8 c32) - note, not i8 because it may not fit. Default (result of 'call') is c8
  • cc16 - constant character that can fit in 8 bits, always positive (supported methods: u8 i8 u16 i16 u32 i32 u64 i64 u128 i128 c8 c32). Default (result of 'call') is c32

24. Memory format / runtime format (NOTE)

24.1. class

class instances are always references (pointers) to a piece of memory.

But what about a class that has no fields? Can it be created with no allocation? Just held in the typesystem? This would be good for iterators with no fields. Still need a place in memory to hold them? Could be a null pointer? The class could still lock a parent and use it for state management - no, need to store reference to parent somewhere.

24.2. record (and other valtypes)

records are always embedded in a container (local var/stack; part of another reftype or class). A valtype can be 'boxed' by the compiler to convert it into a reftype.

As with class, can a record have no fields so a size of 0?

24.3. union

union is always 2 words, stored inline:

  • a descriminator, marking the specific subtype being stored
  • the reference (pointer) to the underlying reftype value

To store a valtype in a union requires it to be boxed.

24.4. variant

Can only store valtypes within it. Its layout is:

  • a descriminator, marking the specific subtype being stored
  • block of memory equal to the largest size of any of the subtypes

24.5. list

'list' is the only object instance that can have a variable size set at runtime. This will need compiler support. list has a capacity and a length. To destroy a list, each object in it will need to be destroyed. A list of classes is a list of 'pointers'. A list of unions or protocols store the top level of those structures inline. A list of valtypes is a single contiguous block storing the valtypes inline. For a list of reftypes, each element needs to be finalized and destroyed. Finally, the list itself is destroyed in both cases (reftype or valtype).

For homogenous types, the compiler knows the correct destructor to call. For object such as protocols, the protocol object knows how to destroy the underlying object and the protocol object itself is a fixed size.

24.6. protocol / facet

For each protocol/facet... the compiler must generate a constant method table for each object type that adopts a protocol/facet and uses it at runtime (ie. not only used for generic purposes or to restrict a public interface). This can be shared by all objects of that type that are converted to this protocol/facet.

Protocol construction is then allocating a small object that has:

  • one pointer to the method table
  • one pointer to the underlying reftype object. If the underlying object is a valtype, it will need to be 'boxed' into a reftype (requires an allocation then a copy of the value)
  • one pointer to the destructor of the underlying object - to destroy and free its memory

For borrowed protocol instances, the allocation is done on the stack and falls out of scope without destroying the underlying object so that it is available after the scope ends. And there is no destructor for these stored in the protocol instance...

Facet construction is allocating an object that has:

  • one pointer to the method table
  • block of memory equal to the largest size of any of the types that claim to implement this facet. The value is then copied into this block during assignment.

'meta' is a special, dynamic namespace provided by the compiler at compile-time only. All items are readonly (?)

meta can sort-of be used to add additional keywords

  • meta.create - the one true constructor for the current object (since, '.create' is automatically used (see Sugar), can also just call meta ...)

meta.function: current function. Error if not in a function

  • meta.function.name - short name (label)
  • meta.function.path - fully qualified name (dotted)
  • meta.function.reference - the function itself (pointer)... can be used for recursive calls without using the name?
  • meta.function.parent - link to containing object (same as meta.object)

  • meta.object.name - short name (label)
  • meta.object.path - fully qualified name (dotted)

meta.unit: details about the current unit

  • meta.unit.name - short name (label)
  • meta.unit.path - fully qualified name (dotted)
  • meta.unit.reference - the object itself, can be used to refer to the current unit without using its name
  • meta.unit.parent - link to containing unit (compilation error if there is no parent unit (at top level))

meta.definition: details about a named definition ('name' - can be a name within the current scope or a fully qualified varname, it is no withing parens and the name is NOT evaluated).

  • meta.definition name - the root of the meta namespace for 'name'.
  • (meta.definition name).name - short name (label)
  • (meta.definition name).path - fully qualified name (dotted)
  • (meta.definition name).reference - this name's definition (why would you ever use this?)
  • (meta.definition name).parent - parent container of this name? (function/object/unit)
  • (meta.definition name).type - type of this name (as a fully qualified string?)

meta.source: details about the current source file

  • meta.source.name - current filename
  • meta.source.path - current filename full path
  • meta.source.line - current linenumber

meta.define: compiler 'definitions' (like c compiler defines/command line -D values)

  • meta.define NAME - return the string value of the definition NAME. The definitions can be passed in on the command line invocation of the compiler (-D name=value)

meta.environment: access to environment variables (should this be done? Or just use defines?)

meta.compiler: details about the compiler and language

  • meta.compiler.name - name of the compiler
  • meta.compiler.version - version of the compiler
  • meta.compiler.commandline - the commandline passed to the compiler?

meta.language: - details about the language being compiled

  • meta.language.name - name of the language being compiled ('ZeroLang' ?)
  • meta.language.version - version being compiled by the compiler
