This document is the canonical style guide for Zerolang source code. It is prescriptive: each rule states what to do, with a brief rationale only where the rule is non-obvious. Status: the standard library, examples and documentation will be migrated to comply with this guide in the next phase of the project; existing source may diverge in places until that migration lands.

1. General Formatting

1.1. Encoding and File Extension

  • Source files are UTF-8. This is mandated by the language.
  • Source files use the extension .z.
  • Files end with a single trailing newline. No trailing whitespace on any line.

1.2. Indentation

  • Indent with spaces. Tabs are not used.
  • One indent step is four spaces.

1.3. Line Length

A soft limit of 100 columns. Wrap when a line would otherwise harm readability; do not wrap purely to satisfy the limit when the result is worse. Treat 100 as a guideline, not a hard fail.

1.4. One Statement Per Line

Use one executable statement or expression per source line. Semicolons are permitted by the grammar as line separators but should not appear in production code -- line feeds are the natural terminator.

1.5. Braces, Parentheses and Line Continuation

Inside code, parentheses suspend line-feed termination; braces reenable it. Line continuation matters in expressions and statements; definitions filter line feeds inside their value parse automatically and do not need this treatment. The following rules cover wrapping in code:

  • An opening brace follows immediately after its related statement or expression on the same line.
  • When a statement continues after a closing brace, the continuation appears on the same line as that closing brace.
  • A braced or parenthesised clause that fits comfortably on a single line is written on a single line, with no whitespace inside the opening or closing delimiter.
  • A clause that does not fit on a single line places its contents on subsequent lines, indented one extra step (four spaces) deeper than the lines bearing the opening and closing delimiters. The closing delimiter sits on its own line at the parent indent.
  • When a logical line will not fit on one file line, use embedded parentheses or braces to wrap. If the expression has none, wrap the whole expression in parentheses so the multi-line construct reads as a single unit.
  • Within parenthesised expressions and braced clauses (including in parameter lists), introduce newlines wherever they improve readability -- for example, one parameter per line.
# single-line clause: no padding inside the parens
result: (format value: 42 width: 6)

# multi-line clause: contents indented one step beyond the delimiters
result: (format
    value: 42
    width: 6
    pad: "0"
)

1.6. Whitespace Around Identifiers

A single space separates identifiers that would otherwise abut and parse as a single identifier. The numeric and comparison operators are ordinary identifiers and must be surrounded by whitespace.

total: a + b           # correct
# totaL: a+b           # parses as a single identifier 'a+b'

1.7. Definitions and the Colon

Definitions are a label, then :, then an expression or definition. A single space follows the colon when the definition continues on the same line. A space before the colon is forbidden by the language.

Do not horizontally align the right-hand sides of consecutive definitions or their types. Aligned columns are noisy under diff and brittle under rename.

# correct
host: String
port: u16
timeout: u32

# avoid: padded for visual alignment
# host:    String
# port:    u16
# timeout: u32

1.8. Parentheses for Precedence Only

Parentheses are used for precedence enforcement, for clarity in dense expressions, and for the multi-line wrapping case above. Do not parenthesise the conditions of if or for, the subject of a single-line match (see Match and Case), or the arguments of a top-level call.

One exception applies when a top-level call takes an argument that is itself a call with named arguments: the argument call must be parenthesised so it is evaluated before the outer call.

# bare top-level call: no parentheses needed
print "hello"

# nested named-arg call: inner call parenthesised
print (format value: 42 width: 6)

1.9. Numeric Literals

Numeric literals may use underscores to group digits for readability:

  • Decimal literals: groups of three from the right.
  • Binary and hexadecimal literals: groups of four. Use groups of eight only for literals 32 bits wide or larger.

Typed literals use the dot suffix form: 65.u8, not 65u8.

population: 7_900_000_000
mask: 0xFFFF_FFFF
marker: 0b1010_0101
byte: 65.u8

2. Match and Case

When the entire match fits comfortably on one line, write it on one line and omit the parentheses around the subject. Use this form only when every arm is a short statement.

match r case some then { print "found" } case none then { print "missing" }

When the match spans multiple lines so each arm sits on its own line, place the subject in parentheses on its own line. The parentheses let the subject align with the case arms below it. Each case and the optional else sits at the parent indent, immediately after the closing delimiter of the previous arm.

match (
    result
) case ok then {
    print "value=\{result.value}"
} case err then {
    print "failed: \{result.detail}"
}

3. Naming

3.1. Units

Unit names are lowercase printable ASCII with no whitespace; this is mandated by the language for filesystem compatibility. In addition:

  • Aim for a single word.
  • Do not use underscores.

3.2. Valtype Definitions

record, variant and facet definitions use a lowercase name. Aim for a single word. If the name needs more than one word, run the words together without underscores.

point: record { x: i64 y: i64 }
filekind: variant {
    file: null
    dir: null
    symlink: null
    other: null
}

3.3. Reftype Definitions

class, union and protocol definitions use PascalCase. The casing distinguishes reftypes from valtypes at the point of use. Compound names split on word boundaries: BufWriter, not Bufwriter; ListView, not Listview.

The predefined reftypes built into the language follow the same rule: String, StringView, Option, OptionView, Result, Any, Text.

File: class { ... }
ListView: class { ... }
ReadStream: protocol { ... }

3.4. Variant Tags

Variant arm names are values, not types, and are written lowercase. Aim for a single word; if more is needed, use lowerCamelCase.

filekind: variant {
    file: null
    dir: null
    symlink: null
    other: null
}

3.5. Functions, Methods and Fields

Aim for a single lowercase word. If the name needs more than one word, use lowerCamelCase.

  • Getters list the name of the item being returned only. Do not prefix getters with get.
  • Setters may be prefixed with set when the alternative would be ambiguous.
  • Predicate methods returning bool may be prefixed with is or has.
# correct
f.size # not f.getSize
user.setName name: "alice"
conn.isOpen
doc.hasErrors

3.6. Constants

Top-level value definitions use lowerCamelCase. Do not use uppercase or shouting snake case.

maxRetries: 3
defaultPort: 8080

3.7. Receivers

The receiver in a method should be a short name -- preferably a single letter -- that relates to the type. For multi-word reftype names, an acronym is acceptable.

  • All methods on the same type must use the same receiver name.
  • The :this shorthand is permitted but a short type-relevant name is preferred.
File: class { ... } as {
    open: function {f: this name: StringView} out File is { ... }
    close: function {f: this} is { ... }
    size: function {f: this} out u64 is { ... }
}

ListView: class { ... } as {
    length: function {lv: this} out u64 is { ... }
    first: function {lv: this} out (Option t: of) is { ... }
}

3.8. Platform and Architecture Identifiers

Platform and architecture identifiers may use their conventional external spelling and are exempt from the no-underscores rule. Identifiability across the wider ecosystem (Linux, Rust, Go, package metadata) outweighs the in-language consistency cost.

archkind: variant {
    x86_64: null
    aarch64: null
    other: null
}

4. Calls

When a function or method call has exactly one argument and that argument is the leading parameter, omit the argument name.

print "hello"
outbuf.append 65.u8

Re-introduce the argument name when omitting it would obscure the call's intent at the call site. The judgement applies most often to native or generic methods where a bare argument's role is not obvious from the receiver and method name.

# clear from receiver and method name -- omit
List.append item

# bare argument's role is not obvious -- name it
buf.write from: chunk

5. Item Definitions

When an item is generic, declare each generic parameter in the as section as name: <Constraint>.generic. Place the generic declarations first in as, before any conformances or static members. Instance fields go in the is body, where they may reference the parameter names directly.

Pair: class {
    first: a
    second: b
} as {
    a: Any.generic
    b: Any.generic
}

6. Functions

6.1. Guard Clauses and Early Return

Use if guards with early return to handle errors and edge cases up front. The remaining body is then the happy path with one less indent level.

read: function {f: this} out (Result t: String) is {
    if not f.isOpen then return Result.err detail: "not open"
    if f.atEnd     then return Result.err detail: "eof"
    ...
    return Result.ok value: data
}

6.2. Last-Expression Return

For short functions, return the last expression and omit the explicit return.

abs: function {n: i64} out i64 is {
    if n < 0 then 0 - n else n
}

6.3. Expression Forms

Prefer if and for as expressions to produce values. Avoid side-effecting an accumulator when an expression form is available.

# prefer
squares: for x: 5.iterate loop { x * x }

# avoid
squares: List of: i64
for x: 5.iterate loop { squares.append x * x }

7. Ownership Annotations

Ownership annotations behave differently in parameter declarations and in inline code. The two are covered separately below.

7.1. Parameter Declarations

  • .borrow is the default for in parameters. Do not write .borrow on an in parameter.
  • .take is the default for out parameters. Do not write .take on an out parameter.
  • .borrow on an out parameter is the only place .borrow is written explicitly. It indicates a borrowed return; at least one in parameter must be marked .lock to provide a parent for the borrow's lifetime.
  • .lock has no default. Always write it explicitly when locking is required.
# in: borrow is implicit; out: take is implicit
area: function {p: point} out f64 is { ... }

# borrowed return: explicit .borrow on out, .lock on the parent
first: function {l: List.lock t: of} out of.borrow is { ... }

7.2. Inline Code

  • Do not write .borrow in function bodies or on call arguments. Borrow is implicit at every position where it is the default.
  • Use .take only when there is a receiving variable or parameter of that type. The compiler enforces .take based on the receiving parameter; the call-site annotation is rarely needed for safety.
  • As a general rule, omit .take on call arguments. Add it only where a specific call's readability is meaningfully improved.
  • .take may be used to move a reftype to a new variable name when the new name improves documentation or clarity. Do not rename for its own sake.
  • .take as a bare expression to release a reftype is forbidden. Use .release.
  • .release is a standalone statement. It returns null and may not appear inside an assignment, expression, or call argument.

7.3. Lock and Release Methods

  • .lock is permitted as a borrow-projection on a reftype variable to produce a borrowed reference for the duration of an expression or protocol creation.
  • .release on a borrowed variable, written as a standalone statement, ends the borrow.
borrowed: provider.borrow from: m2.lock
borrowed.use
borrowed.release
print "m2 still alive: factor=\{m2.factor}"

8. Comments and Documentation

8.1. Line Comments

Zerolang line comments begin with #. Comments document why, not what; do not paraphrase code that names itself. Place comments on the line above the item they describe, not trailing the code.

Reach for a comment when the reader needs context the code cannot give: non-obvious invariants, lock ordering, ownership intent on exported APIs, references to specifications.

8.2. Native Declaration Doc Blocks

For native declarations in standard library units, precede the declaration with a short doc block. The first line is the declared identifier, two dashes, and a one-sentence description. Further lines describe parameter and return semantics.

# env -- look up environment variable `key`. Returns Option.none if
# the variable is not set; Option.some with the value otherwise.
env: function {key: StringView} out (Option t: String) is native

9. Errors and Panics

error msg: "..." is the compile-time form; panic msg: "..." is its runtime counterpart. Apply the same phrasing rules to both:

  • Lowercase first letter.
  • No trailing punctuation.
  • Present tense, declarative.
  • State the failed condition, not a stack-trace narrative.
if size > limit then error msg: "buffer exceeds maximum capacity"

if conn.closed then panic msg: "write on closed connection"

10. File Hygiene

  • One trailing newline at end of file.
  • No trailing whitespace on any line.
  • A unit declaration, when present, sits at the very top of the file, before any imports or item definitions.
