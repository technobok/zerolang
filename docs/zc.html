1. Synopsis

zc <command> [arguments] [flags]

2. Description

zc is the self-hosted zerolang compiler. A command is required: it compiles a zerolang unit (.z) and links it into a native executable (build), builds and runs it (run), emits C (emit), dumps the compiler's internal model (dump), explains a diagnostic code (explain), prints resolved settings (env), or lints and formats source (lint / fmt, thin shims for the standalone zl tool).

The compiler operates on units — each source file defines a unit, and dependencies are discovered automatically through the Virtual File System (VFS), which mounts the user source directory over the system library so user units shadow system units of the same name.

zc is self-locating: run from any directory, it finds its bundled standard library and runtime relative to its own executable (see Tree resolution), so ordinary use needs no flags.

3. Commands

The first argument is always a command. A bare unit name with no command is an error — use zc build <src>.

  • build <src> — compile to a native executable; output is -o OUT or ./<unit>.
  • run <src> [args...] — build to a temporary file, then run it with an inherited terminal, forwarding the trailing args and propagating the program's exit code.
  • emit <src> — emit standalone C source to -o OUT (or stdout).
  • dump <src> — write the typecheck model as SQL (-o or stdout); --canon writes the id-normalized model instead.
  • explain <code> — print an explanation for a diagnostic code, for example zc explain E0200.
  • env — print resolved tool paths and settings (see Inspecting the environment).
  • lint [args...] and fmt [args...] — thin shims that locate the sibling zl binary (the standalone linter/formatter), forward the arguments, inherit the terminal, and propagate its exit code. The rules, flags and formatter semantics live in docs/zl.pdoc. make check invokes bin/zl directly.
  • version — print the version.
  • help [command] — print usage, or detailed help for one command.

For run, flags before the unit configure the build; everything after the unit (or after a -- separator) is passed to the program.

zc build hello                # build ./hello
zc run hello arg1 arg2        # build + run, forwarding arg1 arg2
zc build hello --release -o hello
zc emit hello -o hello.c
zc explain E0200

-h / --help and -V / --version are also accepted in any position, so zc build --help prints the help for build.

4. The source argument

build, run, emit, and dump take a <src> — either a unit name (hello) or a path to a .z file (hello.z, ./sub/hello.z). A path — anything containing / or ending in .z — sets --src to the file's directory and the unit to its base name, unless --src is given explicitly.

Because the command always comes first, a unit whose name collides with a command is reached through the explicit form: zc build run or zc build ./run.z to build a unit named run.

5. Flags

5.1. Global flags

These are accepted in any position.

  • -h / --help — print usage; after a command, print that command's help.
  • -V / --version — print the version.

5.2. Build flags

These apply to build, run, emit, and dump.

  • --src DIR — user source directory (default: the current directory).
  • --system DIR — system library directory (default: self-located).
  • --runtime DIR — C runtime fragment directory (default: self-located).
  • --full — type-check every definition, not just those reachable from main.
  • -o OUT — output path (the executable for build, the C file for emit, the model for dump).
  • --release — optimize (-O2); the default is a debug build (-O0 -g).
  • --save-temps — keep the generated C file and print its path (build / run).
  • --cc CC — the C compiler to invoke.
  • --target TRIPLE — cross-compile (see Cross-compilation).
  • --cflags STR / --ldflags STR — extra compiler / linker flags, split on spaces.
  • --canon — dump only: write the id-normalized model.

6. Tree resolution

zc locates its system library and runtime by this precedence, first match wins:

  • an explicit --system / --runtime flag;
  • $ZEROLANG_SYSTEM / $ZEROLANG_RUNTIME;
  • $ZEROLANG_ROOT (a whole tree: $ZEROLANG_ROOT/lib/system and lib/runtime);
  • the executable's own location — os.exePath gives the real binary path (resolving a bin/zc symlink) and the root is its grandparent; <root>/lib/system plus <root>/lib/runtime, falling back to <root>/src/runtime so a dev checkout works too;
  • the current directory (lib/system, src/runtime) — the final fallback.

--src (user code) always defaults to the current directory, never the install tree. The C compiler is chosen as --cc > --target-derived (<triple>-gcc) > $CC > cc.

7. Building and running

build type-checks the unit, emits self-contained C to a temporary file, and invokes the C compiler to produce an executable. Because the runtime is inlined at emit time, the single .c needs no extra include paths. The debug default is -O0 -g; --release is -O2. --save-temps keeps that temporary .c and prints its path.

run builds the same way to a temporary location, then spawns the executable with the parent's stdin / stdout / stderr (a live terminal), forwards the trailing program arguments, and exits with the program's own exit code.

8. Cross-compilation

--target TRIPLE selects a cross compiler: when --cc is not given, the compiler name is <triple>-gcc (for example x86_64-w64-mingw32-gcc). The executable name gains a .exe suffix for *windows* triples.

Caveat: cross-compilation is wiring only. Programs that use POSIX-only runtime natives (os.spawn, io file operations, directory walks) will not link for Windows until those natives gain a Win32 backend — a follow-on. Core, POSIX-free programs cross-build where the cross cc is installed.

9. Inspecting the environment

zc env prints the resolved tool paths and settings as KEY=VALUE lines — ZEROLANG_ROOT, ZEROLANG_SYSTEM, ZEROLANG_RUNTIME, ZC_CC, ZC_TARGET, ZC_VERSION, and ZC_EXE. It honours the same flags as a build, so zc env --target x86_64-w64-mingw32 reports the cross compiler that would be used.

zc env
zc env --target x86_64-w64-mingw32

10. Installation

make install builds bin/zc and copies a self-contained, GOROOT-style tree to ROOT (default ~/.local/lib/zerolang), symlinking BINDIR/zc (default ~/.local/bin/zc) to it:

make install
make install ROOT=/opt/zerolang BINDIR=/usr/local/bin

The tree contains bin/zc, lib/system, lib/runtime (copied from src/runtime), docs, and src. Because os.exePath resolves the symlink to the real binary, the installed zc finds its stdlib and runtime from any working directory.

11. Exit Codes

  • 0 — Success. For build the executable was produced; for run the program ran (its own exit code is propagated); for emit and dump the output was written.
  • 1 — Compilation failed. One or more errors were found during parsing or type checking. Error messages are written to stderr.
  • 2 — Usage error. Unknown or missing command, invalid unit name, or invalid directory paths.
  • other — for build / run, a non-zero exit from the C compiler or from the program itself is propagated unchanged.

12. Error Format

The compiler uses a rustc-inspired error format designed for readability and tool integration. Each error includes an error code, source location, and the relevant source line with a caret pointing to the error.

12.1. Basic Error Format

error[E0008]: undefined identifier: x
 --> MyFile.z:5:10
     |
   5 |     print x
     |           ^

The first line contains the error code (E0008) and the error message. The second line shows the file path, line number, and column number in the standard File:line:col format used by gcc, clang, and rustc. The source line is displayed with a caret marker pointing to the exact location of the error.

12.2. Notes and Hints

Some errors include additional context as notes (explaining why the error occurred) or hints (suggesting how to fix it):

error[E0200]: function returns 'borrow' but has no 'lock' parameter
 --> MyFile.z:3:1
     |
   3 | f: function out string.borrow is { return "hello" }
     | ^
     = hint: add .lock to a parameter to borrow from it
error[E0008]: undefined identifier: pritn
 --> MyFile.z:2:5
     |
   2 |     pritn "hello"
     |     ^^^^^
     = hint: did you mean 'print'?

12.3. Error Code Numbering

Error codes are grouped by category:

  • E0001–E0099: Parser and syntax errors — malformed source code, unexpected tokens, missing delimiters, invalid definitions
  • E0100–E0199: Type resolution errors — type mismatches, undefined types, incompatible assignments
  • E0200–E0299: Ownership errors — invalid take/borrow/lock usage, use after ownership transfer, lock violations, reftype aliasing
  • E0300–E0399: Call and argument errors — wrong number of arguments, named argument mismatches, type mismatches in function calls
  • E0400–E0499: Generic and monomorphization errors — missing type arguments, failed type inference, invalid generic constraints

Run zc explain <code> for a longer description of any code (for example zc explain E0200). An error count summary is printed at the end:

3 errors found

13. Compiling Emitted C

zc build and zc run invoke the C compiler for you. If you instead emit C, the result is a standalone file that compiles with any C17 compiler — the runtime is inlined, so no extra include paths are needed:

zc emit hello --src examples -o hello.c
gcc -std=c17 -Wno-unused-function -o hello hello.c
./hello

For development, compile emitted C with the sanitizers to catch memory errors:

gcc -std=c17 -fsanitize=address,undefined -Wno-unused-function -o hello hello.c
./hello

14. Examples

Build and run a program:

zc run hello --src examples

Build a release executable at a chosen path:

zc build myapp --src src --release -o build/myapp

Build straight from a file path:

zc build ./examples/hello.z

Emit C and compile it yourself:

zc emit hello --src examples -o hello.c && gcc -std=c17 -o hello hello.c && ./hello

Generate SQL diagnostics:

zc dump hello --src examples | sqlite3 :memory:

Full type-check of a library (not just what is reachable from main):

zc dump mylib --src lib --full

Explain an error code, and inspect the resolved toolchain:

zc explain E0200
zc env
