1. Introduction

This document is the architectural map of the zerolang typechecker. It is intended to be read before opening src/ztypecheck.z -- after reading it, a contributor should know which structures the typechecker maintains, how its code is partitioned, and where a particular feature lives.

The companion document compiler.pdoc covers semantics -- what ownership means, how the multi-granularity lock-conflict rule works, what demand-driven resolution guarantees. This document covers structure: data layout, dispatch, code organisation. The two documents are deliberately split so that semantic rules have a single source of truth (compiler.pdoc / ownership.pdoc) and this one stays a navigation aid that does not rot when rule wording changes.

Audience. A new contributor who has already read spec.pdoc, ownership.pdoc, and the Semantic-Analyser section of compiler.pdoc.

2. Pipeline Position

The typechecker sits between the parser and the C emitter:

  zast.ProgramData  (frozen post-parse)
      ↓   typecheck(program)
  ztyping.ZTyping  (mutable, holds all typecheck output)
      ↓
  zemitterc / zsqldump

The parser hands the typechecker a zast.ProgramData -- an immutable tree of Node values, each with a monotonic nodeid. The typechecker does not mutate AST nodes; instead it writes its results into a single mutable container, ztyping.ZTyping, keyed by nodeid. After typecheck() returns, the emitter and SQL dumper read from both the ProgramData (shape) and the ZTyping (types, ownership, locks, monomorphisations) without further analysis.

Demand-driven, single pass. Resolution starts at main and descends only into names that are actually referenced. Names that are never reached are never typechecked. The same property gives cycle detection for free -- a name that is encountered while it is already on the resolution stack is reported as a circular dependency. See compiler.pdoc:Demand-Driven Resolution for the rationale.

3. Core Data Structures

The typechecker is built around five core record types and one mutable container. Reading this section first makes ztypecheck.z considerably easier to follow, because nearly every function in it produces, mutates, or consults one of these.

3.1. ZType -- the unified type descriptor

ZType (in src/ztypes.z) is the only type representation in the compiler. There is no separate "kind", "shape", or "schema" object: every type -- record, class, variant, union, protocol, facet, function, unit, numeric, native -- is a ZType with a typetype discriminator (the ztypetype enum) and a set of optional fields.

Key fields:

  • name: the simple name (e.g. Point)
  • typetype: the ztypetype enum (RECORD, CLASS, VARIANT, UNION, PROTOCOL, FACET, FUNCTION, UNIT, numerics, ...)
  • enclosingTypeId: the enclosing unit type, or none for top-level units
  • childrenIdMap: per-name monotonic child id, minted lazily by allocChild. The id is the authoritative key for children in ZTyping.typeChild rows and for narrowing entries that need a numeric subject
  • returnTypeId, returnOwnership, thisParamName: populated for function types
  • generic-state fields: genericParams, genericDefaults, genericOriginId, numericGenericParams, isTagGenericOrigin
  • constValue: a literal value carried on a ZType used as either an as-block constant or a numeric generic-arg carrier (the 8 in array<i64, 8>)
  • layout / memory metadata: destructorName, isHeapAllocated, needsFieldCleanup, isNullablePtr, isBox (see representation_of_types.pdoc)
  • createDisabled, metaCreateId: unified-call-dispatch metadata: bare-name calls route through children[create] unless disabled, with metaCreateId holding the raw allocator
  • cname: the C identifier the emitter uses
  • typeId: monotonic id for SQL dumps

A ZType carries identity-data only. Per-edge metadata -- parameter ownership annotations, field defaults, lock/private modifiers -- lives on ZTyping.typeChild rows (the flat children table), not on the parent ZType.

3.2. ZVariable -- per-binding ownership carrier

ZVariable (src/ztypes.z) wraps a ZType with the state that is per-variable rather than per-type. A ZVariable is created only for named runtime bindings -- function parameters, locals introduced by assignment, with-bindings, for-loop iterators, and synth temps -- and referenced from ZEntry.varId. Expression-level results live on ZExprResult, not here; type and function definitions leave ZEntry.varId unset.

  • variableId: monotonic id
  • ztypeId: the variable's resolved type
  • ownership: OWNED or BORROWED (the 2-state model described in compiler.pdoc:Ownership and Lifetime)
  • isPrivateAccess: carried through .private typedef projections
  • borrowOrigin: escape-analysis sentinel naming the function-local source this variable borrows from; none for parameters
  • synthOrigin: pass-name string for variables synthesised by a compiler pass (e.g. "anf" for hoisted call temps); none for user-source variables

Keeping ownership off ZType lets the same type be bound in two scopes with different ownership, which is the common case.

3.3. ZEntry -- a scope binding

ZEntry (src/ztypes.z) is the unit of scope state. Every piece of information the scope chain needs to track about a name lives on an ZEntry:

  • entryId: monotonic id
  • name, ztypeId, isDefinition
  • varId: optional ZVariable id (absent for type/function definitions and lock-only overlays)
  • hasLock, lockType, lockHolder, lockPath: lock state -- the lock kind, its holder, and the addressable path tuple (root, f1, f2, ...)
  • narrowedSubtype, narrowedSubtypeId, excludedSubtypes, excludedSubtypeIds: type narrowing for union / variant subjects in match arms, with id parallels for the future hot path
  • originalZtypeId: the un-narrowed type, so the emitter can generate the C-level unwrap
  • isTaken, takenAt: consumed-binding state

An entry is either a definition (it introduces a name) or a shadow / overlay (it adjusts state for a name defined in an outer scope). Definitions live in BLOCK scopes; shadows live in OVERLAY or CALL scopes and disappear automatically when their scope is popped. See compiler.pdoc:Scope Environment for the three scope kinds.

3.4. ZExprResult -- the result of checking an expression

Most expression-level helpers return ZExprResult (src/ztypes.z), not a bare ZType:

  • ztypeId: the inferred type
  • borrowTarget: the addressable path tuple (root, f1, ...) that this expression borrows from; the enclosing checkAssignment / call-argument check consumes it to install a borrow-scoped lock at the binding site
  • privateAccess: propagated through .private projections

The borrowTarget mechanism keeps borrow-lock installation out of deep expression code: the leaf method that knows the source path stamps it onto the ZExprResult, and the eventual binding site installs the lock. The flow is described under ownership.pdoc:Borrow propagation.

3.5. ZTyping -- the typecheck-output container

ZTyping (src/ztyping.z) is the single mutable container holding everything the typechecker computes. It is intentionally ECS-shaped: instead of decorating parsed AST nodes with extra columns, the typechecker writes into a set of per-nodeid component maps.

Aggregate state:

  • errors, isError: error accumulator
  • monoTypes, monoFunctions: monomorphised generic instantiations, each paired with its origin AST definition
  • funcAliases, clonedMethods, resolved: monomorphisation deduplication caches
  • unitTypesById: Unit-AST nodeid → resolved unit ZType

Component tables (each keyed by parsed-node nodeid):

  • nodeType, nodeConst: per-Node type and compile-time constant
  • callKind, callCallableTypeId, exprCallKind: call classification and control-kind routing
  • doHasBreak, caseSubjectTaken, forIterBindings: per-construct flags
  • ifTakenVars, caseTakenVars: post-branch ownership-cleanup lists
  • atomNarrowedSubtype, atomOriginalZtype, atomChildId: per-AtomId narrowing stamps
  • dpParentTaggedType, dpChildId: per- DottedPath parent-type and child-id stamps
  • withOwnership, withAliasOf, assignAliasOf: per-binding alias/ownership stamps
  • projectedArgs: per-NamedOperation protocol-projection stamps
  • typeChild, typeGenericArg: flat children / generic-args tables

The SQL dumper writes one row per component entry, so every table has a direct SQL representation.

3.6. ZSymbolTable and ZScope

ZSymbolTable (src/zenv.z) is the scope stack the typechecker uses for all variable state -- type, ownership, lock state, narrowing, taken state. There is no separate flow- typing structure: state changes are recorded as new entries in overlay scopes; scope pop is the only thing that undoes them.

  • ZScope: a kind (BLOCK, CALL, or OVERLAY), a name, a monotonic scopeId, and an entryStart index into the table's flat entry list. Entries are scanned linearly (typical scope has fewer than ten); unreachable scopes are tracked in a side set
  • ZSymbolTable: a stack of ZScope frames over a flat entries list, plus history (popped scopes, retained for SQL dump) and scopeLog (append-only log of push/pop events with parent ids and sequence numbers)

Three scope kinds (full description in compiler.pdoc:Scope Environment):

  • BLOCK -- language constructs (function, do, for, if, with, match arm). Can contain definitions.
  • CALL -- pushed before processing call arguments, popped after the call. Locks installed here vanish when the call returns.
  • OVERLAY -- shadow records for a variable defined in an outer scope (new lock, narrowing, taken state).

Lock state is keyed on a path tuple (root, f1, ..., fn) stored on ZEntry.lockPath. ZEntry.name always equals path[0] so the linear scan stays cheap; the full tuple is consulted only by the prefix-overlap rule. Key APIs (all on ZSymbolTable):

  • lookup, lookupVarId, lookupVarNameById -- name → type / variable resolution
  • tryLock -- attempt to install a lock; returns the installed lock or a conflict description. Multi-granularity rule is in lockAcquireConflict
  • findExclusiveLock, isPathLocked -- access- site enforcement (used by lock-escape checks)
  • narrow, exclude -- install a narrowing or exclusion overlay entry on the current scope

The lock conflict logic lives in zenv.z, not in ztypecheck.z. The typechecker only chooses when to install a lock; the rule for whether it conflicts is owned by the scope module.

5. Dispatch Mechanism

Per the project convention (CLAUDE.md), the typechecker dispatches on the nodetype discriminator rather than runtime type tests. The pattern appears in two forms:

  • Closed dispatch keyed by #~nodetype"" -- e.g. the definition resolvers. Used where the set of cases is closed and known up front (type-definition resolvers, statement-line kinds).
  • Inline match on node.nodetype -- used inside checkExpression, checkValue, and a few similar high-fan-out helpers where a flat match reads more clearly than a table.

After the discriminator check, cast() narrows the node to the concrete Node arm; see src/zast.z.

6. Resolution Stack and Cycle Detection

Resolution is keyed by definition nodeid, not by synthesized dotted-string names. Every definition is one AST node with one stable nodeid, and that node is shared across every reference to it, so the same definition always resolves to the same cached type no matter which unit asks. A resolved cache maps defn.nodeid -> ZType; a list-shaped resolution stack records the definitions currently being resolved (each frame carrying its unit name, def-id, ztype, and AST node). The flow is:

  1. Unit-name resolution looks the definition up in the unit's member table, then checks the resolved cache keyed by defn.nodeid. Hit: return immediately.
  2. Cache miss: push a resolution frame for defn.nodeid onto the stack and dispatch to the appropriate resolve* resolver by nodetype.
  3. If a nested resolution re-enters the same definition, the resolver finds its def_id on the stack and reports a cycle error.
  4. On success, the resolver pops the stack frame and writes the final ZType into the resolved cache under defn.nodeid.

The emitter and SQL dump read ZTyping.resolved, a qualified-name ({unit}.{member} -> ZType) projection of the nodeid cache built at the end of typecheck.

The type keyword (in receiver position) resolves to the type at the bottom of the resolving stack, so a method body can refer to its own enclosing type without naming it. this works similarly.

7. Type-name resolution and the canonical-id rule

A bare type name (the i64 in x: i64, a conformance marker, a field or parameter type) resolves current-unit-relative, mirroring the module search path in units.pdoc: the current unit first, then the stdlib units (system, collections, io, core, os, cli). First match wins, so a type defined in the current unit shadows a same-named stdlib builtin. Two functions implement this:

  • definingUnitFor(currentUnit, name) -- the declare-before-bind index (built upfront by indexUnitDefs): which unit defines name for a reference in currentUnit, current-unit first. resolveAtomType routes a name atom through it, get-or-declaring the ONE canonical id via declareCanonicalType.
  • lookupTyperef(mainName, base) -- looks base up in the resolved cache over the same precedence. Despite its parameter name, mainName is the search-root unit: the current unit at most call sites, the program's main unit at a few. It is not unconditionally main-rooted.

Cross-unit references are dotted (zast.Node, io.File), resolved by the named unit; bare names never reach across units except to the stdlib fallback. (The same-named-directory submodule feature in units.pdoc -- a unit foo.z with a private foo/ directory -- is implemented in the parser's loadUnit; the current corpus and the self-hosted compiler use flat units + dotted references, so its bare-name resolution is not exercised end-to-end at typecheck.)

The canonical-id rule. A definition resolves once, into one canonical type id keyed by its defn.nodeid (with defined_in_unit recording the owning unit). That id is the type's identity -- not its name. Downstream passes trust the locked id and never re-resolve a type by its name: the C emitter derives a type's C form from the id (via typeRefC / scalarCTypeFor / cTypeForNameTid), never from the name, so a user type that shadows a builtin scalar (i64: record { ... }) emits its struct, while the system i64 reached through the same name keeps the C scalar. Re-resolving by name would collapse the two onto whichever the search-root unit wins -- the bug class the make shadow-guard ratchet prevents from returning.

8. Cross-Cutting Concerns

Each subsection below points at where a concern lives in the typechecker. For why it works that way, follow the links to the semantic documents.

8.1. Ownership enforcement

The 2-state ownership model (see compiler.pdoc:Ownership and Lifetime) is enforced at the assignment and call sites:

  • checkAssignment -- definition site. Records the new variable's ownership in a fresh ZVariable, consumes any ZExprResult.borrowTarget by installing a borrow lock.
  • checkReassignment -- enforces that the LHS variable is currently OWNED and that the RHS provides the appropriate ownership; resets narrowing and re-runs lock escape checks.
  • checkSwap -- symmetric ownership preservation.
  • the argument-site .take -- transfers ownership from the caller's binding into the callee's parameter.

Parameters default to .borrow; .take and .lock are the opt-in transfers. The annotations live on the ZTyping.typeChild rows (the param-ownership column), not on the parameter ZType.

8.2. Lock checking

The typechecker chooses when to install a lock; zenv.z owns the conflict rule (lockAcquireConflict) and the prefix-overlap path test (pathsOverlap). Two scopes of locks, both described in compiler.pdoc:Lock Checking:

  • Call-scoped -- installed by lockCallArg inside a CALL scope that wraps argument processing. The leaf is EXCLUSIVE; every intermediate prefix is SHARED.
  • Borrow-scoped -- installed by installBorrowLocks when a binding takes a borrow into a BLOCK scope (.borrow, .lock, .stringview, with, or a .lock-param transfer). Persists for the lifetime of the binding's scope; released by scope pop or explicit .release.

The aggregate-lock-escape check runs at storage sites (field writes) and at return sites; it consults isPathLocked on the scope chain to reject a write/return that would let a locked value escape its scope.

Self-overlap during a single call (e.g. f.method bv: f.byteview) is permitted via the call-identity stack and zlockholder tagging: tryLock skips the conflict check when the existing holder matches the current call's identity.

8.3. Borrow correctness for parameters

Deep methods that take a borrow into a return value record the source path on a pending-borrow slot, which checkExpression captures into ZExprResult.borrowTarget and clears so it cannot leak across statements (see takePendingBorrow / drainPendingBorrow). The binding site (checkAssignment or the call-argument check) then installs the borrow lock against the actual destination scope. The dedicated .take / .borrow / .lock / .release / .private handling lives in checkDotted.

8.4. Monomorphisation

Generic types and functions are monomorphised by deep-cloning the original AST subtree, binding the generic parameters to concrete types, and re-running the standard resolvers on the clone.

  • mintMono -- entry for generic type instantiation. Builds a deduplication key via genericTypeKey (over arg (name, constValue) pairs so numeric generics also dedupe), reuses an existing instantiation if present, otherwise clones and registers into ZTyping.monoTypes.
  • cloneUnitMethod -- per-function variant. Clones the function AST, binds generic params, typechecks the clone with the mono context in scope.
  • validateGenericTypeArgs -- enforces generic constraints on a candidate argument type.

Cloned nodes carry synthOrigin (the pass-name string) so the SQL dump can attribute them to monomorphisation rather than to the user. Node synthesis is described in compiler.pdoc:Synthesis Toolkit; the same machinery is used by per-call argument hoisting.

is native methods (the compiler-provided collection methods on array, List, Map, str, etc.) are not special-cased -- they are cloned alongside user methods. The only requirement is that their declaration in lib/system/system.z or lib/system/collections.z carries the is native form. Resolution flows through the standard child-lookup path.

8.5. Type narrowing

Narrowing for union and variant subjects in match arms is implemented via overlay entries. When a match arm narrows subject s to subtype ok, an overlay ZEntry with narrowedSubtype: "ok" is pushed onto the arm's scope. Inside the arm, name lookup for s finds this entry first and returns the narrowed type. The arm scope is popped at end of arm; narrowing resets automatically.

Post-match exclusion is the dual: arms that diverge (return, break, panic) contribute their handled subtypes to an excludedSubtypes entry placed on the parent scope after the match. Subsequent code that re-accesses s as an excluded subtype is rejected.

Assignment-based narrowing (e.g. x: Result.ok 42) places a narrowing entry in the current scope at definition time; reassignment resets it.

Both narrowing and exclusion carry id parallels (narrowedSubtypeId, excludedSubtypeIds) minted via ZType.allocChild. String fields remain authoritative; the ids are the hot-path key prepared for future migration to the SQL-backed representation.

8.6. Auto-call coercion

A no-user-arg method dotted path read in value position (v: s.stringview, p1.distance inside a string interpolation) is auto-called to its return type. The coercion lives entirely in checkDotted and is gated by a coerce-to-return flag. checkCall clears the flag when resolving its own callable, so explicit calls (obj.method named: arg) see the function type and dispatch normally instead of constructing the return type. Path resolution itself is context-free: it always returns the FUNCTION type for a method path. Behaviour is identical for is native and user-defined methods.

8.7. Unified call dispatch

A type used in callable position is rewritten through the unified-dispatch route. The bare-name construction call Point x: 1 y: 2 resolves the callee to the Point ZType, then looks up children[create] on that type to get the actual constructor. The mechanics live in the return-call dispatch; meta.create on the parent ZType carries the compiler-generated raw allocator when one is needed. create: null in the type's as block disables bare-name construction (the case for union and variant where subtype selection is required).

9. Error Reporting

Type errors are zast.ErrorData value objects, appended to ZTyping.errors via the addError helper. Errors are never raised as exceptions; the typechecker tries to continue collecting errors past the first failure where the AST permits. Error codes (the err variant in src/zast.z) are grouped:

  • E0001-E0099 -- parser / syntax (raised earlier in the pipeline)
  • E0100-E0199 -- type resolution
  • E0200-E0299 -- ownership
  • E0300-E0399 -- call / argument
  • E0400-E0499 -- generic / monomorphisation

The diagnostic location is taken from the offending AST node's loc / Token. After typecheck() returns, the caller inspects ZTyping.errors to decide whether to proceed to emission.

10. What Lives Elsewhere

Module boundaries the typechecker relies on:

  • src/ztypes.z -- ZType, ZVariable, ZEntry, ZExprResult, ZTypeRegistry, plus the ztypetype / zownership / zparamownership / zlockstate / zscopekind / zsubtype / zcontrolkind / zbuiltinfunc / zlockholderkind / zlockholder enums, and the shared manglers mangleFuncName / mangleVarName
  • src/ztyping.z -- the ZTyping container, ZTypeChild / ZTypeGenericArg flat-table rows
  • src/zenv.z -- ZSymbolTable, ZScope, ZScopeLogRow, the lock-conflict logic (lockAcquireConflict, pathsOverlap), narrowing / exclusion installers

Collection-type predicates (typeIsReftype, typeIsValtype, ...), node synthesis, and the monomorphisation dedup hashing have no separate module -- they live within src/ztypecheck.z itself.

11. Reading Order

When opening the typechecker for the first time, read in this order:

  1. src/ztypes.z -- the type definitions in this file are the vocabulary for everything else.
  2. src/ztyping.z -- the ZTyping container makes the component-table architecture concrete.
  3. src/zenv.z -- the scope stack and the lock-conflict rule. The semantics of "what makes two locks conflict" lives here, not in the typechecker.
  4. src/ztypecheck.z -- enter via the Code Map above. A good first dive is typecheck → resolveDef → resolveFunction → checkFunctionBody → checkStatement → checkExpression → checkCall. That path touches almost every cross- cutting concern in turn.
