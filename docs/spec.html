1. Introduction

This is a description of the Zero programming language.

2. Lexical structure

Program source code is parsed from file-like objects containing the program text.

Program text is comprised of characters that are combined to form tokens.

Each token is one of: whitespace, newline, eof, comment, delimiter, separator, identifier, label, keyword, reserved word, or string literal.

2.1. Character set

Source code is UTF8 encoded unicode text. Arbitrary non-null unicode characters are allowed in comments and string literals. The remaining elements of the language (identifiers, keywords, delimiters etc.) are limited to a subset of ASCII.

Some ASCII characters are reserved and are a syntax error outside of comments or string literals.

All non-printable characters (ie. that have a character code less than Space (32)) are reserved EXCEPT:

  • Horizontal tab (9)
  • Line feed (10)
  • Carriage return (13)

All characters with a code of Delete (127) or greater are reserved.

Other reserved characters:

  • comma , (44)
  • square brackets [ (91) ] (93)
  • backtick ` (96)

2.2. Whitespace and Newlines

Outside of string literals, the only valid whitespace characters are space, tab, linefeed and carriage return. Linefeed only denotes a new line.

When directly within a primary expression block, linefeeds are statement line separators and are explicitly part of the grammar. In all other locations, linefeeds are treated as whitespace only.

Whitespace can serve as a delimiter between identifiers that would otherwise be considered a single token.

2.3. Comments

Comments begin with a hash # (35) and extend up to (but not including) the following linefeed or else to the end of the file. They may contain any UTF8 text other than NUL (0).

2.4. End of file

A token marking the end of the file. All top level units are contained within a file-like object that ends when the parser returns an end of file token.

2.5. Delimiters

The following are the delimiter characters:

TokensDescription
( )Precedence delimiters
{ }Block delimiters

Parentheses are used for enforcing or clarifying precedence. They surround a single expression or type reference. They are not required for function calls. Linefeeds are ignored (considered whitespace only) when immediately within parentheses.

2.6. Separators

The semicolon (;) is a statement separator along with newline -- when in primary expression blocks.

2.7. Identifiers

Identifiers are used to name constants, variables, functions/methods (including operators) types and numeric literals.

Identifiers consist of one or more letters, digits, special-characters and optionally a single dot in the case of numeric identifiers.

The special-characters are:

! $ % & ' * + - / < = > ? @ \ ^ _ | ~

Numeric identifiers optionally start with + or - then a decimal digit. The + or - is part of the literal and must immediately precede the remainder of the literal without intervening whitespace.

Numeric identifiers may contain a single decimal point (.) which must be followed by a decimal digit in all cases. They then continue up to the word boundary (whitespace, newline, delimiter, separator). Numeric identifiers may not have leading or trailing decimal points. Always use leading or trailing 0s if required to prevent this.

Numeric identifiers may include underscore characters for readability purposes in all but the first character and immediately after the decimal point. The underscores are ignored.

Identifiers are case sensitive.

Identifiers must be separated from surrounding tokens with whitespace or the delimiters or separators. Specifically note that mathematical operators are standard identifiers and so must also be surrounded by whitespace.

Exceptions and special cases:

  • some identifiers are predefined
  • some otherwise valid identifiers have been designated keywords
  • some otherwise valid identifiers have been designated reserved words

2.7.1. Predefined Identifiers

2.7.2. Integer Identifiers

Integer literals follow the rules above for numeric identifiers.

Decimal integer literals default to 64 bits wide and signed. Non-decimal default to 64 bits wide, unsigned.

If an optional sign is provided (+ or -) or there is no base prefix specified, the numeric identifier is decimal. If a sign is not provided, a lower case numeric base prefix can be specified.

Leading zeros are not significant. Specifically, they do NOT denote octal notation.

Character literals are also an integer type. Their underlying storage is equivalent to u8 (for c8; ASCII) or u32 (for c32; Unicode).

2.7.4. Numeric defaults

Bare numeric literals follow the default-type rule when their surrounding context does not pin a target type:

Lexical formDefault type
decimal integer (100, -200)i64
non-decimal integer (0xff, 0b101)u64
literal with . or e (3.14, 1e5)f64

A bare literal in a context with no target type — for example the RHS of a label binding (x: 100) — resolves to its default. Explicit dotted-method suffixes (100.u8, 3.14.f32) override the default and pin the type directly.

2.7.5. Numeric literal coercion at typed locations

When a numeric literal — or a compile-time constant expression built from numeric literals — appears in a position whose surrounding declaration provides a target numeric type, the literal is coerced losslessly to that type. No explicit dotted-method suffix is required.

Coercion sites currently include:

SiteExample
call argumentf val: 100 (parameter val: u32)
function implicit returnf: function out u32 is { 100 }
reassignment RHSa: 10.u8 ; a = 200 + 100 - 250
match-arm numeric patternmatch v case 100 then ... (subject u8)
record/class field initialiserFoo count: 100 (field count: u8)
data-block elementbytes: data { 1.u8 2 3 } (subsequent elements coerce to first)

Bare-name construction (Foo count: 100) is equivalent to a .create call (Foo.create count: 100) — both routes type-check arguments against the corresponding field's declared type.

The rules for "lossless" are:

  • integer literal → integer target: the value must lie in the target's range (e.g. u8 requires 0..255). Constant arithmetic on integer literals folds at unbounded precision, and the final value — not any intermediate — is range-checked at the boundary. So 200 + 100 - 250 coerces cleanly into u8 (folded value 50), even though the intermediate 300 would not fit on its own. The intermediate is allowed to exceed even u64 — e.g. 0xffffffffffffffff + 1 - 1 coerces cleanly into a u64 parameter, because the intermediate 2^64 is just a transient value in the constant folder, never a stored runtime value.
  • integer literal → float target: the value must be exactly representable in the float's mantissa. 16777216 (2^24) fits f32; 16777217 does not.
  • float literal → integer target: rejected. Float literals coerce only to float types. Use .i32 / .i64 / etc. explicitly when an integer conversion is intended.
  • float literal → float target: the value must round-trip exactly through the target's precision (e.g. 0.1 does not fit f32).

A literal that fails any of these checks produces a literal-aware diagnostic at compile time naming the value and the target's range — no runtime overflow.

The inline-suffix form (100u8 with no dot) is not accepted; use the dotted form (100.u8) instead.

Examples:

push_byte: function {b: u8} is { }
small_value: function out u8 is { 200 + 100 - 250 }  # returns 50

main: function is {
    push_byte b: 100              # 100 coerces to u8
    push_byte b: 50 + 25 + 25     # folded to 100, fits u8
    push_byte b: 100.u8           # explicit suffix still works

    a: 10.u8                       # a is u8
    a = 100                        # bare 100 coerces to a's u8
}

2.8. Labels

Labels are used when assigning names to definitions. They are identifiers followed by a colon (:). The name assigned is the identifier only, without the colon.

Labels are case sensitive.

A shortened syntax is available where the definition refers to an item in scope that has the same name -- a single colon precedes the identifier. For example: :id is (somewhat) equivalent to id: id. The lookup of the value specifically excludes mapping to itself but otherwise searches the current environment hierachically as usual.

The colon must immediately precede the identifier without intervening whitespace.

2.10. Reserved Words

Reserved words are syntax errors. The following are reserved:

macro           flag            view
goto            cell            unsafe
repeat          pragma          switch
until           enum

Neither keywords nor reserved words may be used as identifiers -- this includes binding names, parameter names, and type names. The rule keeps the grammar context-free: a single token always carries the same meaning regardless of position.

When a desired name collides with a keyword or reserved word, the established rename pattern in the codebase is to pick a role-flavoured noun -- result, buf, output, outval and similar -- in place of the conflicting word.

2.11. String Literals

A string literal can be raw or interpreted. Both may have embedded newlines and have the same rules for handling the first and last newline and for stripping leading whitespace as detailed below.

2.11.1. Interpreted String Literals

An interpreted string literal is enclosed by double quotes ("). Within the quotes are zero or more literal UTF8 characters or escape sequences. A string may not include NUL (0) characters.

Valid escape sequences are:

\uhhhhhhunicode codepoint, emitted as UTF-8 bytes

2.11.2. Raw String Literals

A raw string literal is free form text included without interpretation, except for special handling of the first and last line and leading whitespace.

A raw string starts with three or more double quotes and ends with the same number of double quotes. Within the quotes are one or more literal UTF8 characters. A string may not include NUL (0) characters.

An empty string cannot be specified with a raw string literal because the number of opening double quotes cannot be determined. Use "" to represent an empty string. Likewise, a single line raw string cannot start with a double quote.

2.11.3. String Newline and Whitespace Handling

Both string types (interpreted and raw) have similar handling of newlines and leading whitespace.

Literal newlines can be included, they are always encoded as LF (\x0a) regardless of the line ending used in the program source.

If the first line is blank after the opening delimiter (whitespace only), then that text including the first newline is excluded from the string literal.

If the last line is blank up to the closing delimiter (whitespace only), then that final newline and the text up to the closing delimiter is excluded from the string literal.

Also, any whitespace on a newline to the left of the closing delimiter, will be stripped from every line in this string literal where possible. The whitespace must match exactly (number and combination of tabs and spaces) to be stripped.

2.11.4. String Literal Types

A string literal types as StringView -- a view over the static program-memory bytes that hold the literal. This is true for both interpreted and raw forms; the runtime stores the literal once in the program image, and bindings, arguments, and returns referencing the literal share that storage with no allocation.

An interpolated interpreted string is the one exception: because the runtime must allocate to splice the interpolated values into the surrounding text, the resulting literal types as String.

Use .string to copy a literal into an owned String -- this is the conventional convert when passing literals to function parameters or storing them into class/record fields declared as String.

A conversion is legal only when it changes the type: String.stringview projects a view and StringView.string makes an owned copy, but the same-type forms are compile errors. .string on a value that is already a String (an interpolated literal, for example) would hide a fresh copy -- use .copy for an owned duplicate, or drop the suffix where an owned value is already at hand. .stringview on a value that is already a StringView is an identity no-op -- drop it.

3. Definitions

A program is comprised of one or more units. Units are made up of zero of more definitions.

The entry point for a standalone program is the function called main in the initial unit. The name of initial unit is provided to the compiler as the compilation target and all referenced units are determined by tracing out references used in that initial unit.

Definitions are one of:

  • function
  • objects: unit, record, class, variant, union, facet, protocol
  • constant if definition
  • data
  • typeref (aliases?)

The descriptions of the syntax for each of these are described in their respective sections, but at a high level, they are built from keywords and blocks.

3.1. Blocks

The larger components of the language syntax are built from blocks.

Blocks are used wherever a series of optionally named items is required, eg. declarations, function arguments, function body, function calls, record/class/list/union members etc.

Blocks are delimitered by the beginning and end of file for a unit file and by matched braces {} in all other cases.

The general format of a block body is a list of items each being: a 'key' (an identifier with a colon or a keyword) then a 'value'. In some contexts, the key or the value may be omitted.

A shortened form of key and value is permitted when a variable of the same name as the required key holds the value. In this case, the identifier with a leading colon can be used to represent both the key and the value.

# Given:
arg: 0

# then:
myfunc :arg

# is equivalent to:
myfunc arg: arg

Each item in the block body ends with the beginning of the next item (the next key) or by a delimiter or a newline or EOF.

Blocks of code also define a unit of scope.

The 'value' depends on context but is one of:

  • type declaration or type reference (Declarations and References)
  • an expression
  • for statement lines in function bodies, can be expression, assignment, swap, 'continue', 'break', 'return'

Summary of uses:

Special cases:

For calls and call-like syntax (ie. type references with generic parameters) the key for the first argument is optional (see Call Arguments and Declarations and References). All other arguments must be supplied by name.

For data literals, the optional key can be an integer. The expression specified will be the value at that index.

3.2. if definition

An if may be used as a definition at the unit level. The syntax is similar to an if expression, but there are different contraints. Each clause of an if definition must evaluate to a constant. Each then arm of an if definition is permitted to produce a constant item of a completely different type and the result of the definition will be the exact type of the first arm whose predicate evaluates to true.

4.2. reassignment

Reassignment replaces the value bound to an existing name or field. For valtype targets the RHS is copied into the slot. For reftype targets the existing value is destroyed first, the RHS is moved into the slot, and the RHS source name is invalidated — use-after-move on the source name is a compile-time error. This "drop-and-transfer" rule keeps both sides free of aliasing while guaranteeing that no storage is lost to a leak and no value is freed twice.

: Holder: class { msg: string }
: h: Holder msg: "initial".string
: fresh: "updated".string
: h.msg = fresh       # destroys the old msg, moves fresh in
: # using `fresh` here is an error.

Reassigning a reftype field whose source is borrowed is an ownership error — the borrow binding must stay live for its full scope.

Use swap when both sides need to remain initialised and addressable: no move occurs, the two slots exchange contents and both stay valid.

4.4. Calls

A call executes a function by passing zero or more arguments and usually returning a value or owned reference.

They are one of the following forms:

  • the function name followed by its arguments (function call syntax)
  • the receiver object followed by a dot then the function name followed by its arguments (method call syntax)
  • the receiver object followed by the method name followed by it's single unnamed argument (operator syntax).

Calls may only be used in function bodies. Type specifiers may only be used where types are required - argument definitions, components of classes, records, units etc.

Note that parentheses are not used for function calls but as the call itself is an expression, the entire call may be surrounded by parentheses for grouping or precedence purposes.

A function call without arguments is still a function call (not a function pointer). A bare function name used as a statement or as a value is therefore a call, evaluated for its effect or result. Use .take on the function name to retrieve a function reference instead.

Named arguments can be specified in any order. Arguments are evaluated left to right as specified in the call.

Function parameters that have default values may be omitted in the call and will use those defaults in the body of the call. If the caller does specify that argument, it must be of the same type as the default value.

Generic arguments must be specified in a call or must be able to be derived from another argument declared to be of that generic type. They must be a member of the union or implement the protocol specified as the 'type' in the parameter definition.

Parameters declared with .take on their type take ownership of the supplied reftype argument in the function body. The variable supplied to a .take parameter is not valid after the call returns. Ownership transfer is performed implicitly; the caller does not need to call .take explicitly (though it may do so for clarity).

Parameters with no ownership annotation are borrowed by default (equivalent to writing .borrow explicitly). For valtypes the borrow is a copy; for reftypes (string, file, bufreader, user classes) it is a pointer-pass at the C ABI — mutations through the parameter are visible to the caller, and the caller retains ownership at scope exit. The implicit-take logic at call sites correctly skips these. Native functions (is native body) follow the same convention: each native is annotated based on what its body does with the input — read-only natives take StringView (no allocation, no ownership question), and natives that store the input directly into a receiver use .take (no per-call deep-copy).

Returning a borrowed parameter as owned (i.e. a function declared out T returning a .borrow-bound or default-borrow parameter) aliases the caller's heap data and is rejected at type-check time. Storing such a borrowed parameter into an aggregate field (Box label: borrowed_s) is rejected for the same reason. Remediations are .copy (clone), .take on the parameter (transfer ownership in), or out T.borrow + .lock-annotated parameter (return a borrow tied to a locked source). See ownership.pdoc for details.

In a call, the 'primary' parameter is the first parameter listed in the in parameter list of the function definition. In a type specifier, the 'primary' is the first generic argument. If that is also the first argument supplied in a call or type specifier, then the parameter name may be omitted in some cases. When either:

  • it is a normal argument (non-take and non-generic)
  • it is a .take parameter and the supplied argument is anonymous (i.e. the result of a function call)
  • it is a generic parameter but a concrete arg is being provided (i.e. not a generic arg)

In all other cases, the parameter name must be supplied in the call or type specifier argument. e.g:

  • all arguments after the first argument, of any type
  • all .take arguments taking their value from a named variable

The argument name rules above are designed to:

  • unambiguously document the arguments to a call or type specifier
  • allow arguments in any order at the call site
  • clearly show when local variable ownership is being given away to a call (by requiring the parameter name to be specified for .take parameters)
  • not overburden calls with names for the 'primary' (or only) argument. The 'primary' argument is often redundant following the name of the function/type

Argument evaluations are also subject to ownership locking. See Ownership.

The expression for an argument value is first evaluated in the current scope. If the value is a single identifier (undotted) and it does not resolve up to the current module level, it is assumed to be within the scope of the argument type. This allows enum arguments to be specified without qualifier (as long as there is no local definition of that same identifier).

Basic function call (no argument):

Operator syntax. Names cannot be specified. Must always be a binary operation (the receiver and one argument). The methodname would tend to usually be one of the operator names.

    calc a: 1 b: 2 c: 3     # -> 6
}

4.6. Flow Control Expressions

There are four flow control related constructs/expressions: with for a sequence within a new scope, if for conditional executtion, match for conditional dissection of variables and for for iteration.

4.6.1. with

Execute a single expression in a new scope with a single new binding. with can be used as an expression: its value is the value of the do expression.

Arguments:

ArgumentDefaultValue/Description

Ownership: the binding follows function-argument rules. A plain name or dotted path on the right-hand side is borrowed — the source is EXCLUSIVE-locked for the do scope; the source remains valid after the do ends. An inline .take moves ownership into the binding and invalidates the source. A call or constructor on the right-hand side produces a fresh value that the binding owns. .release is not a valid right-hand side; use .take instead.

with a: c do print a

# explicit take: c is invalidated
with a: c.take do print a

# dotted path borrows; the root is locked for the body
with v: p.first do print v

# call right-hand side: the with binding owns the returned value.
# The call (bag.iterate b: b) is the single initialization expression
# for the one binding 'iter' -- the named argument 'b:' belongs to
# the call, not to with itself.
with iter: (bag.iterate b: b) do for x: iter loop { print x }

When the RHS is a plain path reference the compiler may emit the binding as an alias for the source -- no stack slot is allocated. See the "Binding Aliases" section in docs/ownership.pdoc for the exact eligibility rules.

4.6.2. do

Execute a sequence of expressions in a new scope.

do can be used as an expression. The result will be the value of the clause (the result of the last expression in the clause).

break can be used for early exit of the do block. When a do block contains break, the expression type becomes optional (optionval(T) for value types, Option(T) for reference types). Normal completion produces some(value), while break produces none.

break and continue are scope-based: each for loop introduces both into its scope; each do block introduces only break. A break always binds to the innermost enclosing scope that defines it. Outside of a loop or do block, break and continue are undefined.

x: {
    if should_skip then { break }
    compute_value
}
# x is optionval — match to extract
match (x) case some then { use_value } case none then { handle_skip }
ArgumentDefaultValue/Description
isYesBlock of expressions

4.6.3. if

if expressions allow conditional execution. Arguments:

ArgumentDefaultDescription
whenYesCondition expression that results in a boolean
thenNoBlock of code
elseNoBlock of code

An if expression has one or more clauses. Each clause is made up of one or more conditions (when) followed by a then block that will be executed if all prior conditions succeed.

Each when condition must be an operation that results in a boolean. when is the default argument and the when keyword may be omitted for the first condition specified immediately after the if.

The conditions are evaluated in turn. If all conditions before a given then block return true, the then block will execute. If any fail, the tests for the remainder of the block stop (short-circuit) and the process continues after that related then block.

If no blocks of conditions completely succeed, the else block will execute, if present.

A successful clause will exit the if expression -- at most, one then or else clause will execute.

When if is used as an expression (e.g. on the right side of an assignment), the if must be exhaustive (must have an else clause) and each arm must evaluate to the same type or must not complete (return, break or continue). The result of the block is the result of the last expression of that block. Variable introduction uses with (see with).

When if is used as a statement — its value is discarded (it is not on the right of an assignment, a return value, or another value position) — the arms need not evaluate to a common type; the arm-type unification above applies only to the expression form.

if may only have one else argument and it must be the last argument.

# statement if
if x == 1 then {
    # will match if 'true'
}

# if-as-expression: result is the value of the executed branch
max: if a > b then a else b

# multi-line branches: result is the last expression
result: if x < 0 then {
    y: 0 - x
    y * 2
} else {
    x
}

# branches ending in return/break/continue are exempt from type matching
x: if n < 0 then { return 0 } else n

If .take is called on a variable with a destructor inside any arm of an if, the compiler ensures the variable is consumed in all arms. Arms where the programmer does not take the variable have it auto-destroyed after the block. The variable is invalidated after the if and cannot be used in subsequent code. If no arm takes the variable, it remains valid.

conn: connection.open "db"
if ready then {
    process from: conn.take     # takes ownership
} else {
    log "not ready"             # conn auto-destroyed after if
}
# conn is invalid here — compile error if accessed

4.6.4. match

A match is used to dissect a union, variant, enum or generic parameter type into its child types. See also type narrowing.

Parameters:

elseNoStatement or block of code (executed if no previous matches)

The case argument specifies the specific child type being tested. case is the default argument and the argument name can be omitted if it is the first argument.

An assignment expression may be specified. This will then become the target variable and all case clauses will specify subitems (ids) relative to this variable. The expression must evaluate to a union/variant/enum instance or must refer to a generic parameter. If the expression (RHS) is a named variable, that name will be unavailable in the match body to prevent aliasing or locking issues. The prepended colon : syntax may be used to nominate an existing variable name as a target within the match. The label in the assignment expression is within scope only within the match expression. If a labelled expression is provided, it must be the first argument to match. There can only be at most one labelled expression in a match.

If an assignment expression is specified, the match must be exhaustive.

A then clause must be specified after each group of case clauses. This clause will execute if the preceding case clauses match the subtype of the target variable. Within the then clause, the type of the target variable will be narrowed to the type(s) matched in the related of. If the matched type is a union/variant that has no type associated with it (null) then the narrowed type is 'null' and it is forbidden to use the target variable.

A then clause that terminates (with a return, break, continue) will narrow the target variable type (if a named variable) for the remainder of the scope after the match completes ie. that terminated subtype is excluded from the type definition of that variable going forward.

An else clause can be specified and will be executed if no previous case clause matches. It must be the final clause if present.

Only one successful then or else clause will execute.

match does not lock its subject variable. Unlike for loops (where the iterator is re-evaluated each iteration and mutation would be dangerous), match arms are mutually exclusive and the subject is evaluated once. This means .take can be used on the match subject inside arm bodies.

If .take is used on the subject in any arm, the subject variable is invalidated after the match and cannot be used in subsequent code. Arms that do not explicitly take ownership will have the subject automatically destroyed after the match completes (in C: a NULL check and conditional destroy).

}
# u is invalid here — compile error if accessed

When used as an expression, all then and else results must then be of the same type. When a match is used as a statement — its value is discarded — the arms need not share a type.

When the match subject is a compile-time constant (a named constant, numeric literal, or numeric generic parameter value), the match is evaluated statically. 4.6.5. for

for expressions allow iteration or repeated execution. It can be used to handle all looping variations:

  • for/while/loop - boolean pre-condition
  • for/loop/while - boolean post-condition
  • for loop - loop forever
  • 'foreach' style loop against an iterator, an iterable, or a function/method returning an option-shaped wrapper

Arguments:

ArgumentDefaultValue/Description
whileYesBoolean expression. Pre or post loop condition
[var]:No'var' is bound to the some-payload of the iterator's per-iteration value
loopNoBlock of code that may be executed 0 or more times

Pre-conditions can be boolean expressions specified by the while keyword. This is the default argument and the argument name can be omitted if it is the first argument.

A single binding ([var]: operation) can be specified before the loop. The binding RHS may be any of: an instance of an iterator type, an instance of an iterable type, a constructor call producing an iterator, or a function/method call producing an option-shaped wrapper. The compiler picks an evaluation strategy from the RHS shape (see For-loop dispatch below). On each iteration a value of type Option(t) / optionval(t) / OptionView(t) is produced; the loop terminates when the value is none, and binds 'var' to the unwrapped some-payload otherwise.

Inside the loop body, 'var' is the some-payload value (narrowed from the wrapper). It is not the wrapper — no match arm is needed to unwrap it. The ownership of 'var' depends on the wrapper kind:

  • Option(t).some - 'var' is the owned reftype t
  • optionval(t).some - 'var' is the owned valtype t (copied)
  • OptionView(t).some - 'var' is a borrowed view of t, lifetime-tied to the iterator's lock on its source. Storing 'var' to an aggregate field or returning it from the loop is a compile error; use .copy first.

A single loop clause may then follow. The body of this clause will be executed if all conditions succeed. The loop may contain a continue (early skip to next iteration) or break (terminate for clause). Or a return.

Postconditions can be boolean expressions specified by the while keyword after the loop. They must succeed for the for to continue on a new loop.

Any combination of pre and post conditions and bindings may be specified on a single for.

Order matters, conditions/bindings can short-circuit on failure and will stop processing subsequent tests.

for requires at least one argument (condition or loop). for <condition> will loop until condition is false; for x: operation will loop until operation produces none (discarding x each loop); for loop { ... } will loop unconditionally and indefinitely.

for can be used as an expression. The result will be a list of the final values of the loop statement.

# normal while
# (precondition)
for x == y loop {
    # ...
}

# normal for..while
# (post condition; at least one iteration guaranteed)
for loop {
    # ...
} while x == y

# iterator-as-callable — MyIter is an instance of a class whose
# .call returns option/optionval/optionview
for x: MyIter loop {
    # x is the some-payload (narrowed)
    # MyIter is locked until 'for' completes
}

# iterable — i64 has an .iterate method returning an integer-range
# iterator. The lean form `for x: 5 loop` is sugar for
# `for x: 5.iterate loop`.
for x: 5 loop {
    # x = 0, 1, 2, 3, 4
}

# constructor call returning an iterator instance (eager)
for line: (TextReader file: f) loop {
    # ...
}

# function reference called per iteration (deferred)
for n: my_supplier loop {
    # my_supplier is invoked before each iteration; loop ends
    # when the result is none
}

# 'foreach' combined with a precondition
cont: true
for x: it while cont loop {
    # multiple conditions all must hold; short-circuit at first false
    cont = false
}

The grammar form for x: in iter (with the in keyword) is reserved and not currently accepted by the parser. The plain binding form for x: iter is canonical.

For-loop dispatch

The compiler classifies the binding RHS by AST shape and result type and picks one of five dispatch modes. Rules are checked in order; first match wins. Let R be the type produced by evaluating the RHS, and call a type "iterator-shaped" if it has a .call method whose only required parameter is :this and whose return type is Option(t), optionval(t), or OptionView(t) for some t.

  1. Rule 1 (call returning iterator): RHS is a call expression AND R is iterator-shaped. The compiler invokes the call once eagerly before the loop and uses R as the iterator; R.call is invoked per iteration. Example: for line: (TextReader File: f) loop ....
  2. Rule 2 (iterable): RHS is a non-call expression AND its type T has an .iterate method whose only required parameter is :this and whose return is iterator-shaped. The compiler synthesises the call to .iterate once before the loop; its result is the iterator. Example: for x: 5 loop lowers to for x: 5.iterate loop. Only the exact name .iterate triggers this rule; other iterator-producing methods such as .iterate_keys or .iterateItems must be invoked explicitly so they fall under rule 1.
  3. Rule 3 (function as iterator): RHS is a call expression AND its return type is Option(t) / optionval(t) / OptionView(t) directly. The compiler defers the call: it is re-invoked once per iteration and the result is the per-iteration value.
  4. Rule 4 (instance with call): RHS is a non-call expression AND its type T is itself iterator-shaped. T is the iterator; T.call is invoked per iteration. This is the typical shape for iterator instances bound by a prior with.
  5. Rule 5 (none of the above): compile error. The diagnostic lists the three accepted shapes and suggests rewriting to a manual for loop { ... break ... } loop when none fit.

The compiler does not auto-synthesise a trivial iterate: function {:this} is { return :this } on iterator types; authors who want their type to be both iterator and iterable must write both .iterate and .call explicitly.

The for-loop expression form (a list of the loop body's final values) is available under all five dispatch rules.

4.7. .take on Function Names

To obtain a function reference value (rather than calling the function), use the .take dotted operator on the function name. This is consistent with the existing .take semantics for value types.

myfunc: function {a: i64} to i64 is { return a }

# obtain a function reference without calling the function
ref: myfunc.take

.take on a spec name is a compiler error because specs have no value — they are types only. Use .take on a function name that matches the spec.

5. Type System

5.1. Overview

A type (or data type) is a group of values and the operations that can be performed on those values.

All function parameter definitions and object fields must have a type specified. Variables and top level definitions have types inferred from their specified values.

All object types may have methods defined on them.

There are two broad categories of types: value types and reference types.

5.1.1. Value Types (valtype)

  • pass-by-value
  • includes all fixed numeric types; and arrays
  • 'record' is the user defined value product type
  • 'variant' is the user defined value sum type
  • 'facet' is the user defined value interface type
  • records and variants can only contain other value types
  • records have constructors
  • no destructor (why?) (default destructor only)
  • no other resources that need freeing (eg. can't have a file, db handle or any reftype as a member)
  • single, contiguous block of memory of a fixed size (for all instances of this type)
  • variant is size of tag + size of largest possible member
  • mutable (but private members may be protected with accessor methods)

5.1.2. Reference Types (reftype)

  • pass-by-reference
  • eg String, File, List, Dict, protocol, union, (others?)
  • 'class' is the user defined reference product type
  • 'union' is the user defined reference sum type
  • 'protocol' is the user defined reference interface type
  • can contain value types -- the value types will be embedded
  • can contain other reftypes (reftypes will not be embedded - store reference to them only). But must have ownership.
  • classes have constructors
  • can have destructor (but it doesn't destroy memory, compiler handles that)
  • single, contiguous block of memory of a fixed size (for all instances ?? what about List?) but may have references to other reftypes
  • mutable (but private members may be protected with accessor methods)
  • a reftype variable must always contain a valid reference. The swap operator can be used to take a new reference and return the old one. The destructor can be called but then the variable is no longer valid (unless reassigned).

5.1.3. Product types

Product types are a compound type that contain one or more fields of specified types. record is the value product type. class is the reference product type.

5.1.4. Sum types

Sum types hold a single value that can be one of a fixed list of types. variant is the value sum type. union is the reference sum type.

Sum types are available at run time but can also be used to specify generic parameter constraints.

Falsy-first convention

For 2-arm sum types where one arm represents absence, failure, or falsity, list that arm first in the declaration so it receives discriminator tag 0. Stdlib types Option, optionval, Result, resultval, and bool all follow this convention.

The convention has three benefits:

  • Zero-initialised memory naturally reads as the falsy variant.
  • Compiler comparisons (if, for termination) can use tag 0 directly without per-type lookup.
  • Visual reading is consistent: the early-exit / no-value case is at the top of the declaration.

User-defined sum types are encouraged but not required to follow this convention; the compiler does not enforce it on user code.

Locked union arms

A union arm may declare its payload type with a .lock suffix, indicating that the arm holds a locked reference into a parent rather than owning its payload:

OptionView: union {
    none: null
    some: t.lock
} as { t: Any.generic }

Per-arm semantics:

  • Owned arms (the default — e.g. some: t): the constructor takes ownership of the payload; the union's destructor frees the payload.
  • Locked arms (some: t.lock): the constructor takes a locked source via the from: parameter (consistent with protocol.borrow from: x.lock); the union's destructor releases the lock without freeing the payload. The union's lifetime cannot exceed the locked source's lifetime; escape analysis prevents storing the union into an aggregate that outlives the source.

Mixed-ownership unions are permitted: arms may independently choose owned or locked semantics. The destructor walks the active arm via the tag and applies per-arm cleanup.

The extension is union-only; variant arms must remain inline-stored valtypes. A locked arm holds a pointer to external data, which inherently introduces ownership/lifetime tracking that valtypes deliberately exclude.

Construction: the locked arm is constructed by passing from: with a locked source:

with v: my_lock_source do {
    ov: OptionView.some from: v
    # ov.some is borrowed from v; ov cannot outlive this 'do' block
}

Destructor elision: if every arm of a union is either null or a locked reference, no runtime cleanup is required and the compiler emits no destructor for the union. For mixed unions, the emitted destructor's switch only contains case clauses for arms that need cleanup.

Type Narrowing

The compiler keeps track of the type of every expression at every point in the program source via a flow-typing system (TypeState). This tracking is intra-function only and does not cross function call boundaries.

Sum types can be 'narrowed' back to their specific child types by using match expressions or by assignment to a specific subtype.

Within a successful then clause of a match, the narrowed name SHADOWS the parent binding — it takes on the payload type for the duration of the arm. Field access, method dispatch, re-matching, and ownership transfer (.take) all flow through the narrowed name as if it were the payload value. Inside case ok then on a Result(filestat, IoError), s IS the filestat — s.size, s.kind, and s.mtimeSeconds resolve directly; match (s) dispatches on filestat's layout; stolen: s.take transfers ownership and the resulting alias retains the narrowed view.

Because the parent binding is shadowed, reaching back into the outer union through an explicit arm hop (s.ok, s.err) is a compile error within the arm — the shadowed view has no such member and the typechecker emits a targeted diagnostic pointing at the shadowing. Passing the narrowed name to a function expecting the outer union/variant is also a type error for the same reason.

Assignment-based narrowing is different — a variable assigned a specific subtype (x: Result.ok 42) keeps its outer type so it remains passable as the full union; the compiler tracks a narrowed_subtype sidecar that rejects accessing other arms.

Also, a match clause that exits the current scope (using a return, break or continue) will also 'narrow' a named target for the remainder of that scope -- ie. the type of that named target will exclude the subtypes where a matching of clause has resulted in exiting the current scope.

Assignment-based narrowing: when a variable of union or variant type is assigned a specific subtype value (eg. x: Result.ok 42), the compiler narrows the variable to that subtype. Reassignment resets narrowing and optionally re-narrows if the new value is also a specific subtype.

An else clause of a match sees the union/variant type with all explicit case subtypes excluded.

If all branches of an if or match exit the current scope (return, break, continue), any statements following that expression are unreachable and are flagged as a compile error.

For-loop binding narrowing: in for x: operation loop {...} where operation returns Option(t) or optionval(t), the loop variable 'x' is narrowed to the some-payload type 't' inside the body. The wrapper is never visible — terminating on none is handled by the loop machinery, and x is the unwrapped value.

Narrowing is purely a compile-time concept. The runtime representation of a variable is not changed by narrowing -- a union/variant variable retains its full tagged representation regardless of narrowing.

5.1.5. Interface types

Interface types specify a list of methods names and signatures. Items may implement an interface by supplying implementations of each those methods listed in an interface and explicitly listing the interface in an as member.

facet is the value interface type. protocol is the reference interface type.

Interface types are used for polymorphism at runtime but can also be used to specify generic parameter constraints.

5.1.6. Generic types

Classes, records, unions, variants, protocols, facets, units and functions/specs can be generic types.

All defined unions, variants, protocols, facets and numeric types have a compiler generated 'generic' method, that allows that defined object to be used as a generic constraint when used as a type in a field or parameter. Also, 'generic' is a predeclared type that takes a 'from' parameter of type 'any' that can be used to specify generic fields or parameters.

For all compound types and functions, generic parameters are declared in the as clause, not in instance fields or function parameters. Functions declare generic parameters via as { name: constraint.generic } alongside their in, out and is clauses. See Function Definition Arguments for details.

Any object (unit, function/spec, union, variant, class, object, protocol, facet) that has one or more such generic fields or parameters is a generic type.

Generic types do not exist at runtime. Top level code must define concrete types for all generic parameters (explicitly or implicitly). At that point, a specialised implementation is created, using the generic type as a template. This specialisation will be shared by other client locations that use this definition with the same type parameters.

Numeric generic parameters are parameterised by the numeric value rather than a type. They are declared using a numeric type constraint with .generic, e.g. size: u64.generic. Only integer types (i8-i64, u8-u64, c8, c32) are valid constraints for numeric generics; float types are excluded.

The call site must provide a constant integer value explicitly: (myrec size: 10). Numeric generic values cannot be inferred from other arguments. The value must be within the valid range of the constraint type.

When a numeric generic parameter is monomorphized, it becomes a runtime field in the concrete type with the constraint as its type and the value as its default. If the user has already declared a field referencing the numeric parameter, only that field is created (no duplication). If not, a field with the parameter's name is auto-synthesized.

Different numeric values produce different monomorphized types (e.g. myrec_10 vs myrec_20). Negative values use a "neg" prefix in mangled names (e.g. myrec_neg5 for value -5).

Example:

sized: record {
    val: t
} as {
    t: any.generic
    size: u64.generic
}

main: function is { a: (sized t: i64 size: 10) val: 42 }

Generic function example:

identity: function as { t: any.generic } in { val: t } out t is {
    return val
}

Note that when passing a generic argument, the type itself is passed to the generic parameter - an instance is not created; create is not automatically called as would be done for non-generic arguments. To make this optionally obvious at the call site, the generic method can be 'called' on the argument.

A generic parameter may have a default type, specified using the call form (constraint.generic default: type). The default must satisfy the constraint. When a generic parameter is not explicitly specified and cannot be inferred from concrete arguments, the default type is used. Explicit arguments and inference both take priority over defaults.

The resolution order for generic type arguments is:

  • Explicit: providing the generic argument by name with a concrete type at the call site
  • Inferred: the compiler derives the type from a concrete value supplied to a parameter/field declared as the generic type. This is the preferred syntax where possible
  • Default: the default type specified in the generic parameter declaration is used
  • Error: if none of the above resolves the type, a compilation error is emitted

If both an explicit generic argument and a concrete value argument are provided, they must be consistent. A conflict produces a compilation error.

Generic function example with inference:

identity: function as { t: Any.generic } in { val: t } out t is {
    return val
}

main: function is {
    x: identity 42          # t inferred as i64
    y: identity "hello"     # t inferred as string
    z: identity t: i64 val: 99   # explicit t=i64
}

Generic parameter with default type:

myfn: function as { t: (Any.generic default: i64) } in { val: t } out t is {
    return val
}

Note that attempting to store a reference type into a value type via a generic type will fail to compile.

The Any type is a compiler defined union of all possible types declared in the current program. Any can ONLY be used for generic definitions, not for creating fields or parameters.

Examples:

# Option type for reference types (nullable pointer)
Option: union {
    none: null
    some: t
} as {
    t: Any.reftype
}

# optionval type for value types (stack-allocated tagged variant)
optionval: variant {
    none: null
    some: t
} as {
    t: Any.valtype
}

# usage:

# Option wraps reference types (classes, unions, strings, lists, maps)
x0: Option.some "hello"
x1: Option.none String

# optionval wraps value types (records, integers, variants)
# the generic field type (t) is inferred from the concrete value supplied
x2: optionval.some 100
# same as:
x3: optionval.some t: i64 from: 100

# an optionval.none type. Note that the generic parameter cannot be inferred
# because 'none' does not take a concrete argument with a specific type,
# so must be specified
# none takes only a single generic argument so the argument name can be
# omitted
x4: optionval.none i32

# OptionView type for borrowed views (used by container iterators).
# Built on top of locked union arms — see "Locked union arms" above.
OptionView: union {
    none: null
    some: t.lock
} as {
    t: Any.generic
}

# OptionView.some takes a locked source via from:, and binds a
# borrowed view. The view's lifetime is tied to the source's lock;
# the OptionView cannot be stored in an aggregate that outlives
# the source.
with v: my_lock_source do {
    v0: OptionView.some from: v
    v1: OptionView.none i64
}
Heterogeneous union constraints

A union used as a generic constraint via .generic may contain a mix of concrete type members and protocol/facet members. A concrete member accepts an argument whose type is exactly that concrete type; a protocol/facet member accepts any argument whose type declares conformance to that protocol or facet (an entry of the form <label>: <protocol-name> or the :<name> shorthand in the type's as clause).

Match precedence is declaration order: the typechecker walks the union's members from first to last and returns at the first match. There is no special "name-match beats protocol-match" rule — each member is tested on its own terms. The lib author controls dispatch entirely through the order in which members are listed.

Compile time vs runtime — this distinction applies to Any heterogeneous union used as a generic constraint, not just the examples below:

  • Compile time (the only case generic constraints use): when T: SomeUnion.generic is being checked, T satisfies the constraint iff the above walk finds a match. This is a pure compile-time set-membership check; the generic function is monomorphized per concrete T, and there is no runtime tag or dispatch.
  • Runtime (allowed but rare): the same union may be constructed at runtime, in which case it is a tagged union — concrete members carry their value directly, protocol members carry a protocol-erased instance (vtable + opaque data) per zerolang's standard protocol-value machinery. Programs that go this route pay one indirect call per protocol-tagged read. No new runtime machinery is introduced; it is the same protocol-erasure used elsewhere.

Example — a StringLike constraint for generic-over-string functions:

# Text protocol: the converter method any "StringLike" conformer provides
Text: protocol {
    stringview: function {:this} out StringView
}

# heterogeneous union: one concrete type + one protocol
StringLike: union { :stringview :Text }

# string and str both declare `:Text` in their `as` clause, so both
# satisfy `StringLike` via the protocol member. stringview satisfies
# via the concrete member.

print: function as { T: StringLike.generic } in { msg: T } is native

main: function is {
    print "hi"                     # T = stringview
    print "hi".string              # T = string (conforms via :Text)
    print "hi".str to: 8           # T = str_8  (conforms via :Text)
}

Note: Option is emitted as a nullable pointer in C. Option.some val stores the reference directly; Option.none is NULL. This means Option(Option(T)) has a known ambiguity: the outer none (NULL) is indistinguishable from the inner none at runtime. Nesting options is legal but inadvisable; wrap in a class if needed.

5.1.7. box

The Box type converts any type into a reference type. It has a generic parameter t of type Any.generic and a from: argument which takes ownership.

When the from: argument is a reference type, box is a zero-cost passthrough: the variable becomes the reference type directly with no runtime overhead.

When the from: argument is a value type, box heap-allocates the value. The resulting variable is a reference type (pointer). Access to the boxed value is transparent: operators and fields work as if the value were unboxed. Destruction of a boxed value type frees the allocated memory.

Example: b: Box from: 42 heap-allocates the integer; r: b + 8 uses transparent arithmetic. s: Box from: "hello" is a zero-cost passthrough.

    aa: Any.generic

5.1.8. Visibility

Definitions within blocks defining: facets, protocols, variants, unions, records, classes and units; may each belong to 0 or more interfaces (protocols/facets).

There are predeclared units called 'public' and 'private' for each but new definitions of interfaces by any other name can be created.

The default definitions for both 'public' and 'private' include all members . But the 'public' name can be redefined within the block body to create a unit that include a different subset of methods and fields. It is not possible to redefine 'private'.

The methods/fields accessible from outside the 'item' or 'unit' (static) namespace are those included in 'public'. By default, 'public' will also list the methods/fields that are available on item instances. But instances can be modified to present a different public interface by specifying 'this.myinterface' as the return type in all constructors of the instance. 'myinterface' will then be the public protocol/facet available on instances. Note that 'units' do not have instances so their public interface is always 'public'; it would not be useful to define an interface with a different name on a 'unit' (other than for inclusion in 'public').

'private' is the interface presented on 'this' when inside the definition of an 'item' or 'unit'.

The 'private' interface can also be explicitly referred to in 'friend' functions/'items'. A function/method accepting 'myitem.private' will be able to access the private interface of a 'myitem' instance passed into it. But note that only a method defined in 'myitem' itself or another friend of 'myitem' can pass the private interface of 'myitem' to this friend function/method.

Types must be public to the caller to be directly instantiated. But if an object is acquired (eg. via another public function in that unit), the public fields and methods of that instance are accessible.

5.1.9. Ownership

Ownership is determined by how the variable is created/declared or received. Each instance has exactly one owner at most. When the owning variable falls out of scope, the object is destroyed. Top level objects (ie. data) never fall out of scope.

See docs/ownership.pdoc for the full ownership and locking model.

Ownership states:

  • Owned - the instance is owned at this location. All instances have exactly one owner. Reftype constructors return owned instances. For valtypes, every copy is a new owned instance.
  • Borrowed - the instance at this location has been borrowed. All function parameters borrow by default (borrow is the default for all in parameters). For reftypes, borrowing passes a reference. For valtypes, borrowing passes a copy (the copy is treated as borrowed by the compiler for uniform ownership tracking).

this within object methods is always borrowed (you cannot own yourself).

Instance fields are always borrowed even if the parent is owned. Valtype fields can be reassigned freely with =. Reftype fields must be set in the constructor and can only be changed later with swap -- this ensures that both locations always contain a valid instance throughout the lifetime of the parent object.

Aliasing is never permitted for reftypes (except for data). A reftype can only appear once as an argument or as a receiver in any single function call (enforced via locking).

Ownership changes only through function calls (.take parameters and return values) and the .take method on runtime instances. Normal arguments are borrowed within the called function. In match statements, the target is borrowed in the body by default; match with .take takes ownership with type narrowing. .take arguments pass ownership into the called function: the caller's name is invalid after the call.

Unnamed instances (the result of a function call not assigned to a label) are transient. They are destroyed at the end of the current statement unless assigned or passed to an owned parameter.

Ownership Type Attributes on Functions

Ownership is specified on function parameters and return types using dotted type paths:

  • type.take - for in parameters, ownership is taken from the caller. The caller must provide an owned reftype. The function must store or destroy the value before it returns. For out return types, .take is the default (function returns a new owned instance to caller) and does not need to be specified.
  • type.borrow - for in parameters, ownership is not taken. The function can call methods on the parameter but cannot store or destroy it. .borrow is the default for all in parameters (both valtypes and reftypes) and does not need to be specified. For valtypes, borrowing means a copy is passed (source not invalidated). For reftypes, borrowing means a reference is passed. Generic type parameters are exempt from type-family restrictions. For out return types, .borrow means a borrowed instance is returned. The parameter housing that instance must be marked .lock.
  • type.lock - for in parameters, this parameter will be locked by the return value for its lifetime. Multiple parameters may be marked .lock; all locks transfer to the return value. It is illegal to specify .lock on a return value. It is illegal to .lock a parameter when there is no return value. .lock on a known valtype parameter is a compile error. Generic type parameters are exempt.
Ownership Methods on Instances

.take and .borrow can also be used as compiler-generated methods on runtime instances:

  • x.take - takes ownership from the variable x. The name x is invalid after the call. Calling .take on an expression that is not used (not assigned or passed) destroys the instance.
  • x.borrow - valid on all types. For reftypes, returns a borrowed reference to x and exclusively locks x for the lifetime of the borrowed reference. For valtypes, returns a copy of x without locking (valtypes have no identity to lock).
Lock Fields on Classes

A class may declare a field with a .lock type modifier. The field stores a locked reference to external data; the reference is locked for the instance's entire lifetime. Classes are single-owner and cannot be copied, so the lock cannot be duplicated.

Iter: class {
  pos: i64
  target: container.private.lock
} as {
  create: function {target: container.private.lock} out this is {
    return meta.create pos: 0 target: target
  }
}

.lock is the only field-type modifier permitted (.take and .borrow on field types are rejected). .lock fields are only permitted on classes. A record with a .lock field is a compile error because records are copied freely, which would duplicate the lock. A class that has any .lock field cannot itself be nested as a field of another user-defined type (its lifetime must be scoped by the same block that owns the lock target).

A function that returns a class holding .lock fields must initialise those fields from .lock parameters of the current function. The compiler performs escape analysis: every .lock field of the returned instance must trace back to a .lock parameter. The locks transfer from those parameters to the caller's variable holding the result, keeping the source locked for the returned instance's lifetime.

See docs/ownership.pdoc for the full design and worked examples.

5.1.10. Locking

Locking is an orthogonal attribute tracked alongside ownership. It prevents aliasing and data races at compile time. Locking is purely a compile-time mechanism with zero runtime overhead.

Lock states:

  • unlocked - the instance is not locked (default)
  • exclusive - an exclusive lock is held over this instance
  • shared - a shared lock is held over this instance

An exclusive lock is required to pass an instance as an argument to a method. A shared lock is taken on the parent when an exclusive lock is taken on a child/grandchild member.

Accessing a reftype field (eg. x: obj.field) produces a borrowed reference to that field. The borrowed reference holds a shared lock on the parent and an exclusive lock on the field.

For a function that returns a borrowed instance: that instance retains an exclusive lock over the .lock parameter where the returned object lives, until the borrowed object falls out of scope.

Locking Algorithm

During a call, locks are taken/checked/assigned on all arguments and return values:

  1. Each argument is processed left to right as specified in the call.
  2. If the argument is a top level (data) item, no lock is taken (immutable, freely shared).
  3. Otherwise, an exclusive lock is taken on the argument.
  4. If the argument is a dotted chain, a shared lock is taken on each parent and grandparent in that chain.
  5. Additional shared locks on already shared-locked parents are permitted.
  6. If there is already an exclusive or shared lock on the argument being passed, it is an error (prevents aliasing).
  7. If the argument is a sub-call, locking is applied recursively; after that call, sub-call argument locks are cleared or assigned to the sub-call result.
  8. After all arguments are processed, the receiver is locked (exclusive lock on each component of the receiver chain).
  9. The call is made.
  10. After the call, for any .lock parameters, their exclusive locks (plus shared locks on parent items) are assigned to the returned value. All other argument locks are freed.

for loops over a reftype iteration target take an exclusive lock on the target for the body's duration, preventing mutation during iteration. Per- iteration locks on the iterator (and on any .lock fields it carries) flow through the per-iteration method dispatch.

match does not lock its subject. Match arms are mutually exclusive and the subject is evaluated once, so there is no aliasing concern. match may .take its subject, in which case the subject is owned with the narrowed type in the arm bodies.

if and with do not lock their condition or target -- expression evaluation and assignment semantics handle lock lifetimes without additional scope-level locking.

When an instance falls out of scope, all of its locks are freed. If the variable is borrowed, the borrow is dropped. If the variable is owned, it is destroyed.

5.1.11. Declarations and References

New user defined types are declared using the class, record, protocol, union, variant, function (for specs) keywords. Each user definition must be bound to a name (no anonymous or inline use). All must be defined directly within a unit.

Every local-variable, constant and each subcomponent of the above user defined types (ie. fields, methods, parameters etc) are declared using a name binding to a type reference (of the form name: typereference).

The declaration name must be a valid identifier.

The declaration typereference follows the name. It is:

  • the name of the type for standard fields and parameters (non-generic, non-default, non-taken types)
  • a constant numeric expression (for fields and parameters with default values)
  • the name of the type followed by each generic argument using syntax similar to function calls (to specify a generic type). As with function calls, the name of the first generic argument may be omitted
  • a type followed by .generic (eg. Any.generic) to create a generic type parameter. For functions and protocols, the generic parameter is declared in the body. For records, unions, classes and variants, the generic parameter is declared in the as section.
  • a type followed by .take, .borrow, or .lock in function parameter definitions to specify ownership and locking semantics. See ownership.pdoc.

5.2. Unit Types

Programs are constructed from one or more units.

A unit is used to hold logically related constant definitions. A unit must have a name and creates a new namespace.

Each definition within a unit may or may not be public. See Visibility.

A source file is a single unit. The name of that unit is the name of the file excluding the file extension. Unit names have additional restrictions over standard identifiers so that they can be represented as filenames across filesystems of differing capabilities. They must:

  • consist only of lowercase ASCII letters and digits

5.2.1. Implementation Subunits (Hidden File Units)

A file unit can have implementation subunits by placing .z files within a directory that has the same name as the parent unit. For example, a unit mymod.z can have implementation subunits in a directory mymod/.

Each subunit file in the directory becomes a private member of the parent unit. The subunit name is its filename excluding the extension. Subunits are loaded on demand: only names explicitly referenced in the parent unit are searched for and loaded from the subunit directory.

Subunits are private to the parent unit. External code accesses the parent unit's public definitions; the subunit implementations are not directly accessible from outside.

# File: mymod.z
# References 'helper' which is loaded from mymod/helper.z
compute: function {n: i64} out i64 is {
    result: helper.square n
    return Result
}
# File: mymod/helper.z
# This is a private implementation subunit of mymod
square: function {n: i64} out i64 is {
    return n * n
}
# File: main.z
# Uses mymod's public interface; cannot access helper directly
main: function is {
    print "\{mymod.compute 7}"   # prints 49
}

5.2.3. Generic Units

Units can be generic. Generic type parameters are declared in the unit's as body alongside regular definitions:

mathops: unit as {
    t: Any.generic
    add: function {a: t b: t} out t is { return a + b }
    double: function {v: t} out t is { return v + v }
}

To use a generic unit, instantiate it at the unit level (outside function bodies) with concrete type arguments. This creates a compile-time alias for the monomorphized (concrete) unit:

# unit-level definitions
intmath: (mathops t: i64)
i32math: (mathops t: i32)

main: function is {
    result: intmath.add a: 3 b: 5
    r2: i32math.double 16.i32
}

Unit instantiation is purely compile-time. No runtime objects are created. The instantiation creates a namespace alias that maps to monomorphized versions of all functions and types in the unit. Unit instantiation is not valid inside function bodies — all generic instantiation that produces compile-time aliases must occur at the unit level.

5.3. Predefined Types and Definitions

Several types or type aliases are predefined.

5.3.1. null

null is used to represent the nil type and nil value. It is not a runtime value — it cannot be assigned directly to a variable, passed as a function argument, or returned from a function. To use null at runtime, it must be encoded into a union or variant (eg. Option.none).

It can be used in some locations to indicate that an item is intentionally omitted or undefined eg. for a subitem in a union or variant that has no associated data.

It is also useful for undefining methods that would otherwise be available. Such as the default create constructor or for hiding methods in a typedef that would be exposed from the underlying type.

MyClass: class {
    # 'undefine' the default create constructor
    create: null
}

5.3.2. never

The never type is a type that is used to represent an expression that never returns. The return value of the expressions: break, continue and return are of type never within the block where they are called.

There is no never value and so the type does not exist at runtime.

The never type plays a part in type narrowing. If a never expression is executed in a then clause of a match expression, the value of the subtype of the target variable is then never from that point. The type of the target is essentially narrowed after that match to exclude the current specific type.

A never type is permitted in if or match that are used as expression values. The never type is ignored when determining the type of the entire expression.

Any statements following a never typed expression in the same scope is dead code and is raised as a compiler error during the type checking phase.

5.3.7. take

take is a predefined type attribute on all types. When used as a dotted path on a type in function parameter or return type declarations, it specifies ownership transfer semantics. See ownership.pdoc for full details. .take must be explicitly written on parameters -- it is not the default for any type family.

As a type attribute on function parameters (type.take): ownership is taken from the caller. As a return type attribute: the default -- function returns a new owned instance.

As a compiler method on instances (x.take): takes ownership from the variable, rendering the name invalid. See take (method).

5.3.8. borrow

borrow is a predefined type attribute on all types. When used as a dotted path on a type in function parameter declarations, it specifies that ownership is not taken. .borrow is the default for all in parameters. For valtypes, borrowing means a copy is passed (source not invalidated). For reftypes, borrowing means a reference is passed.

As a return type attribute (out type.borrow): the function returns a borrowed reference. At least one in parameter must be marked .lock.

As a compiler method on instances (x.borrow): valid on all types. For reftypes, returns a borrowed reference and exclusively locks the receiver. For valtypes, returns a copy without locking. See borrow (method).

5.3.9. lock

lock is a predefined type attribute on reftype types and generic type parameters, for use in function parameter declarations only (type.lock). It indicates that this parameter will be locked by the return value for the lifetime of that return value. Multiple parameters may be marked .lock; all locks transfer to the return value. It is illegal to specify .lock on a return value or when there is no return value. .lock on a known valtype parameter is a compile error -- valtypes are pass-by-value so identity-based locking is meaningless.

5.3.10. typedef

A typedef wraps an underlying type to provide a new type. The new type has no run-time overhead beyond the underlying type, it is simply a compile time construct.

To define a new typedef, define a class, record, union, or variant with a single field whose type uses the .typedef accessor on the underlying type. The typedef may also have generic parameters and methods that operate on the new type. It may satisfy protocols (with as) using a combination of methods defined on the new type and on the underlying type.

A typedef definition must have exactly one field in the is section. That field uses .typedef on the base type. Additional fields are forbidden and produce a compiler error.

A typedef must not have any user-defined constructor methods. Wrapping is handled by the compiler and type system and there is no way to intervene.

All generic parameters in the typedef MUST be used in the this definition line. That is, only generic parts of the underlying type may be referenced. Any generic parameters must be able to be automatically derived from the source item.

Any methods defined in the as section will override methods of the same name on the underlying type. All unshadowed methods of the underlying type will be exposed on the typedef. A method can be hidden by setting it to null in the as section.

The typedef may only use public methods and fields of the underlying type.

To create a reftype typedef use a class or a union. The underlying type must be a reftype.

Create a valtype typedef using a record or a variant. The underlying type must be a valtype.

The compiler generates three constructors for non-generic typedefs:

  • create and take are aliases that take ownership of a value of the base type and return an owned typedef instance. Signature: take: function {from: basetype.take} out typedeftype
  • borrow takes a lock on a value of the base type and returns a borrowed typedef instance. Signature: borrow: function {from: basetype.lock} out typedeftype.borrow

The typedef retains the underlying type and so can continue to be passed to functions that expect the underlying type as is (backward compatible). But the opposite is not true - a function expecting the new typedef must explicitly be given the wrapped type (not forward compatible).

Typedefs can be chained: a typedef of a typedef is permitted. The compatibility chain is walked, so a typedef-of-typedef is accepted where the original base type is expected.

# valtype typedef: meters wrapping i64
meters: record { val: i64.typedef } as {}

main: function is {
    # create an owned typedef instance
    m: meters.create from: 42

    # access base type methods
    x: m.val + 1

    # backward compatible: meters accepted where i64 expected
    show x
}

show: function {x: i64} is { print x }

# reftype typedef with method shadowing
MyDict: class {
    base: (dict k: str v: i64).typedef
} as {
    k: Any.generic
    v: Any.generic
}

Reassigning and swapping to a typedef is permitted. For a valtype, the same (typedef'ed) or the underlying type may be assigned to the typedef'ed valtype. For a reftype, swap may be used to swap the current typedef'ed reftype with another variable of the same or underlying type. Both variables retain their type after the swap. The underlying variable must be owned to be used in a swap, as usual.

5.3.11. tag

A predefined generic type for specifying the discriminator in unions and variants. tag is a predefined identifier, not a keyword.

The compiler automatically generates a .tag subtype on all data types and all numeric types. This .tag is used to specify the discriminator source in a union or variant's as block.

The tag specification must appear in the as block (not the is block). At most one .tag type is allowed per union/variant. The label name in the as block is convention only (commonly tag, but any name is valid).

When a data item is used as a tag source:

- the data labels must exactly match the union/variant subtype names (one-for-one) - all data values must be unique - the data values are used as the discriminator constants

When a numeric type's .tag is used (e.g. u16.tag):

- the compiler auto-generates sequential values starting from 0 - the specified numeric type is used for discriminator storage

If no tag is specified, the compiler uses u8 by default with sequential values starting from 0. If the union/variant has more than 256 subtypes, a custom tag must be specified.

tag may be null only if all of these criteria apply:

- in a union (not a variant) - there are only two subitems - one of the subitems must not have data associated with it (null member), the other must have data associated with it

In that case, no underlying discriminator is needed.

On an instance, .tag returns the current discriminator value (as the underlying numeric type). On the type itself, MyUnion.tag returns the data instance (the discriminator's constant array).

priority_values: data { LOW: 0 MEDIUM: 1 HIGH: 2 CRITICAL: 10 }

Priority: union {
    LOW: null  MEDIUM: null  HIGH: null  CRITICAL: null
} as {
    tag: priority_values.tag
}

# using a numeric type tag (auto-generate sequential values):
MyUnion: union { a: i64  b: null } as { tag: u16.tag }

5.3.17. private

A type's private member exposes friend access to that type's non-public members. Writing the field type as <type>.private in a record or function parameter declaration produces a borrowed view that can read and mutate the target's private fields.

5.3.20. this

this is a predefined identifier within an object definition (record, class etc) that can be used as a type in method signatures on that object.

this allows access to all members of the parent object both private and public. this is a narrower type than type. this is of type type, but it can only be used to refer to the current exact instance that is running the method. type can be used to refer to any instance of the current definition (ie. other objects of the same type).

this must be the return type for a user defined constructor.

this must be the first argument type (the receiver) in an object instance method. It may optionally be declared with a new name for documentation purposes or ease of use.

The name of the this parameter is not considered as part of the method signature for satisfying protocols. this can also be specified using the shortened prepended colon syntax.

# define a method of the 'Parent' object
Parent: class {
    method: function {t: this arg: i32} out i32 is {
        # ...
    }
}

# :this syntax is allowed here
Parent: class {
    method: function {:this arg: i32} out i32 is {
        # ...
    }
}

5.3.21. meta

meta is a predefined identifier that houses a namespace containing compiler metadata. The namespace holds references to compiler defined functions, details about object definitions and is context-aware (ie. can provide data about the current source location (filename, line number etc), current object etc.)

This includes meta.create which is the compiler defined constructor (the one true constuctor) for the current object.

# define a constructor for a class that calls the true constructor on 'meta'

MyObject: class {
    myconstructor: function {arg: i32 arg2: i32} out this is {
        # ...
        meta.create arg: arg arg2: arg2
    }
}

Also available is the fully qualified name for the parent object.

MyObject: class {
    printname: function is {
        # assuming this code is in unit 'myunit', prints 'myunit.MyObject'
        io.print meta.name
    }
}

5.4. Predefined Methods on Types

The compiler provides two tiers of pre-existing methods on types:

  • Intrinsics -- .take, .release, .borrow, .lock, .private. The compiler defines their semantics directly; no is native declaration in the standard library is required. They are also shadowable: a user-defined member with the same name on a type takes precedence over the intrinsic for that type. Resolution walks the parent type's children first; only when the lookup misses does the intrinsic fire. This lets a class export its own .take or .release without colliding with the language-level operation.
  • Native methods declared in lib/system/*.z with the is native form. .copy (per-type), .byteview, .listview, .stringview, .iterate, .each, .append, .length, and the collection / numeric-type methods are all in this tier. They are visible to readers of the standard library and resolve through the ordinary member-lookup path -- there is no syntactic special case for them in the type checker.

The remaining subsections describe the intrinsic tier and a small set of compiler-introspection accessors (.generic, .tag, .typedef). The native tier is covered by the per-type pages in the standard library reference.

5.4.1. take (method)

take is a compiler-generated method predefined on all variable names that refer to a reftype. It takes ownership from the variable, converting it to an anonymous owned value. The variable name is invalid from this point forward.

The take method is zero overhead -- it is a manipulation of the compile time type system only. Use of the .take method is optional when passing to .take function parameters; the compiler will implicitly take if required.

.take takes no arguments.

.take can be used for defining local variables -- the new variable takes the value and the old name becomes undefined. Calling .take as an expression that is not used (not assigned or passed) destroys the instance.

See also the ownership.pdoc.

5.4.2. borrow (method)

borrow is a compiler-generated method predefined on variable names that refer to a reftype (class, union) or a generic type parameter. It returns a borrowed reference to the receiver. The receiver is exclusively locked for the lifetime of the borrowed reference. .borrow on a known valtype (record, variant, integer, enum) is a compile error.

The borrow method is zero overhead -- it is a manipulation of the compile time type system only. Use of the .borrow method is optional when passing to function parameters (parameters are borrowed by default).

.borrow takes no arguments.

.borrow can be used for defining local variables -- the new variable borrows the value and the source name is exclusively locked while the new name is in scope.

See also the ownership.pdoc.

5.4.3. lock (method)

lock is not a runtime method on instances. It is a type attribute used only in function parameter declarations (type.lock) to indicate that the parameter will be locked by the return value. See locking for details.

5.4.4. release (method)

release is a compiler-generated method predefined on all variable names. It ends the variable's lifetime immediately, as if it fell out of scope. The name is invalidated after the call and any subsequent use is a compile error.

For owned reftypes, the destructor runs. For borrowed variables, the borrow ends and any held locks are released. For owned valtypes, the name is simply invalidated (no runtime effect).

.release is always a statement. It cannot be used as a value expression: y: x.release is a compile error. Use .take to transfer ownership to a new name.

Error conditions: cannot release a variable that is currently locked by another variable, and cannot release a top-level definition.

See also the ownership.pdoc.

5.4.5. private (method)

private is the compiler-generated friend-access intrinsic. <receiver>.private yields the receiver at the same type but with the type's private members (those whose names start with an underscore or that the type marks as private through a .private field type) visible. A user-defined .private member on the receiver type shadows the intrinsic.

Only code internal to the receiver type's methods, or code in a friend type that holds the receiver via a .private field, may apply .private to it. Use from outside is a compile error (Cannot access '<type>.private' from outside the type definition).

.private takes no arguments. It does not transfer ownership and does not introduce a new borrow on its own -- it is purely a visibility adjustment. Subsequent suffixes such as .lock / .borrow attach their usual lock semantics to the projected reference.

See examples/ListIter.z for the canonical iterator-friend pattern built on .private.

5.4.6. generic (method)

generic is a 'class method' predefined on all union, variant, protocol, facet and numeric types. It explicitly converts the receiver type to a generic argument. This makes it obvious at the call site that a generic argument is being passed and that a create method is not being called on the type. The generic method is zero overhead -- it is a manipulation of the compile time type system only. Use of the generic method is optional, the compiler will always pass a type to a generic receiver.

generic takes no arguments.

The generic method cannot be used for defining local variables, only on arguments.

See also the built-in generic type or a description of generic types in general.

5.5. data

Creates a constant list of embedded data. It is a reftype. All components must be numeric valtypes.

The definition may use a sparse representation, ie. it may have integer labels to define data at a specific offset. Items may have text labels (these are required for data that is destined to be used as a tag).

For items with a text label or no label, the index (or key) is calculated as 1 higher than the previous index, starting with 0. Unspecified elements within the data range will have a zero value. The indexes always range from 0 to the largest index specified/calculated.

Named data elements can be reference directly elsewhere in the source and this is similar to refrencing a constant.

The element type for the whole block is determined by unifying every explicit type tag — the optional out T argument plus any per-element type tag (name: 0.T). All explicit tags must agree; conflicts are a type error. When no explicit tag appears anywhere, the block defaults to i64 (or f64 if every element is a bare float literal).

A bare numeric element (cNul: 0) is treated as an untyped numeric literal at compile-time access (mydata.cNul) — it has no specific type until forced by location, then coerces to whatever the target slot expects (with range-checking). This mirrors the behaviour of top-level untyped literals. A typed element (cNul: 0.u8) carries that specific type at every access; coercion to a different numeric type is rejected (same as a typed numeric literal anywhere else).

The block's element type also determines the C-level array element type when the data block is emitted as a runtime array (variable indexing, iteration). With out u8 the array becomes static const uint8_t[]; with no out and no typed elements, it defaults to static const int64_t[].

String literal fragments are built on top of a u8 data sequence.

The data when used at runtime has exactly the same API as collections.list except that it is readonly.

# array of i64 (default — no explicit type tag)
mydata_i64: data { 1 2 0x07 }

# array of u8 via `out T` — every element is u8, no per-element tag
mydata_u8: data { 0 64 250 } out u8

# array of u8 via a per-element type tag — same result as `out u8`
mydata_u8b: data { 1.u8 2 0x07 }

# a sparsely defined array of length 5 (1, 2, 0, 3, 4)
mydata_sparse: data { 1 2 4: 4 }

# an array with text keys
mydata_key: data { FIRST: 1 SECOND: 2 THIRD: 3 }
# which can be used as a tag in a sum-type, using these values for
#  the descriminator
myvariant: variant {
    FIRST: null SECOND: null THIRD: null
} as {
    tag: mydata_key.tag
}

# bare elements are untyped literals: coerce to whatever the
# destination expects (with range-checking)
take_u8: function {b: u8} out u8 is { return b }
result: take_u8 b: mydata_key.FIRST   # FIRST is 1, coerces to u8

# can also use elements as constants in top level or in code:
One: mydata_key.FIRST   # same as..
One: 1

5.6. array

Arrays are fixed-length, stack-allocated value types that are generic over both the element type and length. Arrays of different lengths are different types, even when their element types match. Arrays are defined in the collections unit and aliased in core.

The constructor is (array of: <type> to: <size>) where type must be a valtype that can be constructed with zero arguments (numeric types are zero-initialized, records must have a default constructor), and size is the number of elements.

# create a zero-initialized array of 4 i64 values
a: (array of: i64 to: 4)

# compile-time indexed access (bounds checked at compile time)
x: a.0        # read element 0
a.2 = 42      # set element 2

# runtime indexed access (bounds checked at runtime, error on OOB)
r: a.get i: 2          # returns element value (i64)
old: a.set i: 2 val: 99   # replaces element, returns old value

# length
n: a.length   # u64 constant, equals 4

# copy from data
primes: data { 2 3 5 7 11 }
pa: primes.array   # creates array of: i64 to: 5 with data copied in

Out-of-bounds get and set calls print an error to stderr and exit the process. Since arrays are valtypes, they are copied on assignment and when passed to functions. The entire array value is passed on the stack.

5.7. str

str is a fixed-capacity, stack-allocated string valtype. It complements String (whose struct lives on the stack but whose data buffer is on the heap) with a lightweight alternative that can be used in records and variants, passed by value, and requires no ownership tracking or cleanup. str and String are separate types with no implicit coercion.

The constructor is str to: <capacity> where capacity is the maximum number of bytes. An optional from: argument accepts a String value to initialize content; if the source exceeds capacity, it is silently truncated.

# create an empty str with 32-byte capacity
s: (str to: 32)

# create from a string literal
greeting: (str to: 32) from: "hello"

# field access
n: greeting.length     # u64, current content length (5)
c: greeting.capacity   # u64, compile-time constant (32)

# convert to a string (heap-allocated data buffer)
heap: greeting.string   # returns string

# print directly
print greeting

# string interpolation
print "value: \{greeting}"

# str in records (valtype, allowed)
entry: record { name: (str to: 16) }

Since str types are valtypes, they are copied on assignment and when passed to functions. The entire value (length + data buffer) is passed on the stack. Different capacities produce different types: str to: 16 and str to: 32 are incompatible.

5.8. list

List is a growable collection reftype. It is generic over the element type and provides dynamic-length storage with amortized O(1) append. Lists follow reftype ownership semantics: they are tracked for scope cleanup and support .take for ownership transfer. The list struct itself lives on the stack; only its element data buffer is heap-allocated.

The constructor is (List of: <type>) for an empty list, or (List of: <type>) capacity: <u64> to pre-allocate storage. The C representation is a struct with length (u64), a typed data pointer to the heap buffer, and capacity (u64).

# create an empty list
nums: (List of: i64)

# create with pre-allocated capacity
big: (List of: i64) capacity: 100.u64

# append elements
nums.append from: 42
nums.append from: 99

# access by index (runtime bounds check)
first: nums.get i: 0.u64

# replace element, get old value back
old: nums.set i: 0.u64 val: 77

# pop last element
last: nums.pop

# insert at position (shifts right)
nums.insert from: 50 at: 0.u64

# extend from another list (takes ownership of source)
other: (List of: i64)
other.append from: 1
nums.extend from: other.take

# linear-scan membership test
has50: nums.contains item: 50

# stable in-place mergesort
nums.sort

# field access
n: nums.length     # u64, current element count
c: nums.capacity   # u64, current buffer capacity

Out-of-bounds get and set calls, and pop on an empty list, print an error to stderr and exit the process. Growth uses the formula newcap = oldcap + (oldcap >> 1) + 4. Different element types produce different monomorphized types: List of: i64 and List of: u64 are incompatible.

5.9. map

Map is a hash map (dictionary) reftype, generic over key type and value type. Maps use open-addressing with linear probing and automatic resizing.

The constructor is (Map key: <key_type> value: <value_type>) for an empty map, or (Map key: K value: V) capacity: <u64> to pre-allocate buckets.

# create an empty map
ages: (Map key: String value: i64)

# set entries (insert or replace)
ages.set key: "alice" value: 30
ages.set key: "bob" value: 25

# get returns optionval (value-type) or borrowed OptionView (String/reftype)
r: ages.get key: "alice"
match (r) case some then { print "found" } case none then { print "missing" }

# check if key exists
k: "alice"
print ages.has key: k    # prints 1 (true)

# remove entry, taking its owned value
removed: ages.remove key: "bob"
match (removed) case some then { print "was \{removed}" } case none then { print "absent" }

# field access
n: ages.length     # u64, current entry count
c: ages.capacity   # u64, current bucket count

Key types must be numeric types or String. The map takes ownership of both keys and values. On replacement via set, the old value is destroyed. remove key: k deletes the entry and returns its owned value wrapped in Option (for reference-type / String values) or optionval (for value-type values), or none if absent -- the value's ownership transfers to the caller and the key is destroyed. get returns a copy of the value in the same wrappers.

The table grows when the load factor exceeds 2/3. Capacity is always a power of two, starting at 8 and doubling on resize. Removed entries leave tombstones that are reclaimed during resize. Different key/value type combinations produce different monomorphized types.

Bucket dispatch hashes String / StringView keys through SipHash-1-3 with a per-process random seed (initialised once at program start) and numeric keys through the splitmix64 finalizer. Bucket iteration order therefore varies across runs — see String.hash for the user-facing native that shares the same primitive.

5.10. set

Set is a hash-table reftype, generic over element type. It shares the open-addressing + linear-probing hash table with Map, with the value column excised — a Set is a Map without per-key payload.

The constructor is (Set of: <element_type>) for an empty set, or (Set of: T) capacity: <u64> to pre-allocate buckets.

  # create an empty set
  s: (Set of: i64)

  # add elements; .add returns true on first insert, false on duplicate
  s.add item: 1
  s.add item: 2
  s.add item: 1        # returns false; caller's copy is destroyed

  # has / delete (both return bool)
  print s.has item: 1
  deleted: s.delete item: 1

  # field access
  n: s.length          # u64, current element count
  c: s.capacity        # u64, current bucket count

  # iteration via with/do — the iterator holds an EXCLUSIVE lock on the
  # source for its lifetime
  with it: s.iterate do for x: it loop {
      print "x: \\{x}"
  }

Element types must be numeric or String (hashability is enforced at monomorphization). The set takes ownership of items on add; on a duplicate-add the caller's copy is destroyed so the take-on-add contract holds. On delete the stored item is destroyed.

Iteration is in bucket order, not insertion order, and bucket order is randomised by the per-process SipHash seed (same dispatch as Map — SipHash for String, splitmix64 for numerics). The table grows when the load factor exceeds 2/3; capacity starts at 8 and doubles on resize. Tombstones from delete are reclaimed during resize.

5.12. function

The function keyword defines a:

- standard function: a signature with a body defined at unit level or within an object - a method: defined within an object and has a this receiver as the first argument - generator function: a function (or method) whose body contains yield; see Generator functions - 'spec': a signature only; no body (only at unit level)

It may also be generic by having generic parameters.

Functions are structurally equivalent. Two function types are considered equivalent when, in both definitions:

- the names and types of all non-receiver parameters are the same - the return types are the same - if there is a receiver (this), it must refer to the same object. The name of the receiver parameter is ignored for the comparison

Function definitions must be equivalent when (re)assigning Function References.

See also Specs.

5.12.1. Function Definition Arguments

Arguments to function:

ArgumentDefaultDescription
inYesBlock of standard parameters
outNoThe return type of the function
asNoGeneric type parameters for the function
isNoFunction body expression/block

- in is optional; omitted when no parameters are required. If as appears before the parameter block, in must be explicitly stated. - out is optional; omitted for procedures and iterators - as is optional; declares generic type parameters. Generic parameters must be declared here, not in in. - is is optional; omitted when defining a spec.

A function defined with no arguments is a spec that takes no arguments and returns no value. It only has side-effects.

The in argument takes a brace delimited block of parameter definitions of the form name: typereference. See Declarations and References for details. A parameter must not be private (_) but may have ownership attributes (type.take, type.borrow, type.lock) or it may have a default constant numeric value or a default function reference value. Parameters may reference generic types declared in the as clause. Declaring generic parameters directly in in (e.g. t: Any.generic) is a compile error.

The order of the parameters is significant. The first parameter may be the 'primary' argument that can be used without specifying the parameter name in a call. See Calls for details.

The as argument takes a brace delimited block of generic type parameter declarations of the form name: constraint.generic (e.g. t: Any.generic or n: u64.generic for numeric generics). The as block follows the same form as the as clause on compound type definitions. The first generic parameter in as can be specified without name at call sites; the remainder require names. This makes the instantiation interface self-documenting.

Methods (functions with a parameter of type this) must not have an as clause. They inherit generic parameters from the enclosing type definition. Static functions (functions without this, including those in a type's as block) may declare their own generic parameters and are independently monomorphized.

A function usually will return a value. The type of the returned item is specified by the argument named out. Ownership of a returned reftype value after the call implicitly lies with the caller (.take is the default for return types and does not need to be specified). A function may return a borrowed reference by specifying type.borrow as the return type; in this case, at least one parameter must be marked type.lock.

The is parameter to function specifies the body. It may be a single expression or a block of code. Each line of code is a local variable definition or a statement. All arguments supplied can be referenced by name but are read-only and cannot be reassigned. The function result can be the value of the last expression in the body or the return statement can also be called in the body for early return of a value. Iterator results are specified by using the yield statement.

# function takes an int and has no return value
myfunc0: function {a: u32} is {
    print a  # return value of call is ignored
}

# function takes an int and returns an int (implicit return)
myfunc1: function {a: u32} out u32 is {
    a
}

# function with no parameters that returns an int (implicit return)
zerofunc: function out u32 is {
    0.u32
}

# early return with explicit 'return', implicit return at end
clamp: function {n: i64 lo: i64 hi: i64} out i64 is {
    if n < lo then return lo
    if n > hi then return hi
    n
}

# if-expression as implicit return
abs: function {n: i64} out i64 is {
    if n < 0 then 0 - n else n
}

# function with no parameters and no return value
sayhello: function is {
    print "hello"
}

A function may also take ownership of arguments. To do so, the parameter type uses the .take attribute (type.take) and the type must be a reftype or generic type that may include a reftype.

The caller must own the variable before the call. The function body is free to use the taken argument as if it created it itself (i.e. can be stored, returned, destroyed). The caller's variable name is invalid after the call.

Passing a value type via a .take parameter of a generic type passes a copy as usual. This allows the same notation to be used for generic container types regardless of the type of subcomponents being stored (eg. for Lists). In all other cases, it is an error to pass valtypes to .take parameters.

.take arguments from a named variable cannot be specified without the parameter name at the call site even if they are the first argument. This is to make it obvious that ownership is given up at the call site. .take arguments from an anonymous value (eg. from the result of a direct function call) can be specified without the parameter name since the value would be inaccessible/destroyed after the call in any case.

taker: function {arg: reftype.take} is {
    # arg is owned in this scope
}

# call site:
taker arg: MyRefType
# MyRefType is unavailable in the current scope from this point

# passing a lock parameter and returning a borrow:
getter: function {t: MyClass.lock} out refchild.borrow is {
    return t.myrefchild
}

Generic function example:

# generic identity function
identity: function as { t: Any.generic } in { val: t } out t is {
    return val
}

# generic function with multiple type parameters
map: function
    as { t: Any.generic
         u: Any.generic }
    in { val: t
         f: function in { x: t } out u }
    out u
is {
    f x: val
}

# numeric generic parameter
repeat: function as { n: u64.generic } in { val: i64 } out i64 is {
    result: 0
    for n loop { Result = Result + val }
    return Result
}

5.12.2. Function Parameter Types

In the in block, the function argument types specified (after the function argument label), can be:

- a builtin type (eg. u8, i32, f64, c8) - a numeric compile time constant (this is also a type). Parameter will default to this value if not specified in the call - a named user defined type (result of record, class, union, enum, protocol definition etc.) - a spec (a function definition without an is body). A function reference of the same type must be passed at the call - address followed by a function or method name. This will then be a Spec parameter with a default value equal to the function name specified. - a type followed by .generic (eg. Any.generic, MyUnion.generic, myprotocol.generic) to specify a generic type constraint - a type followed by .take (eg. reftype.take) to take ownership of the parameter - a type followed by .borrow (eg. type.borrow) to explicitly borrow (this is the default for all parameters and does not need to be specified) - a type followed by .lock (eg. reftype.lock) to lock the parameter for the lifetime of the return value (compile error on known valtypes; permitted on generic type parameters) - a restrained type: a named type followed by a dot then a child named protocol or named method of that type. The call must pass an instance of the named type and the function declares that it will only use methods in the named protocol or the method named (compiler enforces this constraint)

Function argument types can NOT be:

- a bare function name (use .take on the function name to obtain a function reference for a Spec argument)

5.12.3. Specs

A 'spec' is the type of a function. When defining a complete function (with an is argument), both a spec and a function value are being defined.

If the function definition has no body (is argument) then only the spec (function type) is being defined.

A spec (without body) must be defined at the unit level, within a protocol, or within a record/class is section (as a function pointer field). They cannot be defined in code blocks (function bodies).

Specs and functions use structural equivalence: if two function definitions have the same parameter names and types, and the same return type, they are considered type-compatible. This is one of the few exceptions to Zerolang's general preference for nominal typing.

Function references are valtypes — small, copyable pointers to immutable program text. They can be freely copied and do not participate in ownership tracking beyond simple value semantics.

It is not possible to specify a method spec without a function value.

5.12.4. Native Declarations

The native keyword marks definitions whose implementations are provided by the compiler backend rather than by zerolang source code. Native declarations appear in the system library units (system.z and collections.z) and are not intended for use in programmer-written code.

For functions, native replaces the body expression:

length: function {:this} out u64 is native

For object types (records, classes, etc.), native replaces the instance fields block, indicating that the object's internal state is compiler-provided:

tag: record is native as { t: any.generic }

The type checker resolves signatures of native declarations through the normal resolution path. The emitter maps each native-marked definition by its qualified name to the corresponding C implementation.

5.12.5. Iterators

An iterator in Zerolang is any type whose .call method takes only :this and returns one of Option(t), optionval(t), or OptionView(t). The for loop drives an iterator by repeatedly invoking .call until the result's tag is none.

An iterable is any type that exposes an .iterate method whose only required parameter is :this and whose return type is itself iterator-shaped. The for-loop's binding RHS may be either an iterator or an iterable; rule 2 of the dispatch (see For-loop dispatch) synthesises the .iterate call. Stdlib examples: integers (5.iterate), List (mylist.iterate), Map (mymap.iterate → keys, mymap.iterateItems → entries).

Two iterator patterns are common:

  • Generator iterators yield owned per-iteration values via Option(t) or optionval(t). They synthesise their values from internal state: counters, integer ranges, parsers, network readers. Examples: counter, intrange_i64, TextReader, CpIter, LinesIter, BagIter. Each iteration produces a fresh value and the loop body owns it for the iteration's duration.
  • Container iterators yield non-owning views into a source collection via OptionView(t). The iterator holds a .lock reference into the source; each yielded value is a borrowed view, lifetime-tied to the iterator's lock. Examples: ListIter, MapKeyIter, MapItemIter, SetIter. The loop body cannot transfer ownership of a borrowed value (compile error; use .copy first).

Iterators are typically created and bound explicitly with with so the iterator's lifetime is scoped to the surrounding block; the for-loop binds the loop variable to each .call result:

with iter: (textreader file: f) do for line: iter loop { ... }

When the iterator is constructed inline as the for-loop's binding RHS, the intermediate with can be omitted — rule 1 of the dispatch creates a hidden binding with the same lifetime:

for line: (textreader file: f) loop { ... }

The recommended container-iterator form is a class with a .lock field referencing the container's private members. The container is locked for the iterator's lifetime; the iterator falls out of scope at the end of the for body and the lock on the container is released. Classes are single-owner (cannot be copied), which is what makes the .lock field safe. See examples/ListIter.z for the canonical pattern and docs/ownership.pdoc for the lock-lifecycle walkthrough.

5.12.6. Generator functions

A generator function is one whose body contains at least one yield expression. Its declared return type must be Iterator gives: T (optionally with accepts: U). The compiler synthesises a class implementing the iterator protocol; calling the generator returns an instance of that class with all parameters captured. The body, written straight-line, is lowered into a state machine whose .call method drives one step per invocation.

A function declaring out (Iterator gives: T) without a yield in its body is an ordinary factory (an unsynthesised function returning an iterator built by hand); it is not an error.

# out-only generator with captured state
intrange: function {lo: i64 hi: i64} out (Iterator gives: i64) is {
    i: lo
    for while i < hi loop {
        yield i
        i = i + 1
    }
}

# call site -- the for-loop drives the synthesised .call
for x: (intrange lo: 0 hi: 3) loop { print "\{x}" }

The shape of the .call return type is determined by the form of gives:. Like every other argument position in zerolang, a generic argument defaults to .borrow; .take is explicit.

The typechecker performs the collapse on synth-class .call return types after generator desugaring; user-written Option or OptionView calls outside this context keep their normal kind constraints.

For reftype T, bare and .borrow both yield OptionView(T) (consistent with the reftype .borrow convention -- pass a reference). Owned reftype yield requires explicit T.take.

See the Ownership on Generic Arguments section of the ownership document for the broader rule that governs ownership specifiers on generic arguments at use sites.

Because the synthesised iterator outlives the factory call, every parameter must declare ownership/locking explicitly. A bare .borrow or bare :this is rejected -- the borrow's lifetime would be the factory call, but the iterator outlives that call.

After termination the synthesised class enters an absorbing terminal state: .call returns .none forever after, idempotently. Code placed after the last yield runs on the .call that crosses into terminal state (i.e. the call after the final value is consumed). It does not run when the iterator is dropped early (for-loop break, scope exit before exhaustion). Lock release and owned-field destruction happen on class destruction along both paths. Generators do not have an explicit cleanup block; resource cleanup is RAII -- hold the resource as a field of an object whose destructor handles it.

When accepts: U is declared, callers send values back into the generator. Inside the body, expression-form yield (x: yield v) binds x to the most recent caller-supplied value on resumption. The first .call on a bidirectional generator receives no value, so the first reachable yield must be statement form (yield v). The for-loop only drives generators whose accepts is null; applying for to a bidirectional generator is a compile error pointing at the manual with/.call value: pattern.

The accepts: slot was previously named takes:; the rename removes the collision with the .take ownership method. The compiler emits a targeted rename-hint error when it sees the old name on an iterator return type.

    for line: stdin.lines loop {

Generators compose through existing constructs:

  • Delegation -- relay a sub-iterator through a for-loop inside the outer generator's body. Both forms work: 
      # inline
  for x: (subgen ...) loop { yield x }

  # explicit binding (equivalent)
  inner: (subgen ...)
  for x: inner loop { yield x }
    The desugarer auto-extracts a non-trivial for-loop iterable into a synthetic local, so the iterator persists across the enclosing generator's suspensions. Existing lock-escape rules apply through the synth class's captured-field machinery.
  • Chaining -- sequential for-loops in the same body run consecutively; each drains its source before the next starts (see examples/generator_chain.z).
  • Pipelining -- each stage is a generator taking another iterator parameter. Function composition produces the pipeline; each stage's synthesised class holds a .lock on the previous stage and existing lock-escape rules keep the chain sound.

The synthesised class is otherwise an ordinary class: visible in compiler error messages as <funcname>_iter (or <typename>.<methodname>_iter for methods), goes through the same ownership / lock / monomorphisation paths as any user-written class, and is indistinguishable from a hand-written iterator at the call site.

5.12.7. Methods

Methods are a specialised form of function. They are defined similar to a normal, named function call but they must be defined in the namespace of the receiver definition and the first argument must be of type this.

This function will then be considered a method of that object. The caller may call that method using method syntax (myinstance.mymethod args...). Calling with normal function call syntax is also still permitted (MyObject.mymethod myinstance args...).

The argument designated as this is unusual in that it is the receiver instance itself (i.e. directly addresses the instance, even value types - not pass by value as is usually the case).

Methods can change subcomponents of the receiver. Methods can directly change entire value objects (by assigning to this, but has to be same type).

Methods can be attached to user defined types, usually in the same module as the type definition. But they can also be attached to system types, type aliases and user defined types defined in external modules. In these cases, the method is only available to the local module as an extension method over the type. They do not change the type itself. They must also not collide with existing methods on the type. This may cause a maintenance burden (changes to the external type may cause code that previously compiled correctly to now fail) so should be used with caution.

5.12.8. Constructor

There is only one true low-level constructor for an object and it is meta.create -- the compiler-internal raw allocator. meta.create reserves the memory for the instance and sets each member directly from parameters supplied (or from field defaults). It is the only way to construct a new object.

meta.create is accessible only from inside the enclosing type's method bodies. Outside those methods it is an error. The compiler generates a default create wrapper around meta.create and exposes it as object.create so external callers have a valid constructor. If the user defines their own create method, that user function replaces the generated wrapper; the user body must call meta.create directly (or another constructor that eventually calls meta.create) to produce the instance.

Calling a user-defined constructor recursively -- either explicitly as Type.create args... or implicitly via bare-name Type args... -- from inside that same constructor is a compile-time error with a hint pointing at meta.create. See Implicit Construction for the full dispatch rule.

Constructors must expose (as parameters) any generic fields in the object. It must be able to be determined at the call site the actual type of the item being created (although this can be explicitly passed to a higher level by passing a generic variable at the call site into the constructor.

myObj: record {
    a: u32
    b: String

    create: function {a: 0.u32 b: String} out this is {
        return meta.create :a :b      # no stuttering... can use default args
    }
}

Class constructors whose type carries .lock fields simply return out this like any other class constructor; the type system verifies that the .lock fields are initialised from .lock parameters of the constructor (escape analysis). See Lock Fields on Classes and docs/ownership.pdoc.

Note that constructors are not executed for value types that are copied as those values are a clone of the original. Contructors for value types are only run when the value type is created explicitly.

Implicit Construction

When a type appears in a runtime expression in callable position, the compiler routes the call through the type's create method. This is uniform for every type family: records, classes, protocols, facets and typedefs all resolve construction via <type>.create.

The full call resolution rule is:

  1. If the callee is a function, call it directly.
  2. Otherwise, if the callee is a runtime instance whose type has a call method, dispatch to that call method.
  3. Otherwise, if the callee is a type, look up the type's create method and call that.
  4. Otherwise, error.

Step 3 means bare-name construction (mytype arg1: v1 arg2: v2) is equivalent to mytype.create arg1: v1 arg2: v2. Both forms validate against the actual create signature, which is either user-defined or the compiler's default wrapper around meta.create. When a type has a user-defined create with an alternate parameter list, both the bare-name and explicit forms use that signature.

The implicit construction rule does not apply inside a data clause or as a generic parameter (both of which are type references, not runtime expressions).

Disabling Default Construction (create: null)

Writing create: null in a type's as block suppresses the compiler-generated default constructor. After this, any attempt to construct the type via mytype args... or mytype.create args... is a compile-time error. Only user-defined constructors with other names (e.g. mytype.build) remain callable.

Unions and variants automatically behave as though they have create: null because construction requires selecting a specific subtype (MyUnion.subtype value). Attempting to construct a union or variant via the bare-name form produces a targeted error pointing at the subtype construction form.

Secret: class { value: i64 } as {
    create: null

    build: function {seed: i64} out this is {
        return meta.create value: seed * 7
    }
}

main: function is {
    # s1: Secret value: 42          # error: create disabled
    # s2: Secret.create value: 42   # error: create disabled
    s3: Secret.build seed: 6        # OK -- user-defined constructor
}
# constructor will not be run on copy-by-value creation

    _privatemethod: function {s: this a: "string" b: 1} is {

5.12.10. Function References

A function reference is a variable that stores a pointer to a function. It may be called with the same syntax as used when calling a standard function. It may be reassigned to a reference to an equivalent function (one with the same parameter names, types, and return type). It must always have a valid function as a value.

A function reference variable cannot be defined at the unit level (unit level declarations are immutable).

To obtain a function reference value at run time (within a function body), use .take on the function name.

Function references can be used as:

  • local variables
  • fields of an object (in the is section)
  • function parameters
  • function return values

For local variable function references, declare and assign using .take:

myspec: function {a: i64 b: i64} to i64
add: function {a: i64 b: i64} to i64 is { return a + b }

main: function is {
  ref: add.take                # local function reference
  result: ref a: 1 b: 2       # call through reference
  ref = subtract.take          # reassign to another compatible function
}

For field function references in record/class is sections:

  • define a function in the is section — it becomes a per-instance function pointer field with that function as the default value
  • OR define a spec (no body) in the is section — it becomes a per-instance function pointer field with no default; a function reference must be passed to meta.create using .take
  • the function reference can be reassigned to an equivalent function at any time using .take

Functions in the as section are shared methods on the type, not per-instance fields.

To create a function reference as a function return value:

  • define a function spec at unit level
  • in the function definition, use the name of the spec as the return type
  • in the body of the function, return an equivalent function reference

The above applies to standard function references (non-method pointers; no this parameter). Method pointers have additional restrictions:

  • all methods must be defined within the object (this is not available outside the object definition)
  • method pointers must be declared by referring to an existing method
  • so, method pointers always have a default value (there is no way to define a method pointer spec)
  • method pointers can only be replaced by another equivalent method within the object
  • method pointers cannot be function return values (they can never escape the object)

5.15. 'is' and 'as' Sections

Records, classes, unions and variants have two sections: is (instance members) and as (static/shared members). Names in both sections share a single namespace -- the same name cannot appear in both sections.

The is section contains per-instance data:

  • Fields (data stored in each instance)
  • Function pointer fields (stored per-instance, can be replaced)
  • Protocol implementation declarations

The as section contains type-level static members:

  • Methods (shared functions, not stored per-instance)
  • Generic type parameters
  • Numeric constants — integer and float (immutable, accessed via TypeName.const_name)
  • String constants (static, immutable — e.g., 'name: "hello"')
  • Computed constant expressions (e.g., max: 2 * 1024)
  • References to other definitions (e.g., alias: some_constant)
  • Protocol/facet conformance labels
  • Visibility declarations (public, private)

Functions defined in the is section are stored as function pointer fields in the instance struct. Each instance carries its own copy and these can be replaced with any function of the same signature:

calculator: record {
    op: function {a: i64 b: i64} out i64
}
main: function is {
    c: calculator op: add.take
    c.op = mul.take    # replace the function pointer
}

Functions defined in the as section are compiled as standalone functions shared by all instances. They are called via TypeName.method instance:

point: record { x: f64  y: f64 } as {
    distance: function {p: this} out f64 is {
        return (p.x * p.x) + (p.y * p.y)
    }
}

Integer constants in the as section are type-level immutable values:

page_size: 4096
config: record { x: i64 } as {
    max_val: 100
    pi: 3.14159
    buffer_size: 16 * 1024      # computed constant expression
    limit: page_size             # reference to unit-level constant
}
main: function is {
    print "\{config.max_val}"   # prints 100
    print "\{config.pi}"        # prints 3.14159
    print "\{config.buffer_size}" # prints 16384
    # config.max_val = 200      # compile error: cannot reassign static constant
}

5.16. Default values

Default values may be supplied on function/method parameters and on fields of record, class, variant, and union types. Field defaults propagate to the auto-generated meta.create constructor so the type can be created without supplying every argument.

The default form is name: <value-expression>. The value expression's type determines the parameter / field's stored type -- there is no separate name: T = default syntax.

Eligible value forms.

  • Numeric literal -- bare (42) or typed (0.u32). Field / parameter type is the inferred numeric type.
  • Function reference -- a module-level function with a body. The parameter / field's type is the function's signature and the default value is the named function. Specs (functions without a body) are not eligible as defaults.
  • In-class method reference -- inside the is block of a non-generic record or class, a field may default to a sibling method declared in the same is block (instancemethod: method1). The field's stored type is the method's FUNCTION signature; the default value is the address of the named method. The resulting slot is reassignable, so the field can be used as a swappable behaviour hook (the strategy / policy pattern). Generic types are out of scope -- methods on a generic template do not have a single resolved signature to borrow from at declaration time.
  • Variant or union null-payload subtype -- the qualified <Type>.<subtype> form, or a bare name that is a unit-level alias of such a subtype. The chosen subtype must have a null payload; other subtypes of the same type may carry data, only the chosen one must be null-payload. A bare name (for example false, which core defines as the alias false: bool.false) is accepted when it resolves to a specific null-payload subtype: the alias already names the subtype, so the field-type context is unambiguous -- no separate type / value namespace is introduced. A bare name that is not such an alias still cannot stand in for a subtype; use the qualified <Type>.<subtype> form.
  • Generic parameter type (in the as block) -- name: (<constraint>.generic default: <T>) supplies the type to bind when the parameter is omitted at the monomorphization site. The default type must satisfy the parameter's constraint (the same check user-supplied bindings face), validated at declaration time.

Examples.

  # Numeric parameter default
  inc: function {a: 0} out i64 is { return a + 1 }

  # Function-reference default (callback)
  add: function {a: i64 b: i64} out i64 is { return a + b }
  apply: function {a: i64 b: i64 f: add} out i64 is {
      return f a: a b: b
  }

  # Variant subtype default on a field
  direction: variant {
      north: null
      south: null
      east: null
      west: null
  } as { tag: u8.tag }
  state: record {
      dir: direction.north    # default arm (qualified)
      enabled: false          # bare alias of `bool.false`
      count: 0
  }

  # Generic-param type default (in an `as` block)
  Iterator: protocol {
      gives: Any.generic
      accepts: (Any.generic default: null)
  }

  # In-class method-reference default (strategy / policy pattern)
  Hookc: class {
      val: i64
      method1: function {t: this} out i64 is { return t.val + 1 }
      method2: function {t: this} out i64 is { return t.val * 10 }
      instancemethod: method1    # default hook; reassignable later
  }

Restrictions. Default values must be expressible without constructor arguments. A reftype value (heap-allocated string, list, map, user class) is not a valid default at this layer; for those, supply the value at construction time. Defaulting to an arm that carries a non-null payload is rejected for the same reason.

5.17. protocol

A protocol is defined as a list of function specs. Each spec must include a this parameter (the receiver). Static methods can also be part of a protocol. A protocol cannot include constructors or destructors because they are not on the object, only in its namespace.

Protocol methods cannot have function bodies. The protocol defines the interface only without any implementation. If additional methods over a protocol object are required, a type definition can be defined over the protocol object and addition methods (that can only interact with the object via protocol methods) can be defined.

Types (records, classes, unions, variants and enums, including protocols themselves) can claim support for one or more protocols by naming each protocol in an as clause and implementing (or relisting for protocols) all specs listed in that protocol as methods. All protocols supported must be given a label that exists in the same namespace on the object as its fields and methods.

Protocols can be used:

- to create protocol instances: see protocol instance creation. - as generic constraints: see Generic types

    read: function {:this b: buffer} out boolean

5.17.1. Protocol Instance Creation

Fields, variables and parameters can be declared to be instances of a named protocol. Protocol instances can then be assigned to those fields, variables or call arguments.

Protocol instances are reference types. They contain a pointer to the original data, a pointer to a vtable with function pointers for each spec, and a destructor pointer (NULL for borrowed instances, set for owned instances).

There are three ways to create a protocol instance:

  1. Borrowed via label (preferred): access the named protocol member in the as clause of an object (r: f.myreader). This creates a borrowed protocol instance: the protocol instance lives on the caller's stack and is forgotten when it goes out of scope, but the underlying data is not affected. The source variable is locked for the lifetime of the protocol instance.
  2. Owned via create: call protocol.create from: obj.take or use the bare-name shorthand protocol obj.take which routes through .create via the unified call dispatch. This creates an owned protocol instance: the source object is consumed and the protocol instance owns the data. When the protocol instance goes out of scope, both the protocol instance and the underlying data are destroyed. .take is not a protocol constructor; it remains the inline ownership method on runtime instances (obj.take) only.
  3. Borrowed via borrow: call protocol.borrow from: obj.lock when the label shorthand is not available. This creates a borrowed protocol instance (same semantics as the label form). The source variable is locked for the lifetime of the protocol instance. The protocol instance lives on the stack until it falls out of scope; the underlying data is not destroyed.

For borrowed instances (label and borrow forms): if the protocol instance is created in a lower scope (eg. passing as a standard loaned argument), then after that scope the protocol instance is destroyed and the underlying variable is again available for use.

Implicit projection at call sites. When a function parameter is declared with a protocol (or facet) type, callers may pass a concrete value that declares conformance to that protocol directly — no explicit protocol.borrow from: x / protocol.create from: x wrapper is needed. The compiler synthesises the projection at the call boundary:

  • borrow when the parameter's ownership annotation is .borrow or .lock (the default for protocol parameters). The caller retains ownership; the callee sees a borrowed protocol handle.
  • take when the parameter is annotated .take. The caller's binding is invalidated after the call and the callee's protocol handle owns the data.

Non-conforming arguments are rejected with a standard type-mismatch error. This rule reduces boilerplate at every call site while preserving the explicit forms for cases where the projection kind must be named. Also see the Text / StringLike heterogeneous union pattern for the analogous mechanism on generic constraints.

Reader: protocol {
    read: function {:this b: i64} out i64
}

    fd: i64
} as {
    myreader: Reader
    read: function {f: this b: i64} out i64 is {
        return f.fd + b
    }
}

use_reader: function {r: Reader} out i64 is {
    result: r.read b: 5
    return Result
}

    f: myFile fd: 10

    # Borrowed instance via label (preferred form)
    r: f.myreader              # f is locked while r is in scope

    # r: Reader.create from: f.take   # f is consumed, r owns the data
    # r: Reader.take from: f.take     # equivalent to create

    # Borrowed instance via protocol constructor
    # r: Reader.borrow from: f.lock   # f is locked while r is in scope

    print "\{use_reader r}"  # prints 15
}

5.18. facet

A facet is the value-type equivalent of a protocol. Where protocol instances contain a pointer to the underlying data, facet instances contain a copy of the data itself, making them value types.

Facet definitions follow the same syntax as protocols: the body contains specs (function signatures without bodies) that conforming types must implement. Only value types (records, variants) can implement facets.

measurable: facet {
    measure: function {:this b: i64} out i64
}

5.18.1. Conformance

A value type declares conformance to a facet via a labeled entry in its as section, the same pattern used for protocols:

point: record { x: i64 } as {
    m: measurable
    measure: function {p: this b: i64} out i64 is {
        return p.x + b
    }
}

The compiler checks that the implementing type provides all specs with matching signatures (parameter names, types, and return type).

Only value types may implement facets. Attempting to implement a facet on a class or union produces a compile-time error.

5.18.2. Constructors

The compiler generates three constructors for non-generic facets:

  • create and take copy the source value into the facet instance and return an owned facet. Signature: take: function {from: T.take} out facettype
  • borrow copies the source value and locks the source variable. Signature: borrow: function {from: T.lock} out facettype.borrow

Although the data is always copied, borrow still locks the source to maintain consistent ownership semantics with protocols. This is important when facets and protocols are used interchangeably in generic code.

5.18.3. Runtime layout

Facet instances are stack-allocated value types. The C representation is:

  • A pointer to the vtable (constant offset, first field)
  • An inline data union sized to the largest conforming type

No heap allocation or destructor is needed. Method dispatch goes through the vtable, the same as protocols.

f: measurable.create from: p
r: f.measure b: 10    # dispatches through vtable

5.18.4. Facets as generic constraints

Facets can be used as generic constraints, the same as protocols. A generic parameter constrained by a facet accepts any type that conforms to that facet.

5.18.5. Facets as typedef base

A record typedef can wrap a facet type, just as a class typedef can wrap a protocol type. Both are kind-compatible (record wraps facet because both are value types).

5.19. union

For controlling the discriminator, see also the tag type.

Unions are sum-types that can hold exactly one of their child types at any given time. A discriminator (tag) is auto-generated for each subtype. A union is a reference type with a uniform two-word layout: a tag and a void* data pointer. The union struct itself lives on the stack; the subtype data pointed to by void* is heap-allocated. All member types may be reference types or value types. Valtype payloads are stored in the heap-allocated subtype slot; reftype payloads are stored by pointer.

Methods can be defined on the union body. These methods operate on the union type itself. To access subtype-specific data, the union must first be narrowed via match.

Arguments for union:

ArgumentDefaultDescription
isYesBlock of independent subtypes, generic parameters and associated methods
asOptional block for tag specification and shared attributes

A sum-type definition is a block listing one definition for each subtype, providing a name and a type.

If there is no associated type for a subtype, the type is specified with a null.

MyUnion: union {
    u1: null       # no user stored value. Discriminator only
    :u32           # shortened syntax for 'u32: u32'
    u4: String
    u5: MyClass    # a class defined elsewhere
}

5.19.1. Constructing union instances

After a union type has been defined, instances are constructed using a per-subtype constructor accessed via dot notation: MyUnion.subtype expr. Each subtype automatically gets a constructor. Null subtypes take no argument. Valtype arguments are copied (boxed) into the union. Reftype arguments require ownership transfer via .take or by passing an anonymous/literal value.

Result: union {
    ok: i64
    err: String
    none: null
}

a: Result.ok 42                     # valtype: value is boxed
b: Result.err "something went wrong" # reftype: string literal, ownership taken
c: Result.none                      # null subtype: no argument

# named argument form
d: Result.err from: mystr.take      # explicit .take transfers ownership

5.19.2. Ownership

Union instances are owned reference types. The same ownership rules that apply to classes also apply to unions:

  • .take transfers ownership of the union itself
  • .borrow borrows the union (locks it)
  • swap exchanges two union variables
  • Aliasing prevention applies (same as classes)
  • The union owns its payload — destruction frees both the payload and the union instance

The compiler generates a per-union destructor that switches on the tag to correctly free each subtype's payload: free() for boxed valtypes, the appropriate destructor for reftype payloads, and nothing for null subtypes.

5.19.3. Match on unions

The match statement dispatches on the union's tag. All subtypes must be covered by case clauses, or an else clause must be present. Inside a case clause, the payload is accessed via dotted path on the union variable.

describe: function {r: Result} out String is {
    match r
        case ok then {
            # r.ok accesses the i64 payload (unboxed)
            print "value = \{r.ok}"
            return "ok"
        }
        case err then {
            # r.err accesses the string payload (cast)
            print "error = \{r.err}"
            return "error"
        }
        case none then {
            # no payload to access
            return "none"
        }
}

Accessing subtype data outside of a match clause is a compile error — the union must be narrowed first.

5.19.4. Creating sum-type instances

Creating/setting a union value:

MyClass: class {a: u32 b: u32}

MyUnion2: union {
    t1: null      # discriminator only
    t2: u32       # a valtype in addition to the discriminator
    t3: String
    t5: MyClass
}

v1: MyUnion2.t1                       # null subtype, no argument
v2: MyUnion2.t2 0                     # valtype, boxed
v3: MyUnion2.t3 "hello"              # string literal, ownership taken
myclassi: MyClass a: 1 b: 2
v4: MyUnion2.t5 myclassi.take        # .take transfers ownership
v5: MyUnion2.t5 (MyClass a: 1 b: 2)  # anonymous instance, no .take needed

myenum_vals: data { a: 0 b: 1 }

myvariant: variant {
    a: u32
    b: f32
} as {
    tag: myenum_vals.tag
}

Unions are also used to define generic types along with protocols. Any is a system defined union that includes all possible types used in the current program.

smallint: union { i8: null u8: null }

5.20. variant

Variants are sum-types that can contain only one of it's child types at any given time. A descriminator is usually generated or set for each case. A variant is a value type and all child types must be value types. A variant is always as large as its largest containing type plus the descriminator.

Arguments for variant:

ArgumentDefaultDescription
isYesBlock of independent subtypes, generic parameters and associated methods
asOptional block for tag specification and shared attributes

See also the union section. Other than variants being a value type, all other instructions for a union also apply to a variant.

5.20.1. Enumerated Types

A variant or union without additional data on each variation creates an enumerated type based on an underlying numeric type. To specify that there is no additional data on each arm, null can be specified as the type, or better, leave the type specifier out completely. A constant numeric value can be specified for each arm and provided it is distinct for each option and can be stored within the tag type, then that value will be used as the discriminator value.

The constant values are accessed by dot notation (they are accessed, not constructed). Each value must be unique and if defined without a value, will be one higher than the previous value or start from 0.u8.

The default underlying type is u8 if no values are specified. A numeric constant of a different type can be specified if required. All numeric values specified must be of the same type. Any unspecified values will be one higher than the previous value using the same type as the previous value.

Any member value that collides with another or overflows the underlying type will cause a compilation error.

# underlying type defaults to u8
# first unspecified value starts at 0.u8
myenum: variant {
    FIRST: null
    SECOND: null
}

# can specify unique values if required. Subsequent unspecified members will be
# 1 higher than the largest previous member. A different numeric type for the
# tag can be specified in the 'as' block.

myenum2: variant {
    FIRST: null
    SECOND: null
    THIRD: null
} as { tag: u16.tag }

# can be out of order but is usually poor style
myenum3: variant {
    SECOND: null
    FIRST: null
    ANOTHER: null
}

# usage

v: myenum.FIRST     # myenum.FIRST is a value (of type myenum)

if v.FIRST then {  # v.FIRST is a predicate
    # ...
}

# effectively equivalent to...

if v == myenum.FIRST then {
    # ...
}

Enum-style types (where all subtypes are null) should use `variant`, not `union`. A union is a reference type requiring heap allocation, which is pointless for a type that only carries a tag discriminator. Using `variant` makes the enum a stack-allocated value type with no allocation overhead.

The built-in 'boolean' type is a system defined enumeration. The boolean values may be created directly by name or indirectly via the comparison or logical operators.

# boolean types are a system defined enumeration equivalent to:
boolean: variant { FALSE TRUE }
true: boolean.TRUE
false: boolean.FALSE

5.20.2. Using sum-type instances

A sum type instance can be tested to see if it is a specific subtype. A predicate method is generated for each member named as the member name followed by a question mark. The method returns a boolean.

MyUnion: union {
    t1: u32
    t2: String
    t3: null
}

x: MyUnion t2: "hello"

if x.t3 then {
    print "x is an instance of subtype 't3'"
}

To use submember data from a union instance, it must be narrowed (or destroyed?). Assignment to a union's subtype may only be attempted in an if expression. If successful, the subtype can be used in that body specified in do.

Note that subtypes with no associated data cannot be assigned in this way.

MyUnion: union {
    t1: u32
    t2: String
    t3: _
}

x: MyUnion t2: "hello"

if y: x.t2 then {
    print y     # y is a string ("hello") here
}

if y: x.t3 then {    # ERROR: syntax error, cannot assign MyUnion.t3 to y
    # ...
}

5.21.2. Binary operations

All operations are binary. There are no unary operations. Operation syntax is an alternate function call syntax for methods with a receiver and a single parameter.

Operations are evaluated left to right. Only parentheses can be used to change precedence.

Operator names are simply method names. Identifiers used for method names can include the usual mathematical operator names eg. + - / *

5.22. Auto-Generated Equality

All value types (records, variants, arrays, str) automatically receive == and != operators. This is structural equality: records compare all fields, variants compare the discriminator tag and then the payload, arrays compare elements, and str compares bytes.

!= is always derived from ==: a != b is equivalent to !(a == b). It cannot be independently overridden.

Requirements for auto-generation:

  • All fields (records) or subtypes (variants) must themselves support ==.
  • Function pointer fields are compared by address.
  • Generic types receive == when monomorphized with concrete types that support it.

User Override.

Define == in the as section to replace the auto-generated version:

point: record { x: i64  y: i64 } as {
    ==: function {a: this b: point} out bool is {
        return a.x == b.x    # ignores y field
    }
}

Opt-Out.

Use ==: null in the as section to remove == entirely. Attempting to use == on such a type produces a compile error:

opaque: record { data: i64 } as { ==: null }
# opaque == opaque  --> compile error: No operator '=='

Float Fields.

Records containing float fields (f32, f64) use IEEE 754 comparison: NaN != NaN. This matches the behaviour of C, Rust, and Go.

5.23. Reference Type Equality

Reference types (classes, unions) do not have a default == operator. Using == on a reference type without explicitly defining it is a compile error.

This is deliberate: identity comparison (same pointer) is rarely what the programmer intends, and structural comparison on reference types is complex due to potential cycles, deep object graphs, and ownership semantics. Rather than provide a default that is almost always wrong, zerolang requires the programmer to define == explicitly when needed.

To add == to a class, define it in the as section:

Counter: class { value: i64 } as {
    ==: function {a: this b: Counter} out bool is {
        return a.value == b.value
    }
}

String Equality and Ordering.

String and stringview are the exception: each has built-in == / != that compare contents (byte-by-byte), not pointer identity, plus the full four-operator ordering: < <= > >=. A compare method returns i32 -- classic strcmp sign (-1 / 0 / 1) -- for stable-sort keys and ordered-map comparators. Two separately created strings with the same content are equal; app is less than apple because the shorter prefix loses the length tie-break:

a: "hello"
b: "hello"
if a == b then print "equal"   # prints "equal"

if "app".string < "apple".string then print "shorter wins"
c: "apple".string.compare rhs: "banana".string
# c == -1.i32

Comparison semantics: byte-wise lexicographic, matching C's memcmp. No locale awareness, no Unicode normalization, no case folding -- pure bytes. Callers that need locale-aware collation build it on top. Cross-type comparisons (String < StringView) are not defined; project explicitly with .stringview to bring both operands into the same type.

6. Compile Time Operations

6.1. Constant Folding

Operations on compile time constants are evaluated at compile time. The compiler tracks a const_value on AST nodes during type checking. Numeric literals (integer and f64), named numeric constants, and the results of arithmetic and comparison operations on constant operands are all candidates for folding.

Supported operations: + - * / < > <= >= == !=. Integer division uses truncation toward zero (matching C semantics). Float (f64) division produces a float result without truncation. Comparisons produce bool values (true/false).

Division by a constant zero is a compile time error (for both integer and float types). If the result of a constant integer operation overflows the target type, a compile time error is reported.

f32 operations are not folded. The host compiler (Python) uses 64 bit floats internally, so folding f32 arithmetic could produce results that differ from native f32 computation on the target. This limitation will be addressed when zerolang becomes self hosting.

Named constants propagate: if a: 1 and b: a + 2 are defined at unit level, then b has a compile time constant value of 3. Chains of constant definitions are fully resolved. Float constants propagate the same way.

Foldable operations are maintained in an internal registry within the compiler. All foldable operations are on native types. The registry is extensible for future operations.

OFFSET: 10
SCALE: 3
COMBINED: OFFSET + SCALE    # folded to 13 at compile time
TAU: 3.14159 * 2.0          # f64 folded to 6.28318 at compile time

main: function is {
    x: 1 + 2                # folded to 3
    y: COMBINED * 2          # folded to 26
    z: 1.5 + 2.5            # f64 folded to 4.0
}

6.2. Conditional Compilation

Normal if expressions with constant conditions are evaluated at compile time. The false branch is not emitted in the generated C output (but is still type checked). A new scope is created for the emitted branch code.

An if may be used as a definition at the unit level. Each clause's condition must evaluate to a compile-time constant. Each arm may produce a value of a different type — the result type is the type of the first arm whose condition is true. Branch blocks must be delimited with curly braces at the unit level.

A: 10
# unit-level if: selects branch at compile time
x: if A > 5 then { A } else { 0 }

6.3. Compile-Time Error

The error builtin produces a compile-time error. When the compiler can prove that an error call is reachable, compilation fails and the user's message is reported as the error. This is the primary mechanism for compile-time assertions and constraint validation.

error "this always fails"

When error appears inside an if branch whose condition is a compile-time constant, the compiler evaluates the condition statically. If the condition is false, the branch (and its error) is dead code and is suppressed. If the condition is true, the error triggers at compile time.

MODE: 1

main: function is {
    if MODE == 0 then {
        error "MODE must not be zero"
    }
    # compiles successfully because MODE is 1
}

This interacts naturally with numeric generic parameters. When a generic type is monomorphised with a concrete numeric value, the condition becomes a compile-time constant and the error is either triggered or eliminated.

buffer: record {
    used: u32
} as {
    cap: u32.generic
}

fill: function {b: buffer} is {
    if b.cap == 0 then {
        error "buffer capacity must not be zero"
    }
}

main: function is {
    ok: (buffer cap: 8)
    fill ok                     # compiles: 8 != 0

    # bad: (buffer cap: 0)
    # fill bad                  # would fail: "buffer capacity must not be zero"
}

The same mechanism works with match. When the match subject is a compile-time constant (a named constant, numeric literal, or numeric generic parameter), the compiler determines which arm matches statically. Arms that cannot match have their error calls suppressed. If no arm matches, the else clause is checked. At the C level, only the matching arm is emitted.

LEVEL: 2

check_level: function is {
    match (LEVEL
    ) case 0 then {
        error "LEVEL must not be zero"
    } case 1 then {
        print "level 1"
    } case 2 then {
        print "level 2"
    } else {
        error "unsupported LEVEL"
    }
}

A match can also dissect a generic type parameter. When the subject is a type parameter name (e.g. t from an as clause), the compiler resolves it to the concrete type during monomorphisation and selects the matching arm at compile time. This enables type-specific code paths and compile-time validation of supported types.

Container: class { val: t } as {
    t: Any.generic

    typecheck: function {c: this} is {
        match t case i32 then {
            print "i32 Container"
        } case i64 then {
            print "i64 Container"
        } else {
            error "Container: unsupported element type"
        }
    }
}

main: function is {
    c: Container val: 42        # t = i64: i64 arm selected
    c.typecheck c

    # d: Container val: "hello"  # t = string: would fail with
    #                            # "Container: unsupported element type"
}

If the error message contains string interpolation, the compiler reports a generic "compile-time error" message because interpolation cannot be evaluated at compile time.

Code following an error call is unreachable and flagged by the compiler.

The error directive forces a compilation error (during code generation?). This would usually be within a conditionally compiled section (otherwise compilation will always fail).

error "Constant error string"   # "stack" with lineno's and filenames will
                                # also be shown
